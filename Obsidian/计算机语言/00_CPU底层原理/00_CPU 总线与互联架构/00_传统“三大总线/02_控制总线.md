# 01 | 控制总线：决定硬件“什么时候做什么”的信号通道

> [!abstract] 核心结论
> 控制总线不是用来传普通数据的，而是用来传输各种控制信号。它告诉内存、外设、总线控制器当前应该读、写、等待、响应中断、复位、让出总线或开始传输。

### ① 底层原理：控制总线本质是一组控制信号线

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #2C3E50; color: #333; margin-bottom: 10px;">
<strong>控制总线的硬件本质：</strong>
<ul>
  <li><strong>控制线不是传数据值：</strong> 它传的是读、写、时钟、复位、中断、片选、等待等控制状态。</li>
  <li><strong>一根控制线表示一种控制含义：</strong> 例如 <code>RD</code> 表示读，<code>WR</code> 表示写，<code>RESET</code> 表示复位。</li>
  <li><strong>电平决定信号是否有效：</strong> 有的信号高电平有效，有的信号低电平有效。</li>
  <li><strong>控制信号负责协调时序：</strong> 什么时候输出、什么时候采样、什么时候释放，都由控制信号配合完成。</li>
  <li><strong>控制总线让硬件动作有秩序：</strong> 没有控制信号，硬件不知道当前该读、该写、该等，还是该中断。</li>
</ul>
</div>

### ② 常见控制信号

```text
RD      Read，读信号
WR      Write，写信号
CLK     Clock，时钟信号
RESET   Reset，复位信号
CS      Chip Select，片选信号
OE      Output Enable，输出使能
WE      Write Enable，写使能
INT     Interrupt，中断请求
ACK     Acknowledge，应答信号
WAIT    Wait，等待信号
READY   Ready，就绪信号
REQ     Request，请求信号
GNT     Grant，授权信号
```

### ③ 控制信号不是都同方向

```text
有些控制信号由 CPU 发出：
RD、WR、RESET、CS

有些控制信号由外设或内存返回：
READY、WAIT、INT、ACK

有些信号用于总线仲裁：
REQ、GNT
```

> [!tip] 记忆口诀
> **数据线搬内容，地址线报位置，控制线下命令：读、写、等、停、复位、中断。**

# 02 | 控制信号的有效电平：高有效与低有效

> [!abstract] 核心结论
> 控制线本质是电平信号。某根控制线是高电平有效还是低电平有效，取决于硬件设计和芯片手册。

### ① 高电平有效

```text
信号 = 1，表示有效。
信号 = 0，表示无效。
```

例如：

```text
READY = 1 表示设备已经准备好。
RESET = 1 表示复位有效。
```

### ② 低电平有效

很多硬件信号是低电平有效，常见写法有：

```text
/RD
/WR
/CS
/OE
/WE
RD#
WR#
CS#
OE#
WE#
```

含义：

```text
信号 = 0，表示有效。
信号 = 1，表示无效。
```

例如：

```text
/CS = 0 表示芯片被选中。
/CS = 1 表示芯片未被选中。
```

### ③ 为什么很多信号低电平有效

常见原因：

```text
1. 早期数字电路中低电平驱动能力更强。
2. 多设备共享信号时，低有效更容易做线与逻辑。
3. 抗干扰和硬件实现上有历史原因。
4. 芯片设计习惯沿用下来。
```

### ④ 读芯片手册时要特别注意

例如：

```text
CS   表示高有效片选。
/CS  表示低有效片选。
CS#  表示低有效片选。
```

它们不是一个意思。

> [!tip] 记忆口诀
> **信号有没有效，不看名字看电平；带横线、带斜杠、带井号，多半低电平有效。**

# 03 | 读控制时序：CPU 怎样发起一次读取

> [!abstract] 核心结论
> 读操作不是简单“拿数据”，而是一组控制信号按顺序变化：先选中目标，再发读信号，等待数据稳定，最后采样并结束周期。

### ① 一个典型读周期

```text
1. CPU 给出目标地址。
2. 片选信号有效，选中目标芯片或外设。
3. 读信号有效。
4. 被选中的设备把数据输出到数据线上。
5. 数据稳定后，CPU 采样数据。
6. 读信号无效。
7. 片选信号无效。
8. 本次读周期结束。
```

### ② 低有效读信号示例

假设：

```text
/CS 低电平有效
/RD 低电平有效
```

读周期可以理解为：

```text
/CS:  1 ───── 0 ───────── 0 ───── 1
/RD:  1 ───────── 0 ───── 1 ───── 1
DATA: Z ───── Z ── 有效数据 ─── Z
```

解释：

```text
/CS = 0：设备被选中。
/RD = 0：读操作开始。
DATA 有效：设备把数据输出。
/RD = 1：读操作结束。
/CS = 1：设备取消选中。
DATA = Z：设备释放数据线。
```

### ③ 数据为什么要等稳定

```text
读信号刚有效时，设备不一定立刻输出稳定数据。
数据线电平需要传播延迟和稳定时间。
CPU 必须在规定时刻采样。
```

如果采样太早：

```text
可能读到正在变化的数据。
```

如果采样太晚：

```text
设备可能已经释放总线。
```

> [!tip] 记忆口诀
> **读操作：先选中，再读使能；数据稳定后采样，读完释放。**

# 04 | 写控制时序：CPU 怎样把数据写出去

> [!abstract] 核心结论
> 写操作需要 CPU 先准备好地址和数据，再拉起写控制信号。被选中的设备在写信号有效期间，把数据锁存进内部存储单元或寄存器。

### ① 一个典型写周期

```text
1. CPU 给出目标地址。
2. CPU 把要写的数据放到数据线上。
3. 片选信号有效，选中目标设备。
4. 写信号有效。
5. 目标设备在规定时刻锁存数据。
6. 写信号无效。
7. 片选信号无效。
8. CPU 释放或改变数据线。
```

### ② 低有效写信号示例

假设：

```text
/CS 低电平有效
/WR 低电平有效
```

写周期可以理解为：

```text
/CS:  1 ───── 0 ───────── 0 ───── 1
/WR:  1 ───────── 0 ───── 1 ───── 1
DATA: Z ─── 有效数据 ───────── Z
```

解释：

```text
DATA 有效：CPU 已经把数据放好。
/CS = 0：目标设备被选中。
/WR = 0：写入动作有效。
/WR 从 0 变回 1 时，很多设备会锁存数据。
```

### ③ 写入的关键时序要求

```text
建立时间：
数据必须在写入采样点之前稳定。

保持时间：
写入采样点之后，数据还要保持一小段时间。

写脉冲宽度：
写信号有效时间必须足够长。
```

### ④ 为什么写控制很重要

如果没有写控制信号：

```text
设备不知道什么时候应该把数据线上的值写进去。
```

如果写信号乱跳：

```text
可能把错误数据写入寄存器或内存。
```

> [!tip] 记忆口诀
> **写操作：地址先到，数据摆好；写信号一来，设备锁存。**

# 05 | 时钟信号：控制动作发生在哪个节拍

> [!abstract] 核心结论
> 时钟信号是同步数字系统的节拍器。很多控制信号、数据采样、寄存器锁存，都要跟随时钟边沿发生。

### ① 时钟是什么

```text
时钟信号 CLK 是周期性变化的电平信号。
```

示意：

```text
CLK:  ┌─┐ ┌─┐ ┌─┐ ┌─┐
      ┘ └─┘ └─┘ └─┘ └─
```

它提供：

```text
统一节拍。
统一采样时刻。
统一状态切换时刻。
```

### ② 上升沿和下降沿

```text
上升沿：0 → 1
下降沿：1 → 0
```

很多硬件会规定：

```text
在上升沿采样。
在下降沿更新。
```

或者反过来。

### ③ 为什么需要时钟

如果没有时钟：

```text
各个硬件模块不知道什么时候读数据。
不知道什么时候写寄存器。
不知道什么时候进入下一个状态。
```

有了时钟：

```text
所有模块按统一节拍推进。
```

### ④ 建立时间和保持时间

```text
建立时间 setup time：
数据必须在时钟边沿到来前稳定一段时间。

保持时间 hold time：
时钟边沿到来后，数据必须继续保持一段时间。
```

示意：

```text
DATA: xxxx 10101010 xxxxx
             │
CLK:      ───┌─
             ↑
          采样边沿
```

要求：

```text
采样边沿前后，DATA 都必须稳定。
```

> [!tip] 记忆口诀
> **时钟是节拍，边沿是动作点；数据要提前稳，采完还要保持。**

# 06 | READY / WAIT：慢设备如何让 CPU 等一下

> [!abstract] 核心结论
> 并不是所有设备都能跟上 CPU 的速度。READY 或 WAIT 信号用于告诉 CPU：设备是否准备好，是否需要插入等待周期。

### ① 为什么需要等待信号

CPU 很快，外设可能很慢。

例如：

```text
CPU 一个周期就想读数据。
外设需要多个周期才能准备好数据。
```

如果 CPU 不等：

```text
可能读到无效数据。
```

所以需要：

```text
READY：设备准备好了。
WAIT：设备还没好，请等一下。
```

### ② READY 信号

```text
READY = 1：设备已经准备好，可以完成传输。
READY = 0：设备还没准备好，需要继续等待。
```

读操作中：

```text
CPU 发起读请求。
设备还没准备好，READY = 0。
CPU 插入等待周期。
设备准备好后，READY = 1。
CPU 完成读取。
```

### ③ WAIT 信号

```text
WAIT = 1：请求 CPU 等待。
WAIT = 0：可以继续。
```

不同系统中有效电平可能不同，要看芯片手册。

### ④ 等待周期示意

```text
周期 1：CPU 发起访问
周期 2：设备未准备好，插入等待
周期 3：设备仍未准备好，继续等待
周期 4：设备准备好，完成访问
```

### ⑤ 为什么这很重要

```text
没有等待机制：
CPU 和慢设备无法可靠通信。

有等待机制：
快 CPU 可以兼容慢内存、慢外设。
```

> [!tip] 记忆口诀
> **CPU 想快跑，外设说等等；READY 说好了，WAIT 说别急。**

# 07 | 中断信号：外设主动通知 CPU

> [!abstract] 核心结论
> 中断信号让外设可以主动通知 CPU：我有事情需要处理。CPU 不必一直轮询外设状态，而是在中断到来时暂停当前任务，转去执行中断服务程序。

### ① 什么是中断

```text
中断 = 外设或内部事件向 CPU 发出的紧急通知。
```

例如：

```text
键盘有输入。
定时器到时间。
串口收到数据。
网卡收到数据包。
ADC 转换完成。
```

### ② 中断信号线

常见命名：

```text
INT
IRQ
NMI
```

含义：

```text
INT / IRQ：普通中断请求。
NMI：不可屏蔽中断，优先级更高。
```

### ③ 中断执行过程

```text
1. 外设拉起中断请求信号。
2. CPU 在合适时刻检测到中断。
3. CPU 保存当前执行现场。
4. CPU 跳转到中断服务程序。
5. 中断服务程序处理外设事件。
6. 清除中断标志。
7. CPU 恢复现场，继续原来的程序。
```

### ④ 为什么需要中断

如果没有中断：

```text
CPU 必须不断检查外设有没有事件。
这叫轮询。
```

轮询缺点：

```text
浪费 CPU 时间。
响应可能不及时。
```

中断优点：

```text
外设有事才通知 CPU。
CPU 平时可以执行其他任务。
```

### ⑤ 中断也需要控制信号配合

```text
外设发出 IRQ。
CPU 响应中断。
中断控制器判断优先级。
CPU 进入对应中断服务程序。
```

> [!tip] 记忆口诀
> **中断就是外设举手：CPU，我有事！CPU 先存现场，再去处理。**

# 08 | 总线仲裁：多个主设备谁有权使用总线

> [!abstract] 核心结论
> 当系统中不止 CPU 一个设备能主动发起总线访问时，就需要总线仲裁。仲裁逻辑决定当前谁获得总线使用权，避免多个主设备同时控制总线。

### ① 什么是总线主设备

```text
能主动发起总线访问的设备，叫总线主设备。
```

常见主设备：

```text
CPU
DMA 控制器
某些总线控制器
高性能外设
```

### ② 为什么需要仲裁

如果 CPU 和 DMA 同时想访问内存：

```text
CPU 想用总线。
DMA 也想用总线。
```

如果没有仲裁：

```text
两者可能同时控制总线。
造成冲突。
```

### ③ REQ / GNT 信号

常见仲裁信号：

```text
REQ：Request，请求总线。
GNT：Grant，授权总线。
```

流程：

```text
1. DMA 拉起 REQ，请求使用总线。
2. 仲裁器判断当前是否允许。
3. 仲裁器发出 GNT，授权 DMA。
4. DMA 获得总线控制权。
5. DMA 完成传输后释放总线。
6. CPU 或其他设备继续使用。
```

### ④ 仲裁策略

常见策略：

```text
固定优先级：
某些设备永远优先。

轮询仲裁：
多个设备轮流获得机会。

时间片：
每个设备使用固定时间。

动态优先级：
根据实时情况调整优先级。
```

### ⑤ 仲裁的目标

```text
1. 避免总线冲突。
2. 保证高优先级设备及时响应。
3. 尽量公平分配总线使用权。
4. 提高系统整体吞吐量。
```

> [!tip] 记忆口诀
> **多个主设备抢总线，仲裁器说了算；先请求，再授权，用完释放。**

# 09 | DMA 控制信号：让搬运不必 CPU 亲自做

> [!abstract] 核心结论
> DMA 需要控制总线配合才能工作。CPU 先配置 DMA，DMA 通过请求、授权、读写、完成中断等控制信号，完成外设和内存之间的数据搬运。

### ① DMA 为什么需要控制信号

DMA 要做的事情是：

```text
从一个地址搬数据到另一个地址。
搬多少。
什么时候开始。
什么时候结束。
结束后通知 CPU。
```

这些都需要控制信号或控制寄存器配合。

### ② DMA 基本流程

```text
1. CPU 配置 DMA 控制寄存器：
   源地址、目标地址、长度、方向、模式。

2. CPU 启动 DMA。

3. DMA 请求总线使用权。

4. 仲裁器授权 DMA。

5. DMA 发起读写周期搬运数据。

6. 搬运完成后，DMA 释放总线。

7. DMA 产生完成中断通知 CPU。
```

### ③ DMA 中常见控制含义

```text
启动信号：
开始 DMA 传输。

方向控制：
内存到外设，或者外设到内存。

传输完成标志：
表示 DMA 已经搬完。

错误标志：
表示传输过程中出错。

中断使能：
传输完成后是否通知 CPU。
```

### ④ DMA 对 CPU 的意义

传统 CPU 搬运：

```text
CPU 读外设 → CPU 写内存 → 重复很多次
```

DMA 搬运：

```text
CPU 配置一次 → DMA 自动搬运 → 完成后通知 CPU
```

优势：

```text
CPU 不用每个字节都亲自搬。
适合大块数据传输。
提高系统并行度。
```

> [!tip] 记忆口诀
> **DMA 搬数据，控制信号管流程；CPU 先安排，DMA 自己干，干完再中断。**

# 10 | 控制总线总结

> [!abstract] 核心结论
> 控制总线的底层核心是“控制信号线 + 有效电平 + 时序约束 + 状态协调”。它不负责保存数据，也不负责表示普通数值，而是负责让硬件在正确时间执行正确动作。

### ① 核心总结

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #27AE60; color: #333; margin-bottom: 10px;">
<strong>控制总线底层总结：</strong>
<ul>
  <li><strong>本质：</strong> 一组控制信号线。</li>
  <li><strong>作用：</strong> 决定硬件什么时候读、写、等待、复位、中断、授权。</li>
  <li><strong>有效电平：</strong> 控制信号可能高有效，也可能低有效。</li>
  <li><strong>读写时序：</strong> 控制信号必须按顺序变化，保证数据可靠传输。</li>
  <li><strong>时钟：</strong> 提供统一节拍，决定采样和锁存时刻。</li>
  <li><strong>READY / WAIT：</strong> 让慢设备可以请求 CPU 等待。</li>
  <li><strong>中断：</strong> 让外设主动通知 CPU 有事件发生。</li>
  <li><strong>仲裁：</strong> 多个主设备抢总线时决定谁能使用。</li>
  <li><strong>DMA：</strong> 通过控制信号接管总线，替 CPU 完成数据搬运。</li>
</ul>
</div>

### ② 最底层一句话

```text
控制总线 = 一组决定硬件动作时机和方向的电平信号。
```

### ③ 最终记忆口诀

> [!tip] 记忆口诀
> **控制总线不搬数，只发命令和节拍；读写靠使能，慢了发等待；外设要通知，中断来打断；多主抢总线，仲裁来裁判。**
