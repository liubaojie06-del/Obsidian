# 01 | 功能文件：真正实现功能的 `.c` 文件

> [!abstract] 核心结论
> 功能文件通常指模块对应的 `.c` 源文件。它负责写函数定义、全局变量定义、模块内部逻辑、`static` 私有函数等内容。头文件负责“声明有什么”，功能文件负责“具体怎么做”。

### ① 底层原理

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #2C3E50; color: #333; margin-bottom: 10px;">
<strong>功能文件的本质：</strong>
<ul>
  <li><strong>后缀：</strong>通常是 <code>.c</code>。</li>
  <li><strong>核心作用：</strong>实现头文件中声明的函数。</li>
  <li><strong>真正定义：</strong>函数体、全局变量定义、模块内部变量通常放在功能文件中。</li>
  <li><strong>参与编译：</strong><code>.c</code> 文件会被 GCC 编译成 <code>.o</code> 目标文件。</li>
  <li><strong>和头文件配合：</strong><code>.h</code> 放接口声明，<code>.c</code> 放功能实现。</li>
</ul>
</div>

### ② 代码用法

一个模块通常由两个文件组成：

```text
calc.h    // 头文件：声明接口
calc.c    // 功能文件：实现功能
```

calc.h：

```c
#ifndef CALC_H
#define CALC_H

int add(int a, int b);
int sub(int a, int b);

#endif
```

calc.c：

```c
#include "calc.h"

int add(int a, int b) {
    return a + b;
}

int sub(int a, int b) {
    return a - b;
}
```

### ③ 使用场景

| 文件 | 作用 | 放什么 |
| :--- | :--- | :--- |
| `.h` | 声明接口 | 函数声明、结构体声明、宏、extern 声明 |
| `.c` | 实现功能 | 函数定义、变量定义、内部逻辑 |
| `main.c` | 调用功能 | 程序入口和主流程 |
| `.o` | 编译结果 | `.c` 编译后的目标文件 |

> [!tip] 记忆口诀
> **头文件说能干什么，功能文件写怎么干。**

# 02 | 功能文件里应该放什么

> [!abstract] 核心结论
> 功能文件应该放真正的实现代码，包括函数定义、全局变量定义、模块内部 `static` 变量、模块内部辅助函数等。

### ① 函数定义

头文件声明：

```c
int add(int a, int b);
```

功能文件实现：

```c
int add(int a, int b) {
    return a + b;
}
```

说明：

```text
函数声明只告诉编译器有这个函数。
函数定义才是真正的代码实现。
```

### ② 全局变量定义

头文件声明：

```c
extern int g_debug_mode;
```

功能文件定义：

```c
int g_debug_mode = 1;
```

说明：

```text
extern 只是声明。
真正分配空间的定义要放在某个 .c 文件里。
```

### ③ static 私有变量

```c
static int g_count = 0;
```

说明：

```text
static 全局变量只能在当前功能文件中使用。
外部文件不能直接访问。
```

### ④ static 私有函数

```c
static int check_value(int value) {
    return value >= 0 && value <= 100;
}
```

说明：

```text
这个函数只给当前 .c 文件内部使用。
不暴露给其他模块。
```

### ⑤ 模块内部逻辑

```text
数据处理。
条件判断。
算法实现。
文件操作。
字符串处理。
业务流程。
```

> [!tip] 记忆口诀
> **别人要用的写声明，自己实现的放 c；内部工具加 static，别暴露太多。**

# 03 | 功能文件和头文件如何配合

> [!abstract] 核心结论
> 功能文件通常要包含自己的头文件，这样编译器可以检查函数定义是否和头文件声明一致，避免参数类型、返回值类型不匹配。

### ① 推荐写法

calc.c：

```c
#include "calc.h"

int add(int a, int b) {
    return a + b;
}
```

原因：

```text
calc.c 包含 calc.h。
编译器能检查 add 的定义是否和声明一致。
```

### ② 错误风险

calc.h：

```c
int add(int a, int b);
```

calc.c：

```c
double add(double a, double b) {
    return a + b;
}
```

问题：

```text
声明和定义不一致。
容易导致编译错误或链接问题。
```

如果 calc.c 包含 calc.h，编译器更容易发现问题。

### ③ main.c 调用

```c
#include <stdio.h>
#include "calc.h"

int main() {
    printf("%d\n", add(10, 20));
    return 0;
}
```

main.c 不需要知道 add 里面怎么实现。

它只需要知道：

```text
函数名。
参数。
返回值。
```

这些由 calc.h 提供。

### ④ 编译

```bash
gcc main.c calc.c -o app
```

说明：

```text
main.c 负责调用。
calc.c 负责实现。
链接器把两者连接起来。
```

### ⑤ 重点

```text
.c 文件提供真实功能。
.h 文件提供使用说明。
```

> [!tip] 记忆口诀
> **功能文件 include 自己的头文件，声明实现对得上。**

# 04 | 功能文件示例：计算模块

> [!abstract] 核心结论
> 一个简单的功能模块通常包含一个头文件和一个功能文件。头文件暴露接口，功能文件实现接口，主函数只调用接口。

### ① 项目结构

```text
project/
├── main.c
├── calc.h
└── calc.c
```

### ② calc.h

```c
#ifndef CALC_H
#define CALC_H

int add(int a, int b);
int sub(int a, int b);
int mul(int a, int b);
int divide(int a, int b);

#endif
```

### ③ calc.c

```c
#include "calc.h"

int add(int a, int b) {
    return a + b;
}

int sub(int a, int b) {
    return a - b;
}

int mul(int a, int b) {
    return a * b;
}

int divide(int a, int b) {
    if (b == 0) {
        return 0;
    }

    return a / b;
}
```

### ④ main.c

```c
#include <stdio.h>
#include "calc.h"

int main() {
    int a = 20;
    int b = 10;

    printf("add = %d\n", add(a, b));
    printf("sub = %d\n", sub(a, b));
    printf("mul = %d\n", mul(a, b));
    printf("divide = %d\n", divide(a, b));

    return 0;
}
```

### ⑤ 编译运行

```bash
gcc main.c calc.c -o app
./app
```

输出：

```text
add = 30
sub = 10
mul = 200
divide = 2
```

> [!tip] 记忆口诀
> **一个模块一对文件，h 写接口，c 写实现。**

# 05 | 功能文件中的 static 私有函数

> [!abstract] 核心结论
> 如果某个函数只在当前功能文件内部使用，不希望外部调用，应该在函数前加 `static`。这样可以隐藏实现细节，避免函数名冲突。

### ① 普通函数会暴露

```c
int helper(int value) {
    return value * 2;
}
```

问题：

```text
其他 .c 文件也可能声明并调用 helper。
项目变大后容易产生命名冲突。
```

### ② static 私有函数

```c
static int helper(int value) {
    return value * 2;
}
```

含义：

```text
helper 只在当前 .c 文件内部可见。
其他 .c 文件无法链接到它。
```

### ③ 示例代码

score.c：

```c
#include "score.h"

static int check_score(int score) {
    return score >= 0 && score <= 100;
}

int is_pass(int score) {
    if (!check_score(score)) {
        return 0;
    }

    return score >= 60;
}
```

score.h：

```c
#ifndef SCORE_H
#define SCORE_H

int is_pass(int score);

#endif
```

说明：

```text
is_pass 是对外接口。
check_score 是内部工具函数。
```

### ④ 好处

```text
隐藏内部实现。
减少外部依赖。
避免函数名冲突。
模块边界更清晰。
方便以后修改内部逻辑。
```

### ⑤ 重点

```text
头文件只声明外部需要调用的函数。
内部辅助函数直接写在 .c 文件，并加 static。
```

> [!tip] 记忆口诀
> **对外函数放头文件，内部函数加 static。**

# 06 | 功能文件中的 static 私有变量

> [!abstract] 核心结论
> 功能文件中的 `static` 全局变量是模块私有状态，只能在当前 `.c` 文件中访问。外部不能直接修改它，只能通过模块提供的函数间接操作。

### ① 示例结构

```text
counter.h
counter.c
main.c
```

### ② counter.h

```c
#ifndef COUNTER_H
#define COUNTER_H

void counter_add(void);
void counter_reset(void);
int counter_get(void);

#endif
```

### ③ counter.c

```c
#include "counter.h"

static int g_count = 0;

void counter_add(void) {
    g_count++;
}

void counter_reset(void) {
    g_count = 0;
}

int counter_get(void) {
    return g_count;
}
```

### ④ main.c

```c
#include <stdio.h>
#include "counter.h"

int main() {
    counter_add();
    counter_add();

    printf("%d\n", counter_get());

    counter_reset();

    printf("%d\n", counter_get());

    return 0;
}
```

### ⑤ 编译运行

```bash
gcc main.c counter.c -o app
./app
```

输出：

```text
2
0
```

说明：

```text
外部文件不能直接访问 g_count。
只能通过 counter_add、counter_reset、counter_get 操作。
```

> [!tip] 记忆口诀
> **模块状态用 static 藏起来，对外只给函数接口。**

# 07 | 功能文件不要直接被 include

> [!abstract] 核心结论
> 正常多文件编程中，应该 `#include` 头文件，而不是 `#include` 功能文件。功能文件应该参与编译链接，而不是被另一个源文件展开进去。

### ① 错误写法

main.c：

```c
#include "calc.c"

int main() {
    return add(10, 20);
}
```

问题：

```text
把 calc.c 直接展开进 main.c。
容易导致重复定义。
破坏模块化。
编译结构混乱。
```

### ② 正确写法

main.c：

```c
#include "calc.h"

int main() {
    return add(10, 20);
}
```

编译时：

```bash
gcc main.c calc.c -o app
```

### ③ 为什么不能 include `.c`

```text
.h 是声明文件，适合 include。
.c 是实现文件，适合编译。
```

include 的本质是：

```text
文本展开。
```

如果 include `.c`：

```text
函数定义会被复制到当前文件中。
多个文件 include 时容易重复定义。
```

### ④ 正确流程

```text
main.c include calc.h。
calc.c include calc.h。
main.c 和 calc.c 分别编译。
链接器把 add 调用和 add 实现连接起来。
```

### ⑤ 重点

```text
include 头文件。
编译源文件。
链接目标文件。
```

> [!tip] 记忆口诀
> **h 用来 include，c 用来编译；别把 c 当头文件包含。**

# 08 | 功能文件的编译过程

> [!abstract] 核心结论
> 每个功能文件 `.c` 都会被单独编译成 `.o` 目标文件。最后链接器把所有 `.o` 文件组合成可执行程序。

### ① 一步编译

```bash
gcc main.c calc.c counter.c -o app
```

说明：

```text
GCC 自动完成预处理、编译、汇编、链接。
```

### ② 分步编译

```bash
gcc -c main.c -o main.o
gcc -c calc.c -o calc.o
gcc -c counter.c -o counter.o
gcc main.o calc.o counter.o -o app
```

说明：

```text
-c 表示只编译成 .o，不进行链接。
```

### ③ 编译过程理解

```text
main.c     → main.o
calc.c     → calc.o
counter.c  → counter.o

main.o + calc.o + counter.o → app
```

### ④ 如果忘记编译功能文件

错误：

```bash
gcc main.c -o app
```

如果 main.c 调用了 calc.c 中的函数：

```text
链接器找不到函数定义。
可能报 undefined reference。
```

正确：

```bash
gcc main.c calc.c -o app
```

### ⑤ 加头文件路径

如果头文件在 include 目录：

```bash
gcc src/main.c src/calc.c -Iinclude -o app
```

说明：

```text
-Iinclude 告诉 GCC 去 include 目录找头文件。
```

> [!tip] 记忆口诀
> **每个功能文件都要参与编译，不然链接器找不到实现。**

# 09 | 功能文件常见错误

> [!abstract] 核心结论
> 功能文件常见错误包括忘记包含自己的头文件、声明和定义不一致、函数定义重复、内部函数没有加 `static`、编译时漏掉功能文件。

### ① 错误 1：功能文件不包含自己的头文件

不推荐：

```c
// calc.c
int add(int a, int b) {
    return a + b;
}
```

更推荐：

```c
// calc.c
#include "calc.h"

int add(int a, int b) {
    return a + b;
}
```

原因：

```text
可以检查 calc.h 中的声明和 calc.c 中的定义是否一致。
```

### ② 错误 2：声明和定义不一致

calc.h：

```c
int add(int a, int b);
```

calc.c：

```c
double add(double a, double b) {
    return a + b;
}
```

问题：

```text
函数返回值和参数类型不一致。
```

### ③ 错误 3：多个功能文件定义同名全局变量

a.c：

```c
int g_num = 10;
```

b.c：

```c
int g_num = 20;
```

问题：

```text
链接时重复定义。
```

### ④ 错误 4：内部函数不加 static

```c
int helper(void) {
    return 1;
}
```

如果多个模块都有 helper：

```text
可能产生命名冲突。
```

更推荐：

```c
static int helper(void) {
    return 1;
}
```

### ⑤ 错误 5：编译时漏掉功能文件

错误：

```bash
gcc main.c -o app
```

正确：

```bash
gcc main.c calc.c -o app
```

> [!tip] 记忆口诀
> **功能文件要 include 自己的 h，内部工具加 static，编译别漏 c。**

# 10 | 功能文件总结

> [!abstract] 核心结论
> 功能文件 `.c` 的底层核心是“模块真正实现区”。它负责函数定义、变量定义和内部逻辑。头文件对外声明接口，功能文件实现接口，主程序通过头文件调用功能，最终由链接器把所有功能文件拼成完整程序。

### ① 核心总结

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #27AE60; color: #333; margin-bottom: 10px;">
<strong>功能文件核心总结：</strong>
<ul>
  <li><strong>后缀：</strong>通常是 <code>.c</code>。</li>
  <li><strong>作用：</strong>实现模块功能。</li>
  <li><strong>放函数定义：</strong>真正的函数体写在功能文件中。</li>
  <li><strong>放变量定义：</strong>真正分配空间的全局变量定义放在功能文件中。</li>
  <li><strong>配合头文件：</strong>功能文件通常 include 自己对应的 <code>.h</code>。</li>
  <li><strong>内部函数：</strong>只在当前文件用的函数建议加 <code>static</code>。</li>
  <li><strong>内部变量：</strong>模块私有状态建议用 <code>static</code> 隐藏。</li>
  <li><strong>不要 include .c：</strong>其他文件应该 include 头文件，不要 include 功能文件。</li>
  <li><strong>参与编译：</strong>每个功能文件都要加入 GCC 编译命令。</li>
  <li><strong>链接作用：</strong>链接器把 main 中的调用和功能文件中的实现连接起来。</li>
</ul>
</div>

### ② 最底层一句话

```text
功能文件 = 模块的真正实现文件；头文件告诉别人怎么用，功能文件负责把功能做出来。
```

### ③ 最终记忆口诀

> [!tip] 记忆口诀
> **头文件写接口，功能文件写实现；内部细节 static 藏，编译时别漏 .c。**
