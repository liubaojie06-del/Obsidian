# 01 | 分支预测器：CPU 前端用来提前猜路的硬件

> [!abstract] 核心结论
> 分支预测器是 CPU 前端的重要部件，用来预测条件跳转、函数调用、函数返回、间接跳转等控制流的下一步走向。它的目标是让取指单元不断拿到正确路径上的指令，避免流水线因为等待分支结果而停顿。

### ① 底层原理：CPU 不能等分支算完才取下一条指令

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #2C3E50; color: #333; margin-bottom: 10px;">
<strong>分支预测器的硬件本质：</strong>
<ul>
  <li><strong>分支会改变 PC：</strong> 普通指令让 PC 顺序前进，分支指令可能让 PC 跳到另一个地址。</li>
  <li><strong>流水线需要提前取指：</strong> CPU 前端必须提前知道下一批指令从哪里取。</li>
  <li><strong>分支结果通常较晚才知道：</strong> 条件判断可能要等 ALU 比较、寄存器读取或执行阶段完成。</li>
  <li><strong>预测器提前猜方向：</strong> 猜这个分支会不会跳转。</li>
  <li><strong>预测器提前猜目标：</strong> 如果会跳，猜跳到哪个地址。</li>
</ul>
</div>

### ② 一句话理解

```text
分支预测器 = CPU 前端的“导航系统”，在真正算出结果前，先猜下一条指令该去哪取。
```

例如：

```c
if (x > 0) {
    a++;
} else {
    b++;
}
```

CPU 在真正知道 `x > 0` 是否成立之前，就要先猜：

```text
继续取 if 分支里的指令？
还是取 else 分支里的指令？
```

> [!tip] 记忆口诀
> **分支没算完，CPU 先猜路；猜对不断流，猜错倒回来。**

# 02 | 为什么分支预测对流水线这么重要

> [!abstract] 核心结论
> 流水线 CPU 同时处理多条指令的不同阶段。分支指令如果不预测，CPU 必须等分支结果出来后才能继续取指，前端会断粮，后端执行单元也会空转。

### ① 流水线中的分支问题

典型流水线：

```text
IF   取指
ID   译码
EX   执行 / 比较
MEM  访存
WB   写回
```

条件分支可能要到：

```text
EX 阶段
```

才知道：

```text
到底跳不跳。
```

但是取指单元在 IF 阶段就需要知道：

```text
下一条指令地址。
```

### ② 不预测会怎样

如果没有分支预测：

```text
1. 遇到条件分支。
2. CPU 停止继续取后续指令。
3. 等分支条件计算完成。
4. 确认跳转方向。
5. 再从正确地址取指。
```

结果：

```text
流水线出现空泡。
执行单元没活干。
性能下降。
```

### ③ 有预测会怎样

有分支预测后：

```text
1. 遇到分支。
2. 预测器猜测跳转方向和目标地址。
3. 取指单元沿预测路径继续取指。
4. 后端实际计算分支结果。
5. 如果猜对，流水线继续推进。
6. 如果猜错，清空错误路径并重取。
```

### ④ 分支越多，预测越重要

程序中常见控制流：

```text
if / else
for
while
switch
函数调用
函数返回
异常处理
虚函数调用
函数指针跳转
```

这些都会影响取指路径。

### ⑤ 高性能 CPU 更依赖预测

流水线越深、前端越宽：

```text
一次误预测浪费的周期和指令越多。
```

所以现代 CPU 会设计非常复杂的分支预测器。

> [!tip] 记忆口诀
> **流水线怕等，分支最会打断；预测就是不断粮，误预测就是白干一段。**

# 03 | 分支预测器到底预测什么

> [!abstract] 核心结论
> 分支预测器不只预测“跳不跳”，还要预测“跳到哪里”。不同类型的控制流需要不同预测结构，例如条件分支用方向预测，跳转目标用 BTB，函数返回用 RAS。

### ① 预测方向

```text
方向预测 = 判断分支是否会跳转。
```

例如：

```c
if (x > 0)
```

预测器要猜：

```text
x > 0 大概率成立？
还是不成立？
```

对应：

```text
Taken：跳转。
Not Taken：不跳转，顺序执行。
```

### ② 预测目标

```text
目标预测 = 如果跳转，跳到哪个地址。
```

例如：

```asm
JMP 0x400800
```

目标地址是：

```text
0x400800
```

但在复杂 CPU 中，前端需要很早知道这个地址，不能等完整解码后才知道。

### ③ 函数返回预测

函数返回指令：

```text
return
```

它的目标地址取决于之前的函数调用位置。

所以 CPU 常用：

```text
RAS：Return Address Stack
返回地址栈
```

来预测返回地址。

### ④ 间接跳转预测

间接跳转目标不是固定立即数，而是来自寄存器或内存。

例如：

```c
fp();        // 函数指针调用
obj->func(); // 虚函数调用
```

底层目标地址可能运行时才知道。

预测器需要猜：

```text
这次间接跳转会去哪个目标函数。
```

### ⑤ 分支预测包含多类预测器

```text
方向预测器：
猜跳不跳。

BTB：
猜跳转目标地址。

RAS：
猜函数返回地址。

间接分支预测器：
猜动态目标地址。
```

> [!tip] 记忆口诀
> **分支预测有两问：跳不跳？跳去哪？函数返回看 RAS，目标地址看 BTB。**

# 04 | 静态预测与动态预测

> [!abstract] 核心结论
> 静态预测不依赖运行历史，规则简单但准确率有限；动态预测会根据程序过去的分支行为学习规律，通常准确率更高，是现代 CPU 的主流方式。

### ① 静态预测

静态预测使用固定规则，例如：

```text
默认不跳。
默认跳。
向后跳预测为跳。
向前跳预测为不跳。
编译器给提示。
```

### ② 为什么向后跳常预测为跳

循环通常会向后跳：

```c
for (int i = 0; i < 1000; i++) {
    sum += i;
}
```

底层大致是：

```text
循环体执行完后，跳回循环开头。
```

前 999 次都跳，最后一次才不跳。

所以：

```text
向后跳大概率是循环，预测为跳通常不错。
```

### ③ 动态预测

动态预测根据历史行为学习：

```text
这个分支上次跳了吗？
最近几次跳了吗？
当前分支和之前其他分支有没有关系？
```

预测器会保存历史状态，下次遇到相同分支时参考历史。

### ④ 动态预测的优点

```text
能适应程序实际运行行为。
能学习循环。
能学习 if 的偏向性。
能利用上下文关系。
准确率更高。
```

### ⑤ 动态预测的代价

```text
需要硬件表项。
需要更新历史。
消耗芯片面积。
消耗功耗。
可能被错误历史污染。
可能出现别名冲突。
```

> [!tip] 记忆口诀
> **静态预测靠规则，动态预测靠记忆；程序有规律，预测器就能学。**

# 05 | BHT 与两位饱和计数器：最经典的方向预测

> [!abstract] 核心结论
> BHT 用来记录分支历史状态。经典实现是两位饱和计数器：不是一次错就立刻改主意，而是需要连续趋势变化才改变预测方向，这样能更稳定地预测循环分支。

### ① BHT 是什么

```text
BHT = Branch History Table
分支历史表
```

它保存：

```text
某些分支指令过去是跳转还是不跳转。
```

CPU 用分支指令地址的一部分去索引 BHT。

### ② 一位预测器的问题

一位预测器只记：

```text
上次跳了没有。
```

例如循环执行 100 次：

```text
前 99 次跳。
最后 1 次不跳。
```

一位预测器的问题：

```text
最后一次退出循环会预测错。
下一次重新进入循环时，又可能预测错。
```

### ③ 两位饱和计数器

两位状态通常是：

```text
00：强不跳
01：弱不跳
10：弱跳
11：强跳
```

预测规则：

```text
00 / 01 → 预测不跳
10 / 11 → 预测跳
```

状态更新：

```text
实际跳转：计数器加 1，最多到 11。
实际不跳：计数器减 1，最少到 00。
```

### ④ 为什么两位更稳

如果一个循环长期跳转：

```text
状态会变成强跳 11。
```

最后一次退出循环时实际不跳：

```text
状态从 11 变成 10。
```

下次遇到这个分支：

```text
仍然预测跳。
```

这样不会因为一次退出循环就完全改变判断。

### ⑤ 状态机示意

```text
实际跳转方向推动状态：

强不跳 00 ←→ 弱不跳 01 ←→ 弱跳 10 ←→ 强跳 11
```

越靠右：

```text
越相信会跳。
```

越靠左：

```text
越相信不跳。
```

> [!tip] 记忆口诀
> **两位计数器不轻易变心；一次反常只是动摇，连续反常才改方向。**

# 06 | BTB：提前知道跳转目标地址

> [!abstract] 核心结论
> BTB 用来缓存分支指令的目标地址。当前端取到某个 PC 时，如果 BTB 命中，就能提前知道这条指令可能跳到哪里，从而立即从目标地址继续取指。

### ① BTB 是什么

```text
BTB = Branch Target Buffer
分支目标缓冲区
```

它保存：

```text
分支指令地址 → 跳转目标地址
```

例如：

```text
PC = 0x1000 的分支
历史目标 = 0x2000
```

BTB 表项可能记录：

```text
0x1000 → 0x2000
```

### ② 为什么需要 BTB

即使预测器知道：

```text
这个分支会跳。
```

也还需要知道：

```text
跳到哪里。
```

如果等解码器算出目标地址：

```text
前端可能已经晚了几个周期。
```

BTB 可以在取指早期就给出目标地址。

### ③ BTB 命中

```text
当前 PC 查 BTB。
如果找到匹配表项。
得到目标地址。
前端直接从目标地址取指。
```

### ④ BTB 未命中

```text
没有找到目标地址。
前端可能按顺序取指。
等后面解码或执行阶段算出真实目标。
如果发现应该跳转，就要重定向取指。
```

### ⑤ BTB 的挑战

BTB 容量有限，会有：

```text
容量冲突。
地址别名。
旧目标污染。
间接分支多目标问题。
上下文切换影响。
```

如果 BTB 给错目标：

```text
即使方向预测对了，也可能取错路径。
```

> [!tip] 记忆口诀
> **BHT 猜跳不跳，BTB 猜跳去哪；方向和目标都对，前端才真顺。**

# 07 | 局部历史、全局历史与相关性预测

> [!abstract] 核心结论
> 简单预测只看某个分支自己的过去，而高级预测会看更多上下文。局部历史记录单个分支的模式，全局历史记录最近所有分支的走向，用来捕捉分支之间的相关性。

### ① 局部历史

局部历史关注：

```text
同一个分支自己最近几次的行为。
```

例如某个分支历史是：

```text
跳、不跳、跳、不跳、跳、不跳
```

说明它可能有交替规律。

局部预测器可以学习这种模式。

### ② 全局历史

全局历史关注：

```text
最近若干个分支整体的跳转序列。
```

例如：

```text
最近 8 个分支结果 = 11010110
```

当前分支的结果可能和前面几个分支有关。

### ③ 分支相关性

例如：

```c
if (a > 0) {
    flag = 1;
}

if (flag) {
    do_something();
}
```

第二个分支和第一个分支有关系。

如果预测器只看第二个分支自己，可能不够准。

如果看全局历史：

```text
刚才第一个分支跳了。
那么第二个分支也更可能跳。
```

### ④ GShare 思想

一种经典思路是：

```text
用分支地址和全局历史做异或。
再索引预测表。
```

这样可以同时利用：

```text
当前分支是谁。
之前分支发生了什么。
```

### ⑤ 现代预测器更复杂

现代 CPU 可能使用：

```text
局部预测器。
全局预测器。
路径历史。
混合预测器。
TAGE 类预测器。
神经网络式感知机预测器。
间接目标预测器。
```

目标都是：

```text
提高预测准确率。
降低误预测惩罚。
```

> [!tip] 记忆口诀
> **局部历史看自己，全局历史看上下文；分支之间有关联，预测就能更聪明。**

# 08 | RAS：函数返回地址预测

> [!abstract] 核心结论
> 函数调用和返回非常频繁。返回指令的目标地址不是固定写在指令里，而是来自调用点保存的返回地址。RAS 用栈结构记录调用返回关系，专门预测 return 要回到哪里。

### ① 为什么 return 难预测

函数调用：

```c
foo();
```

调用时会保存：

```text
返回地址
```

函数执行完：

```c
return;
```

需要跳回调用点后面的指令。

问题是：

```text
同一个函数可能从很多地方被调用。
同一条 return 指令可能返回不同地址。
```

### ② RAS 是什么

```text
RAS = Return Address Stack
返回地址栈
```

它用硬件栈记录：

```text
函数调用时的返回地址。
```

### ③ RAS 工作流程

函数调用时：

```text
1. CPU 预测到 call。
2. 把 call 后面的地址压入 RAS。
3. PC 跳到函数入口。
```

函数返回时：

```text
1. CPU 预测到 return。
2. 从 RAS 弹出顶部返回地址。
3. 前端从该地址继续取指。
```

### ④ 为什么栈结构适合函数调用

函数调用天然是后进先出：

```text
main 调 A
A 调 B
B 返回 A
A 返回 main
```

顺序是：

```text
最后调用的函数最先返回。
```

这正好适合栈。

### ⑤ RAS 可能失准的情况

```text
函数调用和返回不匹配。
异常或中断打断流程。
栈溢出或 RAS 容量不足。
长跳转 longjmp。
异常处理。
递归太深。
安全机制改变返回路径。
```

> [!tip] 记忆口诀
> **call 时压返回地址，return 时弹出来；函数调用像栈，RAS 就专门猜返回。**

# 09 | 误预测惩罚与投机执行

> [!abstract] 核心结论
> 分支预测正确时，CPU 可以提前执行未来路径上的指令；这叫投机执行。预测错误时，CPU 必须丢弃错误路径上的结果，恢复正确状态，重新从正确地址取指。

### ① 什么是投机执行

```text
Speculative Execution
投机执行
```

含义：

```text
CPU 在分支结果尚未确定前，
先按预测路径取指、解码、执行。
```

如果预测正确：

```text
这些工作就真的有用。
```

如果预测错误：

```text
这些工作要被丢弃。
```

### ② 预测正确时

```text
前端持续供给指令。
后端持续执行。
流水线保持高利用率。
性能提升。
```

### ③ 预测错误时

CPU 需要：

```text
1. 发现真实分支结果与预测不一致。
2. 清空错误路径上的指令和 uop。
3. 丢弃错误路径上的临时结果。
4. 恢复架构可见状态。
5. 把 PC 改到正确地址。
6. 重新取指。
```

### ④ 误预测惩罚

误预测惩罚取决于：

```text
流水线深度。
分支解析位置。
前端重新取指时间。
uop 队列清空代价。
缓存命中情况。
```

常见表现：

```text
一次误预测可能浪费十几个甚至更多周期。
```

### ⑤ 为什么投机结果不能随便提交

乱序 CPU 中，投机执行的结果通常先进入内部结构：

```text
物理寄存器。
重排序缓冲区 ROB。
Store Buffer。
Load Queue。
```

只有确认路径正确后：

```text
结果才按程序顺序提交。
```

这样可以保证：

```text
架构状态看起来仍然像按顺序执行。
```

> [!tip] 记忆口诀
> **预测正确提前赚，预测错误全白干；投机可以先算，但确认后才能提交。**

# 10 | 分支预测器总结

> [!abstract] 核心结论
> 分支预测器的底层核心是“记录历史、预测方向、预测目标、支持投机执行”。它通过 BHT、BTB、RAS、局部/全局历史等结构，让 CPU 前端尽量沿正确路径持续取指，减少流水线停顿。

### ① 核心总结

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #27AE60; color: #333; margin-bottom: 10px;">
<strong>分支预测器底层总结：</strong>
<ul>
  <li><strong>作用：</strong> 预测控制流下一步走向，避免取指停顿。</li>
  <li><strong>方向预测：</strong> 判断分支是跳转还是不跳转。</li>
  <li><strong>目标预测：</strong> 判断跳转目标地址。</li>
  <li><strong>BHT：</strong> 记录分支历史方向。</li>
  <li><strong>两位计数器：</strong> 常用强弱状态稳定预测方向。</li>
  <li><strong>BTB：</strong> 缓存分支目标地址。</li>
  <li><strong>局部历史：</strong> 学习单个分支自己的规律。</li>
  <li><strong>全局历史：</strong> 学习多个分支之间的相关性。</li>
  <li><strong>RAS：</strong> 专门预测函数返回地址。</li>
  <li><strong>误预测：</strong> 清空错误路径，重新取指，浪费多个周期。</li>
</ul>
</div>
