# 01 | 高速缓存 Cache：夹在 CPU 和内存之间的高速缓冲层

> [!abstract] 核心结论
> 高速缓存 Cache 是 CPU 内部或附近的高速小容量存储器，用来保存最近可能会被再次访问的数据和指令。它的核心目的，是弥补 CPU 运算速度和内存访问速度之间的巨大差距。

### ① 底层原理：CPU 太快，内存太慢

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #2C3E50; color: #333; margin-bottom: 10px;">
<strong>Cache 的硬件本质：</strong>
<ul>
  <li><strong>Cache 是高速存储层：</strong> 它比 DRAM 内存更快，但容量更小、成本更高。</li>
  <li><strong>CPU 优先访问 Cache：</strong> 数据如果在 Cache 中，就不必去慢速内存读取。</li>
  <li><strong>Cache 保存的是内存副本：</strong> 它不是独立的新数据空间，而是内存中部分数据的临时副本。</li>
  <li><strong>Cache 以缓存行管理：</strong> CPU 通常不是一个字节一个字节缓存，而是一整块一整块搬运。</li>
  <li><strong>Cache 利用局部性：</strong> 程序经常访问刚访问过的数据，或附近的数据。</li>
</ul>
</div>

### ② 一句话理解

```text
高速缓存 = CPU 身边的小仓库。
常用数据先放小仓库里，CPU 不用每次都跑去远处内存取。
```

访问路径通常是：

```text
CPU Core
 ↓
L1 Cache
 ↓
L2 Cache
 ↓
L3 Cache
 ↓
内存 DRAM
```

如果 L1 命中：

```text
速度最快。
```

如果一路未命中到内存：

```text
CPU 等待时间明显变长。
```

> [!tip] 记忆口诀
> **CPU 快，内存慢；Cache 夹中间，常用数据先缓存。**

# 02 | L1、L2、L3：多级缓存层次

> [!abstract] 核心结论
> 现代 CPU 通常有多级缓存。L1 最小最快，L2 较大较慢，L3 更大但更慢，常被多个核心共享。越靠近 CPU 核心，速度越快、容量越小。

### ① L1 Cache

```text
L1 Cache：
最靠近 CPU 核心。
容量小。
速度最快。
通常分为 L1 指令缓存和 L1 数据缓存。
```

常见：

```text
L1 I-Cache：保存机器指令。
L1 D-Cache：保存数据。
```

### ② L2 Cache

```text
L2 Cache：
比 L1 大。
比 L1 慢。
通常每个核心独享一份。
```

作用：

```text
当 L1 没命中时，先查 L2。
如果 L2 命中，就不用去更慢的 L3 或内存。
```

### ③ L3 Cache

```text
L3 Cache：
容量更大。
速度比 L1 / L2 慢。
常被多个 CPU 核心共享。
```

作用：

```text
减少多个核心访问内存的次数。
作为核心之间共享数据的缓冲层。
```

### ④ 多级缓存结构示意

```text
Core 0 ─ L1 ─ L2 ┐
                 │
Core 1 ─ L1 ─ L2 ├── L3 Cache ── 内存
                 │
Core 2 ─ L1 ─ L2 ┘
```

### ⑤ 为什么不只做一个超大 L1

因为：

```text
容量越大，访问越慢。
端口越多，功耗越高。
离核心越近，时序要求越严格。
大缓存占芯片面积。
```

所以采用层级设计：

```text
小而快的 L1。
中等的 L2。
大而共享的 L3。
```

> [!tip] 记忆口诀
> **L1 小而快，L2 中间站，L3 大共享；越近越快，越大越慢。**

# 03 | 缓存行 Cache Line：Cache 搬运数据的基本单位

> [!abstract] 核心结论
> Cache 不是按单个变量管理，而是按缓存行管理。一次从内存加载数据时，CPU 通常会把目标地址附近的一整块数据一起搬入 Cache。

### ① 什么是缓存行

```text
Cache Line = Cache 和内存之间搬运数据的基本块。
```

常见大小：

```text
64 字节
```

也就是说：

```text
CPU 读取一个 int 变量时，可能会把它所在的 64 字节整块数据一起装入 Cache。
```

### ② 示例

假设数组：

```c
int arr[16];
```

如果：

```text
int 占 4 字节
缓存行大小 64 字节
```

那么一个缓存行刚好可以放：

```text
64 / 4 = 16 个 int
```

当 CPU 读取：

```c
arr[0]
```

可能同时把：

```text
arr[0] ~ arr[15]
```

所在的缓存行一起加载进 Cache。

### ③ 为什么数组连续访问快

```c
for (int i = 0; i < 16; i++) {
    sum += arr[i];
}
```

因为：

```text
读取 arr[0] 时，arr[1]、arr[2]、arr[3] 等可能已经在同一缓存行里。
后续访问直接命中 Cache。
```

### ④ 缓存行内部示意

```text
一个 64 字节缓存行：

┌────┬────┬────┬────┬────┬────┬────┬────┐
│数据│数据│数据│数据│数据│数据│数据│数据│ ...
└────┴────┴────┴────┴────┴────┴────┴────┘
```

Cache 管理时通常记录：

```text
Tag：这行来自内存哪个地址范围。
Valid：这行是否有效。
Dirty：这行是否被修改过。
Data：真正的数据内容。
```

### ⑤ 缓存行带来的副作用

因为按缓存行管理：

```text
两个变量即使不是同一个变量，
只要在同一个缓存行里，
也可能互相影响缓存行为。
```

这会引出：

```text
伪共享 False Sharing。
```

> [!tip] 记忆口诀
> **Cache 不是搬一个变量，而是搬一整行；数组连续，所以容易命中。**

# 04 | 局部性原理：Cache 为什么有效

> [!abstract] 核心结论
> Cache 之所以有效，是因为程序访问数据和指令通常有规律：刚访问过的东西可能很快再访问，访问某个地址后也很可能访问附近地址。

### ① 时间局部性

```text
时间局部性 = 最近访问过的数据，短时间内可能再次访问。
```

例子：

```c
for (int i = 0; i < 1000000; i++) {
    sum += x;
}
```

变量 `x` 被反复使用。

所以：

```text
x 一旦进入 Cache，后面多次访问都可能命中。
```

### ② 空间局部性

```text
空间局部性 = 访问某个地址后，很可能访问它附近的地址。
```

例子：

```c
for (int i = 0; i < n; i++) {
    sum += arr[i];
}
```

数组元素连续存放。

访问：

```text
arr[0]
```

之后很可能访问：

```text
arr[1]
arr[2]
arr[3]
```

### ③ 指令局部性

程序指令也有局部性：

```text
顺序执行时，下一条指令就在附近。
循环执行时，同一小段代码反复运行。
```

所以：

```text
I-Cache 可以缓存热点代码。
```

### ④ Cache 利用局部性的方式

```text
时间局部性：
保留最近访问过的数据。

空间局部性：
一次加载整个缓存行。

指令局部性：
缓存最近执行过的代码。
```

### ⑤ 局部性差会怎样

如果程序访问模式非常随机：

```text
每次访问都跳到很远位置。
很少重复。
很少访问附近数据。
```

那么：

```text
Cache 命中率下降。
内存访问变慢。
CPU 等待时间增加。
```

> [!tip] 记忆口诀
> **刚用过还会用，是时间局部性；用了这一个，附近也会用，是空间局部性。**

# 05 | Cache 命中与未命中

> [!abstract] 核心结论
> CPU 访问数据时，如果目标数据已经在 Cache 中，就是命中；如果不在 Cache 中，就要去更低层缓存或内存取，这就是未命中。Cache 命中率直接影响程序性能。

### ① Cache Hit 命中

```text
Cache Hit = 目标数据已经在当前缓存层中。
```

例如：

```text
CPU 查 L1 Cache。
发现目标地址所在缓存行就在 L1。
直接读取。
```

结果：

```text
访问速度很快。
```

### ② Cache Miss 未命中

```text
Cache Miss = 目标数据不在当前缓存层中。
```

例如：

```text
L1 没有。
去 L2 查。
L2 没有。
去 L3 查。
L3 没有。
最后去内存。
```

结果：

```text
访问延迟明显增加。
```

### ③ 未命中代价

未命中时需要：

```text
查找下一级缓存。
可能访问内存。
搬运整个缓存行。
更新 Cache 元数据。
可能替换旧缓存行。
可能写回脏数据。
```

所以：

```text
一次 Cache Miss 可能浪费很多 CPU 周期。
```

### ④ 命中率

```text
命中率 = Cache 命中的访问次数 / 总访问次数
```

例如：

```text
1000 次访问中 950 次命中。
命中率 = 95%
```

命中率越高：

```text
CPU 越少等待内存。
程序通常越快。
```

### ⑤ 三类常见 Miss

```text
冷启动 Miss：
第一次访问，Cache 中还没有。

容量 Miss：
Cache 太小，装不下工作集。

冲突 Miss：
多个地址映射到同一组，互相挤掉。
```

> [!tip] 记忆口诀
> **命中就近拿，未命中向下找；Miss 越多，CPU 等得越久。**

# 06 | Cache 映射方式：直接映射、全相联、组相联

> [!abstract] 核心结论
> 内存中的数据要放进 Cache，必须决定放在哪个位置。常见映射方式有直接映射、全相联和组相联。现代 CPU 多使用组相联，在速度、成本和冲突率之间折中。

### ① 直接映射

```text
Direct Mapped Cache
```

规则：

```text
一个内存块只能放到 Cache 中固定的一个位置。
```

优点：

```text
硬件简单。
查找速度快。
```

缺点：

```text
冲突严重。
两个常用地址如果映射到同一位置，会互相挤掉。
```

### ② 全相联

```text
Fully Associative Cache
```

规则：

```text
一个内存块可以放到 Cache 中任意位置。
```

优点：

```text
冲突最少。
空间利用灵活。
```

缺点：

```text
硬件复杂。
查找时需要同时比较很多 Tag。
功耗和面积较高。
```

### ③ 组相联

```text
Set Associative Cache
```

规则：

```text
Cache 分成很多组。
一个内存块先映射到某一组。
然后可以放在该组内任意一路。
```

例如：

```text
8 路组相联：
每个组里有 8 个位置可以放。
```

### ④ 组相联结构

```text
Cache
├── Set 0：Way0 Way1 Way2 ... Way7
├── Set 1：Way0 Way1 Way2 ... Way7
├── Set 2：Way0 Way1 Way2 ... Way7
└── ...
```

查找时：

```text
先根据地址找到 Set。
再在 Set 内比较多个 Way 的 Tag。
```

### ⑤ 为什么组相联常用

因为它折中：

```text
比直接映射冲突少。
比全相联成本低。
查找速度和硬件复杂度可控。
```

> [!tip] 记忆口诀
> **直接映射位置死，全相联随便放；组相联先找组，组里多路挑位置。**

# 07 | Cache 地址拆分：Tag、Index、Offset

> [!abstract] 核心结论
> CPU 用内存地址访问 Cache 时，会把地址拆成 Tag、Index 和 Offset。Index 用来找组，Tag 用来确认是不是目标缓存行，Offset 用来定位缓存行内具体字节。

### ① 地址拆分

一个内存地址可以拆成：

```text
┌────────────┬────────────┬────────────┐
│    Tag     │   Index    │   Offset   │
└────────────┴────────────┴────────────┘
```

含义：

```text
Tag：
标识这行数据来自内存哪个高位地址范围。

Index：
选择 Cache 中的哪一组。

Offset：
选择缓存行内部的哪个字节。
```

### ② Offset

如果缓存行大小是：

```text
64 字节
```

那么需要：

```text
6 bit
```

来表示行内偏移。

因为：

```text
2^6 = 64
```

所以地址最低 6 位通常用于：

```text
Offset
```

### ③ Index

Index 用来选择 Cache 的 Set。

如果 Cache 有：

```text
1024 个 Set
```

那么需要：

```text
10 bit
```

因为：

```text
2^10 = 1024
```

### ④ Tag

剩下的高位作为 Tag。

Cache 查找时：

```text
1. 用 Index 找到对应 Set。
2. 比较该 Set 中每一路的 Tag。
3. Tag 相等且 Valid 有效，就是命中。
4. 用 Offset 取出行内目标数据。
```

### ⑤ 为什么要 Tag

因为不同内存地址可能映射到同一个 Set。

Tag 用来区分：

```text
当前 Set 里这条缓存行，到底是不是我要的那一条。
```

没有 Tag：

```text
只知道位置相同，不知道内容来自哪里。
```

> [!tip] 记忆口诀
> **Index 找组，Tag 验身份，Offset 找行内位置。**

# 08 | 写策略：Write-Through 与 Write-Back

> [!abstract] 核心结论
> CPU 写数据时，Cache 必须决定什么时候把修改同步到下一级缓存或内存。写直达简单但流量大，写回效率高但需要 Dirty 位管理脏数据。

### ① 写直达 Write-Through

```text
Write-Through = 写 Cache 的同时，也写下一级缓存或内存。
```

优点：

```text
数据一致性简单。
下一级总是比较新。
实现逻辑相对直接。
```

缺点：

```text
写流量大。
频繁写会占用下一级带宽。
性能可能较低。
```

### ② 写回 Write-Back

```text
Write-Back = 先只写 Cache。
等缓存行被替换时，再把修改过的数据写回下一级。
```

优点：

```text
减少写流量。
多次修改同一缓存行，只需最后写回一次。
性能更好。
```

缺点：

```text
需要 Dirty 位。
一致性管理更复杂。
掉电或错误处理更麻烦。
```

### ③ Dirty 位

```text
Dirty = 脏位
```

含义：

```text
这条缓存行已经被 CPU 修改过，
和下一级缓存或内存中的副本不一致。
```

如果 Dirty = 1 的缓存行要被替换：

```text
必须先写回。
```

否则：

```text
修改会丢失。
```

### ④ 写分配 Write Allocate

写未命中时可以选择：

```text
Write Allocate：
先把目标缓存行加载到 Cache，再写。

No Write Allocate：
不加载到 Cache，直接写下一级。
```

### ⑤ 常见组合

高性能 CPU 常见：

```text
Write-Back + Write Allocate
```

因为：

```text
适合利用时间局部性。
如果刚写过，很可能马上还会读或写。
```

> [!tip] 记忆口诀
> **写直达马上同步，写回先记脏；脏行被赶走，必须先写回。**

# 09 | 替换策略与缓存一致性

> [!abstract] 核心结论
> 当 Cache 某一组已经满了，新缓存行要进来时，必须选择旧行替换出去。多核系统中，还必须通过缓存一致性协议保证多个核心看到的数据不矛盾。

### ① 替换策略

当目标 Set 已满：

```text
新缓存行要进入。
必须踢掉一个旧缓存行。
```

常见策略：

```text
LRU：最近最少使用。
Pseudo-LRU：近似 LRU，硬件更省。
Random：随机替换。
FIFO：先进先出。
```

### ② LRU 思想

```text
Least Recently Used
最近最少使用
```

含义：

```text
最久没被访问的缓存行，未来也可能较少用。
优先替换它。
```

### ③ 为什么不用完美 LRU

完美 LRU 需要精确记录访问顺序。

如果组相联路数很多：

```text
记录和更新成本高。
硬件复杂。
功耗增加。
```

所以硬件常用：

```text
近似 LRU。
```

### ④ 多核缓存一致性

多个核心都有自己的 Cache。

如果 Core 0 修改了某个缓存行：

```text
Core 1 如果也缓存了旧副本，就可能读到旧数据。
```

所以需要一致性协议，例如：

```text
MESI
MOESI
MESIF
```

### ⑤ MESI 状态

常见状态：

```text
M：Modified，已修改，和内存不同。
E：Exclusive，独占，只有自己有，和内存相同。
S：Shared，共享，多个缓存可能有。
I：Invalid，无效，不能使用。
```

### ⑥ 一致性消息

多核之间会传递：

```text
读请求。
写权限请求。
失效请求。
写回响应。
共享响应。
所有权转移。
```

这些消息会走：

```text
片上互联。
环形总线。
Mesh 网络。
多 CPU 互联。
```

> [!tip] 记忆口诀
> **Cache 满了要赶人，替换策略来决定；多核各有缓存，一致性协议防旧数据。**

# 10 | Cache 总结

> [!abstract] 核心结论
> 高速缓存的底层核心是“缓存行 + 局部性 + 命中判断 + 替换策略 + 写策略 + 一致性协议”。它让 CPU 大多数时候不必直接访问慢速内存，是现代处理器性能的关键基础。

### ① 核心总结

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #27AE60; color: #333; margin-bottom: 10px;">
<strong>高速缓存 Cache 底层总结：</strong>
<ul>
  <li><strong>本质：</strong> CPU 和内存之间的小容量高速存储层。</li>
  <li><strong>作用：</strong> 缓存最近或附近可能再次访问的数据和指令。</li>
  <li><strong>L1：</strong> 最小最快，通常分指令缓存和数据缓存。</li>
  <li><strong>L2：</strong> 比 L1 大，比 L1 慢，常为核心私有。</li>
  <li><strong>L3：</strong> 更大更慢，常被多个核心共享。</li>
  <li><strong>缓存行：</strong> Cache 管理和搬运数据的基本单位，常见 64 字节。</li>
  <li><strong>局部性：</strong> 时间局部性和空间局部性是 Cache 有效的根本原因。</li>
  <li><strong>命中 / 未命中：</strong> 命中快，未命中需要向下一级甚至内存查找。</li>
  <li><strong>写策略：</strong> 写直达简单，写回高效但要管理 Dirty 位。</li>
  <li><strong>一致性：</strong> 多核系统需要 MESI 等协议维护缓存副本一致。</li>
</ul>
</div>

### ② 最底层一句话

```text
高速缓存 = CPU 用缓存行形式保存内存热点副本，用更近、更快的小存储层减少访问内存的等待。
```
