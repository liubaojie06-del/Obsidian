# 01 | 内存管理单元 MMU：把虚拟地址翻译成物理地址

> [!abstract] 核心结论
> 内存管理单元 MMU 是 CPU 内部负责地址转换和内存保护的硬件。程序使用虚拟地址访问内存，MMU 根据页表把虚拟地址翻译成物理地址，并检查访问权限，保证进程隔离和系统安全。

### ① 底层原理：程序看到的地址不一定是真实内存地址

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #2C3E50; color: #333; margin-bottom: 10px;">
<strong>MMU 的硬件本质：</strong>
<ul>
  <li><strong>MMU 全称：</strong> Memory Management Unit，内存管理单元。</li>
  <li><strong>核心功能：</strong> 把虚拟地址转换成物理地址。</li>
  <li><strong>权限检查：</strong> 判断当前访问是否允许读、写、执行。</li>
  <li><strong>进程隔离：</strong> 让不同进程拥有各自独立的虚拟地址空间。</li>
  <li><strong>配合操作系统：</strong> 页表由操作系统维护，MMU 负责硬件级快速翻译。</li>
</ul>
</div>

### ② 一句话理解

```text
MMU = CPU 内部的地址翻译官。
```

程序以为自己访问的是：

```text
虚拟地址 0x400000
```

但真正访问内存时，MMU 可能把它翻译成：

```text
物理地址 0x1A300000
```

也就是说：

```text
程序看到的是虚拟地址。
内存芯片看到的是物理地址。
中间由 MMU 翻译。
```

> [!tip] 记忆口诀
> **程序发虚拟地址，MMU 查表翻译；权限通过才访问，权限不对就异常。**

# 02 | 虚拟地址与物理地址

> [!abstract] 核心结论
> 虚拟地址是程序看到的地址，物理地址是真实内存条上的地址。现代操作系统通常让每个进程都运行在自己的虚拟地址空间中，再由 MMU 映射到真实物理内存。

### ① 虚拟地址

```text
Virtual Address
虚拟地址
```

含义：

```text
程序代码中使用的地址。
指针变量里保存的地址。
CPU 执行 load / store 指令时首先产生的地址。
```

例如：

```c
int a = 10;
printf("%p\n", (void*)&a);
```

输出的地址通常是：

```text
虚拟地址。
```

### ② 物理地址

```text
Physical Address
物理地址
```

含义：

```text
真实 DRAM 物理内存中的位置编号。
```

内存控制器最终要访问的是：

```text
物理地址。
```

### ③ 为什么需要虚拟地址

虚拟地址带来的好处：

```text
1. 每个进程拥有独立地址空间。
2. 进程之间不能随便访问彼此内存。
3. 程序不需要关心真实内存放在哪里。
4. 支持内存分页和换页。
5. 支持共享库、内存映射文件、懒加载。
6. 支持权限保护和内核空间隔离。
```

### ④ 同一个虚拟地址可以对应不同物理地址

例如：

```text
进程 A 的虚拟地址 0x400000 → 物理地址 0x10000000
进程 B 的虚拟地址 0x400000 → 物理地址 0x22000000
```

两个进程看到的虚拟地址相同，但实际物理内存不同。

### ⑤ 同一个物理地址也可以映射到多个虚拟地址

例如共享库：

```text
多个进程都使用同一份 libc 代码。
```

操作系统可以把它们的虚拟地址映射到同一份物理页面：

```text
减少内存浪费。
提高共享效率。
```

> [!tip] 记忆口诀
> **虚拟地址给程序看，物理地址给内存看；同名虚拟地址，不一定是真同一块内存。**

# 03 | 分页机制：把内存切成固定大小的页

> [!abstract] 核心结论
> MMU 通常以“页”为单位进行地址映射。虚拟地址空间被切成虚拟页，物理内存被切成物理页框，页表记录虚拟页到物理页框的映射关系。

### ① 什么是页

```text
Page
页
```

含义：

```text
虚拟地址空间中的固定大小内存块。
```

常见页大小：

```text
4 KB
```

也可能有大页：

```text
2 MB
1 GB
```

### ② 什么是页框

```text
Page Frame
物理页框
```

含义：

```text
物理内存中固定大小的块。
```

如果页大小是 4KB：

```text
虚拟页大小 = 4KB
物理页框大小 = 4KB
```

### ③ 页表负责映射

```text
虚拟页号 → 物理页框号
```

例如：

```text
虚拟页 0x400 → 物理页框 0x1A300
```

页内偏移不变。

### ④ 地址拆分

一个虚拟地址可以拆成：

```text
┌──────────────┬──────────────┐
│ 虚拟页号 VPN  │ 页内偏移 Offset │
└──────────────┴──────────────┘
```

如果页大小是 4KB：

```text
4KB = 4096 = 2^12
```

所以：

```text
低 12 位是页内偏移。
高位是虚拟页号。
```

### ⑤ 翻译后结构

MMU 用虚拟页号查页表，得到物理页框号：

```text
┌──────────────┬──────────────┐
│ 物理页框 PPN  │ 页内偏移 Offset │
└──────────────┴──────────────┘
```

注意：

```text
页内偏移不需要翻译。
虚拟页号被替换成物理页框号。
```

> [!tip] 记忆口诀
> **地址高位查页表，低位页内偏移不变；虚拟页换物理页框。**

# 04 | 页表：虚拟地址到物理地址的映射表

> [!abstract] 核心结论
> 页表是操作系统维护的数据结构，用来记录每个虚拟页对应哪个物理页框，以及该页是否可读、可写、可执行、是否在内存中等状态。

### ① 页表项 PTE

```text
PTE = Page Table Entry
页表项
```

一个页表项通常包含：

```text
物理页框号。
有效位 Present / Valid。
读权限。
写权限。
执行权限。
用户 / 内核权限。
脏位 Dirty。
访问位 Accessed。
缓存属性。
全局页标志。
```

### ② 页表项示意

```text
┌────────────┬──────┬──────┬──────┬──────┬──────┐
│ 物理页框号  │ 有效位 │ 读权限 │ 写权限 │ 执行权 │ 状态位 │
└────────────┴──────┴──────┴──────┴──────┴──────┘
```

### ③ 有效位

```text
Valid / Present
```

含义：

```text
该虚拟页当前是否有有效映射。
```

如果有效位为 1：

```text
MMU 可以继续翻译。
```

如果有效位为 0：

```text
访问该页会触发缺页异常。
```

### ④ 权限位

常见权限：

```text
R：可读
W：可写
X：可执行
U：用户态可访问
S：内核态可访问
```

例如：

```text
代码页：可读、可执行、不可写。
数据页：可读、可写、不可执行。
内核页：用户态不可访问。
```

### ⑤ 页表由谁维护

```text
操作系统负责创建、修改、切换页表。
MMU 负责按照页表进行硬件翻译。
```

进程切换时：

```text
操作系统会切换当前页表根地址。
```

这样不同进程看到的虚拟地址空间就不同。

> [!tip] 记忆口诀
> **页表项不只存地址，还存权限和状态；操作系统管页表，MMU 负责查表。**

# 05 | 多级页表：为什么页表要分层

> [!abstract] 核心结论
> 虚拟地址空间很大，如果为每个虚拟页都直接准备一个连续大页表，会非常浪费内存。多级页表把页表分层，只为实际使用的地址范围分配页表页。

### ① 单级页表的问题

假设虚拟地址空间很大：

```text
64 位地址空间极其庞大。
```

如果每个虚拟页都要一个页表项：

```text
页表会巨大无比。
绝大多数地址空间可能根本没用。
```

### ② 多级页表思想

把虚拟页号拆成多段：

```text
┌──────┬──────┬──────┬──────┬────────┐
│ L4索引 │ L3索引 │ L2索引 │ L1索引 │ 页内偏移 │
└──────┴──────┴──────┴──────┴────────┘
```

每一级索引查一级页表。

### ③ 翻译过程

```text
1. 从页表根地址找到最高级页表。
2. 用 L4 索引找到下一级页表地址。
3. 用 L3 索引找到下一级页表地址。
4. 用 L2 索引找到下一级页表地址。
5. 用 L1 索引找到最终页表项。
6. 得到物理页框号。
7. 拼接页内偏移形成物理地址。
```

### ④ 多级页表的好处

```text
没用到的虚拟地址范围，不需要分配下级页表。
节省内存。
适合稀疏地址空间。
支持巨大虚拟地址范围。
```

### ⑤ 多级页表的代价

如果每次访问都真的走多级页表：

```text
一次内存访问可能变成多次内存访问。
性能很差。
```

所以需要：

```text
TLB 缓存最近的地址翻译结果。
```

> [!tip] 记忆口诀
> **虚拟空间太大，页表必须分层；用到哪段建哪段，没用就不浪费。**

# 06 | TLB：地址翻译缓存

> [!abstract] 核心结论
> TLB 是 MMU 附近的小型高速缓存，用来保存最近使用过的虚拟页到物理页框的翻译结果。没有 TLB，每次访问内存都要查多级页表，性能会很差。

### ① TLB 是什么

```text
TLB = Translation Lookaside Buffer
地址转换旁路缓存
```

它缓存：

```text
虚拟页号 VPN → 物理页框号 PPN
```

以及：

```text
权限位。
页面属性。
地址空间标识。
```

### ② TLB 命中

如果 CPU 访问某个虚拟地址：

```text
MMU 先查 TLB。
```

如果 TLB 中有对应虚拟页：

```text
直接得到物理页框号。
快速形成物理地址。
```

这叫：

```text
TLB Hit
TLB 命中。
```

### ③ TLB 未命中

如果 TLB 中没有对应翻译：

```text
TLB Miss
```

则需要：

```text
查页表。
得到页表项。
把翻译结果填入 TLB。
重新执行或继续访问。
```

### ④ 硬件 Page Walk

很多 CPU 有硬件页表遍历器：

```text
Page Walker
```

TLB Miss 时，它会自动按照页表格式查多级页表。

也有一些架构：

```text
TLB Miss 由操作系统异常处理程序负责填充。
```

### ⑤ 为什么 TLB 很关键

程序访问内存非常频繁。

如果每次都查多级页表：

```text
性能会崩。
```

TLB 命中时：

```text
地址翻译几乎和缓存访问并行完成。
```

因此：

```text
TLB 命中率对性能非常重要。
```

> [!tip] 记忆口诀
> **页表是地址字典，TLB 是常用翻译缓存；命中就快，未命中就查表。**

# 07 | 缺页异常 Page Fault

> [!abstract] 核心结论
> 当程序访问的虚拟页没有有效映射，或者权限不允许时，MMU 会触发缺页异常。操作系统接管后，决定是分配页面、从磁盘加载、修改权限，还是终止程序。

### ① 什么情况会缺页

常见情况：

```text
访问尚未分配的虚拟地址。
页面被换出到磁盘。
第一次访问懒分配页面。
访问内存映射文件中尚未加载的部分。
写入只读页面。
用户态访问内核页。
执行不可执行页面。
```

### ② Page Fault 流程

```text
1. CPU 发出虚拟地址访问。
2. MMU 查 TLB / 页表。
3. 发现无效或权限不允许。
4. MMU 触发缺页异常。
5. CPU 切换到内核异常处理程序。
6. 操作系统分析原因。
7. 如果合法，就建立映射并恢复执行。
8. 如果非法，就终止进程或发送信号。
```

### ③ 合法缺页

例如：

```c
char *p = malloc(1024 * 1024);
```

操作系统可能暂时只分配虚拟地址范围。

真正第一次写入时：

```text
发生缺页。
内核分配物理页。
建立页表映射。
程序继续执行。
```

这叫：

```text
按需分页 / 懒分配。
```

### ④ 非法缺页

例如：

```c
int *p = NULL;
*p = 10;
```

访问空指针附近地址。

操作系统发现：

```text
该地址没有合法映射。
```

于是：

```text
终止程序。
```

常见表现：

```text
Segmentation fault
段错误
```

### ⑤ 缺页不一定是错误

```text
合法缺页：
操作系统可以处理，程序继续运行。

非法缺页：
访问越界或权限错误，程序崩溃。
```

> [!tip] 记忆口诀
> **缺页不是一定出错；合法缺页补页面，非法缺页杀进程。**

# 08 | 权限保护：MMU 如何隔离进程和内核

> [!abstract] 核心结论
> MMU 不只翻译地址，还会检查访问权限。它能阻止用户程序访问内核内存、阻止写代码段、阻止执行数据页，从硬件层面保护操作系统和进程安全。

### ① 用户态和内核态

CPU 通常有不同特权级：

```text
用户态：
普通应用程序运行。

内核态：
操作系统内核运行。
```

用户态权限低，不能随便访问硬件和内核内存。

### ② 页级权限

页表项里可以设置：

```text
用户态是否可访问。
是否可读。
是否可写。
是否可执行。
是否可缓存。
```

例如：

```text
内核代码页：
用户态不可访问。

用户代码页：
可读可执行，不可写。

用户数据页：
可读可写，不可执行。
```

### ③ 写保护

代码段通常不可写：

```text
防止程序把自己的代码随便改掉。
防止攻击者注入代码。
```

如果程序尝试写代码页：

```text
MMU 触发权限异常。
```

### ④ NX / XD 位

```text
NX = No Execute
XD = Execute Disable
```

含义：

```text
该页面不可执行。
```

常用于：

```text
栈不可执行。
堆不可执行。
数据页不可执行。
```

这样可以降低代码注入攻击风险。

### ⑤ 进程隔离

每个进程有自己的页表。

进程 A 的虚拟地址：

```text
0x500000
```

可能映射到自己的物理页。

进程 B 无法直接访问 A 的页，因为：

```text
B 的页表中没有对应映射。
```

或者权限不允许。

> [!tip] 记忆口诀
> **MMU 不只翻译地址，还当门卫；能不能读写执行，都要看页表权限。**

# 09 | MMU、Cache 与地址翻译顺序

> [!abstract] 核心结论
> CPU 访问内存时，通常要同时处理地址翻译和缓存查找。不同架构可能使用虚拟索引、物理标签等设计。核心目标是在保证正确性的同时，让 Cache 访问尽量快。

### ① 访问内存的大致流程

```text
CPU 生成虚拟地址
      ↓
TLB 查询虚拟页号
      ↓
得到物理页框号
      ↓
形成物理地址
      ↓
访问 Cache / 内存
```

### ② 为什么要和 Cache 配合

Cache 需要知道：

```text
目标数据是否已经缓存。
```

但程序给出的是：

```text
虚拟地址。
```

而物理内存真正位置是：

```text
物理地址。
```

所以要解决：

```text
地址翻译和 Cache 查找如何高效配合。
```

### ③ 常见思路：VIPT

```text
VIPT = Virtually Indexed, Physically Tagged
虚拟索引，物理标签
```

含义：

```text
用虚拟地址的一部分先定位 Cache 组。
同时查 TLB 得到物理地址。
最后用物理 Tag 确认是否命中。
```

好处：

```text
TLB 查询和 Cache 索引可以并行。
降低访问延迟。
```

### ④ 为什么用物理 Tag

如果只用虚拟地址做 Tag，可能遇到：

```text
同一个物理页被多个虚拟地址映射。
不同进程有相同虚拟地址。
别名问题。
一致性复杂。
```

物理 Tag 可以更准确判断：

```text
Cache 中这条线是否真的来自目标物理地址。
```

### ⑤ TLB Miss 会影响 Cache 访问

如果 TLB 未命中：

```text
无法快速得到物理地址。
Cache 命中判断可能无法完成。
CPU 需要等待 Page Walk。
```

所以：

```text
TLB Miss 有时像 Cache Miss 一样昂贵。
```

> [!tip] 记忆口诀
> **程序给虚拟地址，Cache 最终认物理身份；TLB 命中，Cache 才能快判断。**

# 10 | MMU 总结

> [!abstract] 核心结论
> MMU 的底层核心是“虚拟地址翻译 + 页表查询 + TLB 加速 + 权限检查 + 异常处理”。它让现代操作系统能够实现虚拟内存、进程隔离、内存保护、按需分页和共享映射。

### ① 核心总结

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #27AE60; color: #333; margin-bottom: 10px;">
<strong>MMU 底层总结：</strong>
<ul>
  <li><strong>全称：</strong> Memory Management Unit，内存管理单元。</li>
  <li><strong>核心功能：</strong> 把虚拟地址翻译成物理地址。</li>
  <li><strong>虚拟地址：</strong> 程序看到和使用的地址。</li>
  <li><strong>物理地址：</strong> 内存控制器最终访问的真实地址。</li>
  <li><strong>分页：</strong> 以页为单位建立虚拟页到物理页框的映射。</li>
  <li><strong>页表：</strong> 操作系统维护的地址映射和权限表。</li>
  <li><strong>PTE：</strong> 页表项，保存物理页框号、有效位、权限位和状态位。</li>
  <li><strong>TLB：</strong> 地址翻译缓存，加速最近使用过的虚拟页翻译。</li>
  <li><strong>缺页异常：</strong> 页不存在或权限错误时触发，由操作系统处理。</li>
  <li><strong>权限保护：</strong> 阻止非法读写执行，实现进程隔离和内核保护。</li>
</ul>
</div>

### ② 最底层一句话

```text
MMU = CPU 内部根据页表把虚拟地址翻译成物理地址，并顺手检查访问权限的硬件单元。
```

### ③ 最终记忆口诀

> [!tip] 记忆口诀
> **虚拟地址给程序，物理地址给内存；MMU 查页表，TLB 加速查；权限不对就异常，缺页交给内核管。**
