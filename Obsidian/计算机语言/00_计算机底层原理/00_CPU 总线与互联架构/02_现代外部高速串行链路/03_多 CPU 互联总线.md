# 01 | 多 CPU 互联总线：让多个处理器组成一个共享系统

> [!abstract] 核心结论
> 多 CPU 互联总线不是简单把几个 CPU 接在一起，而是让多个处理器之间可以交换数据、访问内存、同步缓存状态、传递中断和协调任务。它是多路服务器、NUMA 架构和高性能计算平台的底层基础。

### ① 底层原理：多 CPU 互联的本质

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #2C3E50; color: #333; margin-bottom: 10px;">
<strong>多 CPU 互联的硬件本质：</strong>
<ul>
  <li><strong>连接多个处理器：</strong> 让 CPU0、CPU1、CPU2 等多个物理 CPU 可以组成一个统一系统。</li>
  <li><strong>传输数据和控制信息：</strong> 包括内存请求、缓存一致性消息、中断、同步信号等。</li>
  <li><strong>支持共享内存：</strong> 一个 CPU 可以访问自己本地内存，也可以访问其他 CPU 连接的远端内存。</li>
  <li><strong>维护缓存一致性：</strong> 多个 CPU 都有缓存，必须保证同一份数据不会出现互相矛盾的副本。</li>
  <li><strong>常见实现：</strong> Intel QPI / UPI、AMD HyperTransport / Infinity Fabric、服务器片间互联等。</li>
</ul>
</div>

### ② 一句话理解

```text
多 CPU 互联总线 = 多个物理 CPU 之间交换数据、请求、缓存状态和控制消息的高速通道。
```

可以类比：

```text
单 CPU 系统：
一个大脑管理自己的内存和设备。

多 CPU 系统：
多个大脑一起工作，中间必须有高速神经网络互相沟通。
```

> [!tip] 记忆口诀
> **单 CPU 自己算，多 CPU 要互联；数据要共享，缓存要一致，内存还分远近。**

# 02 | 为什么多 CPU 不能只靠普通总线连接

> [!abstract] 核心结论
> 早期多处理器系统可以共享一条总线，但 CPU 数量变多后，共享总线会成为严重瓶颈。现代多 CPU 系统通常使用点对点高速互联，而不是所有 CPU 抢同一条老式共享总线。

### ① 老式共享总线结构

```text
CPU0 ─┐
CPU1 ─┼── 共享总线 ── 内存
CPU2 ─┤
CPU3 ─┘
```

特点：

```text
所有 CPU 共用一条通道。
谁要访问内存或发消息，都要抢这条总线。
```

问题：

```text
CPU 数量越多，竞争越严重。
同一时刻只能有限设备使用总线。
带宽很快被吃满。
缓存一致性广播压力很大。
扩展性差。
```

### ② 现代点对点互联

```text
CPU0 ───── CPU1
 │          │
 │          │
CPU2 ───── CPU3
```

特点：

```text
CPU 之间通过高速链路直接或间接连接。
多个链路可以并行工作。
扩展性比共享总线更好。
```

### ③ 从“总线”到“互联网络”

现代多 CPU 互联更像：

```text
芯片之间的小型高速网络。
```

而不是：

```text
大家共用一根老式并行总线。
```

它会涉及：

```text
路由。
包传输。
缓存一致性协议。
链路训练。
流控。
错误检测。
重传。
拓扑设计。
```

> [!tip] 记忆口诀
> **共享总线像独木桥，CPU 多了就堵车；点对点互联像路网，多条路能同时跑。**

# 03 | NUMA：多 CPU 系统里的内存有远近之分

> [!abstract] 核心结论
> 多 CPU 系统通常是 NUMA 架构。每个 CPU 有自己的本地内存，访问本地内存最快；访问其他 CPU 旁边的远端内存，需要经过 CPU 互联通道，延迟更高。

### ① UMA 与 NUMA

**UMA：**

```text
Uniform Memory Access
统一内存访问
```

含义：

```text
所有 CPU 访问内存延迟大致一样。
```

**NUMA：**

```text
Non-Uniform Memory Access
非统一内存访问
```

含义：

```text
不同 CPU 访问不同位置内存，延迟不一样。
```

### ② NUMA 结构示意

```text
┌──────────────┐        ┌──────────────┐
│    CPU 0      │───────│    CPU 1      │
│              │       │              │
│ 本地内存 0    │       │ 本地内存 1    │
└──────────────┘        └──────────────┘
```

CPU0 访问本地内存：

```text
CPU0 → 本地内存0
路径短，延迟低。
```

CPU0 访问 CPU1 的内存：

```text
CPU0 → CPU互联通道 → CPU1 → 本地内存1
路径长，延迟高。
```

### ③ 为什么会有远端内存

因为每个 CPU 插槽通常都有自己的内存控制器：

```text
CPU0 连接一组内存条。
CPU1 连接另一组内存条。
```

操作系统把它们统一管理成一个大内存池。

但是底层路径不同：

```text
本地访问近。
跨 CPU 访问远。
```

### ④ NUMA 对性能的影响

如果线程在 CPU0 上运行，但频繁访问 CPU1 的内存：

```text
每次访问都要跨 CPU 互联。
延迟更高。
互联带宽也会被占用。
```

所以高性能程序常讲：

```text
线程绑定 CPU。
内存绑定 NUMA 节点。
让线程尽量访问本地内存。
```

> [!tip] 记忆口诀
> **NUMA 里内存有远近，本地内存最快，跨 CPU 访问更慢。**

# 04 | 多 CPU 互联传输什么

> [!abstract] 核心结论
> 多 CPU 互联通道上传输的不只是普通数据，还包括内存访问请求、缓存一致性消息、远端内存响应、中断、同步控制和错误管理信息。

### ① 远端内存访问请求

例如 CPU0 要读取 CPU1 本地内存中的数据：

```text
CPU0 发出读请求。
请求通过 CPU 互联通道到 CPU1。
CPU1 的内存控制器访问本地内存。
数据再通过互联通道返回 CPU0。
```

路径：

```text
CPU0
 ↓
CPU互联
 ↓
CPU1
 ↓
CPU1 本地内存
 ↓
CPU1
 ↓
CPU互联
 ↓
CPU0
```

### ② 缓存一致性消息

如果 CPU0 想修改某个缓存行，而 CPU1 缓存中也有副本：

```text
CPU0 必须通知 CPU1。
CPU1 需要失效或写回自己的缓存副本。
CPU0 才能安全修改。
```

这类消息也走 CPU 互联。

### ③ 中断和控制信息

多 CPU 系统中还会传：

```text
核间中断 IPI。
电源管理消息。
错误报告。
同步请求。
锁相关通信。
系统管理事件。
```

### ④ 数据包化传输

现代 CPU 互联通常不是裸电平表达固定地址线和数据线，而是类似网络一样传输包：

```text
请求包。
响应包。
数据包。
一致性包。
控制包。
错误包。
```

数据包可能包含：

```text
源 CPU ID
目标 CPU ID
请求类型
物理地址
缓存行状态
数据内容
校验信息
事务编号
```

> [!tip] 记忆口诀
> **多 CPU 互联不只搬数据，还搬请求、响应、一致性、中断和控制消息。**

# 05 | 缓存一致性：多 CPU 互联的核心难题

> [!abstract] 核心结论
> 多 CPU 系统中，每个 CPU 都有自己的缓存。如果多个 CPU 缓存了同一块内存数据，其中一个 CPU 修改后，其他 CPU 必须知道，否则就会读到旧数据。缓存一致性协议就是为了解决这个问题。

### ① 问题示例

假设内存中：

```text
x = 10
```

CPU0 缓存了 x：

```text
CPU0 Cache：x = 10
```

CPU1 也缓存了 x：

```text
CPU1 Cache：x = 10
```

现在 CPU0 修改：

```text
x = 20
```

如果 CPU1 不知道：

```text
CPU1 仍然可能读到 x = 10
```

这就出错了。

### ② 一致性协议要做什么

当 CPU0 要写某个缓存行：

```text
1. CPU0 请求获得该缓存行的独占权限。
2. 系统检查其他 CPU 是否有副本。
3. 如果有，通知它们失效或写回。
4. CPU0 获得修改权限。
5. CPU0 修改缓存行。
```

### ③ 常见缓存行状态

常见一致性协议会维护类似状态：

```text
M：Modified，已修改，和内存不一致。
E：Exclusive，独占，只有自己有。
S：Shared，共享，多个缓存可能有。
I：Invalid，无效，不能使用。
```

这类协议常被称为：

```text
MESI / MOESI 类协议。
```

### ④ 多 CPU 互联承担什么角色

多 CPU 互联负责传递：

```text
读取某缓存行的请求。
获得写权限的请求。
失效其他副本的消息。
写回脏数据的响应。
共享 / 独占状态变化。
```

### ⑤ 一致性流量会影响性能

如果多个 CPU 频繁修改同一缓存行：

```text
缓存行会在 CPU 之间来回转移。
互联流量暴涨。
程序性能下降。
```

典型问题：

```text
锁竞争。
原子变量热点。
伪共享。
多线程频繁写共享数据。
```

> [!tip] 记忆口诀
> **多 CPU 各有缓存，谁改数据要通知别人；一致性消息越多，互联压力越大。**

# 06 | 远端内存访问：一次跨 CPU 读写如何发生

> [!abstract] 核心结论
> 当 CPU 访问不属于自己本地控制器管理的内存时，请求必须跨过 CPU 互联通道。这个过程比本地内存访问更慢，也更消耗互联带宽。

### ① 本地内存访问

```text
CPU0 → CPU0 内存控制器 → CPU0 本地内存
```

特点：

```text
路径短。
延迟低。
不占用 CPU 间互联。
```

### ② 远端内存访问

```text
CPU0 → CPU互联 → CPU1 → CPU1 内存控制器 → CPU1 本地内存
```

然后数据返回：

```text
CPU1 本地内存 → CPU1 → CPU互联 → CPU0
```

特点：

```text
路径长。
延迟高。
占用 CPU 间互联带宽。
可能影响其他跨 CPU 流量。
```

### ③ 远端访问为什么慢

原因包括：

```text
多经过一个 CPU 插槽。
多经过互联链路。
需要路由和协议处理。
可能涉及缓存一致性检查。
可能排队等待链路空闲。
```

### ④ 程序如何避免远端访问

常见优化：

```text
线程绑定到固定 CPU / NUMA 节点。
内存分配在同一 NUMA 节点。
减少跨 CPU 共享写入。
把数据按 NUMA 节点分片。
让生产者和消费者尽量在同一节点。
```

### ⑤ Linux 中的 NUMA 相关工具

```bash
numactl --hardware
numactl --show
lscpu
```

常见用途：

```bash
numactl --cpunodebind=0 --membind=0 ./app
```

含义：

```text
让程序尽量运行在 NUMA 节点 0，并使用节点 0 的内存。
```

> [!tip] 记忆口诀
> **本地内存直达快，远端内存跨 CPU；跨得越多，延迟越高。**

# 07 | 互联拓扑：CPU 之间怎么连

> [!abstract] 核心结论
> 多 CPU 互联不只有一种连接方式。常见拓扑包括直连、环形、网格、全互联、交换结构等。拓扑决定了 CPU 之间的跳数、带宽、延迟和扩展能力。

### ① 双路直连

两个 CPU 最简单：

```text
CPU0 ───── CPU1
```

特点：

```text
结构简单。
CPU0 到 CPU1 只有一跳。
适合双路服务器。
```

### ② 四路环形

```text
CPU0 ─ CPU1
 │       │
CPU3 ─ CPU2
```

特点：

```text
每个 CPU 连接相邻 CPU。
访问远处 CPU 可能需要多跳。
链路数量比全互联少。
```

### ③ 全互联

```text
CPU0 ───── CPU1
 │  ╲     ╱  │
 │    ╲ ╱    │
 │    ╱ ╲    │
 │  ╱     ╲  │
CPU2 ───── CPU3
```

特点：

```text
任意两个 CPU 可直接通信。
延迟低。
链路数量多。
硬件成本高。
扩展困难。
```

### ④ Mesh / 网格

```text
CPU0 ─ CPU1 ─ CPU2
 │      │      │
CPU3 ─ CPU4 ─ CPU5
 │      │      │
CPU6 ─ CPU7 ─ CPU8
```

特点：

```text
适合更大规模。
路由更复杂。
平均跳数取决于位置。
```

### ⑤ 拓扑影响什么

```text
CPU 到 CPU 的延迟。
跨 CPU 内存访问速度。
一致性消息传播速度。
总互联带宽。
系统扩展能力。
故障容错能力。
```

> [!tip] 记忆口诀
> **拓扑决定路怎么走，跳数决定远不远；直连快，全互联贵，网格更能扩展。**

# 08 | 互联带宽与瓶颈

> [!abstract] 核心结论
> 多 CPU 互联通道有固定带宽。远端内存访问、缓存一致性、核间同步、DMA 和中断消息都会占用这条通道。跨 CPU 共享越多，互联越容易成为瓶颈。

### ① 哪些流量会占用 CPU 互联

```text
远端内存读写。
缓存行失效消息。
缓存行写回。
锁竞争导致的缓存行迁移。
核间中断。
跨 CPU 任务调度。
I/O 一致性消息。
远端 DMA 相关流量。
```

### ② 互联带宽不足会怎样

表现可能是：

```text
CPU 核心利用率看似很高，但性能上不去。
多线程扩展性变差。
增加 CPU 数量后性能不线性增长。
远端内存访问延迟升高。
锁竞争严重。
内存带宽测试跨节点性能下降。
```

### ③ 最容易压垮互联的场景

```text
多个 CPU 频繁写同一共享数据。
跨 NUMA 节点访问大数组。
线程迁移导致缓存数据跨 CPU 搬运。
内存分配和线程运行不在同一 NUMA 节点。
大量原子操作和全局锁。
```

### ④ 优化思路

```text
减少跨 CPU 共享写。
按 NUMA 节点划分数据。
每个 CPU 使用本地队列。
减少全局锁。
使用分片计数器。
避免伪共享。
绑定线程和内存。
```

### ⑤ 带宽和延迟都重要

```text
带宽：
决定单位时间能传多少数据。

延迟：
决定一次请求往返要等多久。
```

很多并发程序：

```text
不是单纯被带宽限制，而是被远端访问延迟和一致性同步拖慢。
```

> [!tip] 记忆口诀
> **跨 CPU 访问越多，互联越忙；共享写越频繁，一致性越拖后腿。**

# 09 | 多 CPU 互联与操作系统调度

> [!abstract] 核心结论
> 操作系统必须理解多 CPU 拓扑和 NUMA 距离。好的调度会尽量让线程在合适的 CPU 上运行，并让它访问本地内存，减少跨 CPU 互联流量。

### ① 线程迁移的问题

如果一个线程原来在 CPU0 上运行：

```text
它的数据可能已经在 CPU0 的缓存里。
```

突然被调度到 CPU1：

```text
CPU1 缓存没有这些数据。
可能需要从 CPU0 或内存重新取。
```

结果：

```text
缓存热数据丢失。
一致性流量增加。
互联压力增加。
性能下降。
```

### ② NUMA 感知调度

操作系统会尽量：

```text
让线程留在原来的 CPU 附近。
让线程运行在数据所在 NUMA 节点。
避免频繁跨节点迁移。
把相关线程放在合适位置。
```

### ③ First Touch 原则

很多系统中，内存页会分配到第一次访问它的 CPU 所属 NUMA 节点。

```text
哪个 CPU 第一次写这块内存，
这块内存就可能被分配到哪个 NUMA 节点。
```

所以初始化数据的位置很重要。

例如：

```text
线程在 CPU0 初始化大数组。
后续却让 CPU1 长期使用这个数组。
```

可能导致：

```text
CPU1 长期访问远端内存。
```

### ④ 程序员能做什么

```text
绑定线程。
分区初始化数据。
让每个线程初始化自己将来使用的数据。
使用 NUMA 分配库。
减少线程跨节点迁移。
```

### ⑤ 常见工具

```bash
lscpu
numactl --hardware
numastat
hwloc-ls
taskset
```

> [!tip] 记忆口诀
> **线程在哪跑，数据就该在哪放；调度乱迁移，缓存和互联都受罪。**

# 10 | 多 CPU 互联总线总结

> [!abstract] 核心结论
> 多 CPU 互联总线的底层核心是“处理器之间的高速一致性互联”。它让多个物理 CPU 能组成一个共享内存系统，但也带来 NUMA、远端访问、缓存一致性、互联瓶颈和操作系统调度问题。

### ① 核心总结

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #27AE60; color: #333; margin-bottom: 10px;">
<strong>多 CPU 互联总线底层总结：</strong>
<ul>
  <li><strong>本质：</strong> 多个物理 CPU 之间的高速通信通道。</li>
  <li><strong>作用：</strong> 传输远端内存请求、缓存一致性消息、中断、同步和控制信息。</li>
  <li><strong>现代形式：</strong> 更像点对点高速互联网络，而不是老式共享总线。</li>
  <li><strong>NUMA：</strong> 每个 CPU 有本地内存，访问远端内存更慢。</li>
  <li><strong>缓存一致性：</strong> 多个 CPU 缓存同一数据时，必须维护状态一致。</li>
  <li><strong>拓扑：</strong> 直连、环形、全互联、网格等结构会影响跳数和延迟。</li>
  <li><strong>瓶颈：</strong> 远端内存、共享写、锁竞争会增加互联压力。</li>
  <li><strong>调度：</strong> 操作系统需要让线程和内存尽量靠近。</li>
  <li><strong>优化：</strong> 减少跨 CPU 共享，绑定线程和内存，按 NUMA 节点拆分数据。</li>
</ul>
</div>

### ② 最底层一句话

```text
多 CPU 互联总线 = 多个处理器之间传递远端内存访问和缓存一致性消息的高速片间网络。
```

### ③ 最终记忆口诀
