# 01 | 网格互联架构：把芯片内部节点连成二维网络

> [!abstract] 核心结论
> 网格互联架构是一种把 CPU 核心、缓存切片、内存控制器、I/O 控制器等模块连接成二维网格的片上互连结构。数据不再只沿一条环路传递，而是可以在网格中按照路由规则逐跳转发。

### ① 底层原理：网格互联本质是片上网络

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #2C3E50; color: #333; margin-bottom: 10px;">
<strong>网格互联的硬件本质：</strong>
<ul>
  <li><strong>节点按二维网格排列：</strong> 每个节点可以是 CPU 核心、缓存、内存控制器或外设接口。</li>
  <li><strong>节点之间通过链路连接：</strong> 每个节点通常连接上、下、左、右相邻节点。</li>
  <li><strong>每个节点附近有路由器：</strong> 路由器负责接收、判断、转发数据包。</li>
  <li><strong>数据以包的形式传输：</strong> 包中包含源地址、目标地址、控制信息和数据内容。</li>
  <li><strong>网格本质是 NoC：</strong> Network on Chip，也就是片上网络。</li>
</ul>
</div>

### ② 二维 Mesh 示意

```text
┌─────┐───┌─────┐───┌─────┐───┌─────┐
│ N00 │   │ N01 │   │ N02 │   │ N03 │
└──┬──┘   └──┬──┘   └──┬──┘   └──┬──┘
   │         │         │         │
┌──┴──┐───┌──┴──┐───┌──┴──┐───┌──┴──┐
│ N10 │   │ N11 │   │ N12 │   │ N13 │
└──┬──┘   └──┬──┘   └──┬──┘   └──┬──┘
   │         │         │         │
┌──┴──┐───┌──┴──┐───┌──┴──┐───┌──┴──┐
│ N20 │   │ N21 │   │ N22 │   │ N23 │
└─────┘   └─────┘   └─────┘   └─────┘
```

可以理解为：

```text
每个节点都是一个路口。
数据包从源节点出发。
经过若干个路由器。
最终到达目标节点。
```

> [!tip] 记忆口诀
> **Mesh 像城市路网，节点是路口，链路是道路，数据包像车辆，路由器负责指路。**

# 02 | Mesh 节点：计算、缓存、内存控制器都可以挂在网格上

> [!abstract] 核心结论
> 网格互联中的节点不是单纯的“接线点”，而是芯片内部功能模块的接入点。每个节点通常连接一个本地模块和一个路由器。

### ① 节点里通常有什么

一个 Mesh 节点可能包含：

```text
CPU 核心
L1 / L2 缓存
共享 LLC 缓存切片
内存控制器
I/O 控制器
GPU / NPU / 加速器
路由器 Router
网络接口 Network Interface
```

### ② 节点的两个部分

```text
本地模块：
真正执行计算、存储或 I/O 的单元。

路由器：
负责把数据包转发到正确方向。
```

示意：

```text
┌────────────────────┐
│       节点 N11      │
│                    │
│  ┌──────────────┐  │
│  │ CPU / Cache  │  │
│  └──────┬───────┘  │
│         │           │
│  ┌──────▼───────┐  │
│  │   Router     │  │
│  └─┬──┬──┬──┬───┘  │
│    │  │  │  │      │
│   上 下 左 右 本地  │
└────────────────────┘
```

### ③ 路由器常见端口

```text
North：连接上方节点
South：连接下方节点
West：连接左侧节点
East：连接右侧节点
Local：连接本地核心或缓存
```

也就是：

```text
一个典型二维 Mesh 路由器可能有 5 个端口。
```

### ④ 节点的三种角色

```text
源节点：
生成请求或数据包。

中间节点：
只负责转发数据包。

目标节点：
接收并处理数据包。
```

例如：

```text
N00 要访问 N23 的缓存切片。

N00 是源节点。
N01、N02、N03、N13 是中间节点。
N23 是目标节点。
```

> [!tip] 记忆口诀
> **Mesh 节点不是单纯接点，而是本地模块加路由器；能计算、能缓存、也能转发。**

# 03 | 数据包：Mesh 中传输的不是裸数据，而是带头部的信息包

> [!abstract] 核心结论
> 网格互联中通常以数据包形式通信。数据包不只包含数据内容，还包含源节点、目标节点、操作类型、地址、控制位和校验信息。

### ① 数据包基本结构

```text
┌────────┬────────┬────────┬────────┬────────┬────────┐
│ 源节点 │ 目标节点 │ 操作类型 │ 地址信息 │ 数据内容 │ 校验/状态 │
└────────┴────────┴────────┴────────┴────────┴────────┘
```

### ② 常见字段

```text
源节点 ID：
表示这个包从哪里来。

目标节点 ID：
表示这个包要送到哪里。

操作类型：
读请求、写请求、响应、失效、写回等。

地址信息：
表示访问的内存地址或缓存行地址。

数据内容：
真正要传输的数据。

控制信息：
优先级、有效位、事务 ID、缓存状态等。
```

### ③ 请求包和响应包

**请求包：**

```text
Core 0 想读取某个地址。
它会发送一个读请求包。
```

示意：

```text
源节点：Core 0
目标节点：缓存切片 / 内存控制器
操作：Read
地址：0x1000
```

**响应包：**

```text
目标节点找到数据后，会返回一个响应包。
```

示意：

```text
源节点：缓存切片 / 内存控制器
目标节点：Core 0
操作：Read Response
数据：目标地址中的内容
```

### ④ 为什么要用包

因为 Mesh 是网络结构：

```text
数据要经过多个路由器。
中间节点需要知道往哪里转发。
目标节点需要知道怎么处理。
源节点需要知道响应属于哪个请求。
```

所以必须携带：

```text
目标信息。
操作信息。
事务信息。
数据内容。
```

> [!tip] 记忆口诀
> **Mesh 上传的不是裸数据，而是带地址、带目标、带操作类型的数据包。**

# 04 | 路由：数据包如何在网格中找到目标

> [!abstract] 核心结论
> 路由决定数据包在 Mesh 中每一跳往哪个方向走。最常见的简单策略是 XY 路由：先沿 X 方向走到目标列，再沿 Y 方向走到目标行。

### ① 坐标化理解

把每个节点看成二维坐标：

```text
N00 = (0,0)
N01 = (0,1)
N02 = (0,2)

N10 = (1,0)
N11 = (1,1)
N12 = (1,2)

N20 = (2,0)
N21 = (2,1)
N22 = (2,2)
```

如果源节点是：

```text
N00 = (0,0)
```

目标节点是：

```text
N22 = (2,2)
```

就需要从：

```text
行 0 列 0
```

走到：

```text
行 2 列 2
```

### ② XY 路由

XY 路由规则：

```text
先走 X 方向。
再走 Y 方向。
```

例如从 N00 到 N22：

```text
N00 → N01 → N02 → N12 → N22
```

也可以按另一种坐标定义理解为：

```text
先横向，再纵向。
```

### ③ 为什么 XY 路由简单

```text
每个路由器只需要比较当前坐标和目标坐标。
如果列还没对齐，就先横向走。
如果列对齐了，再纵向走。
```

伪逻辑：

```text
if 当前列 < 目标列:
    向右转发
else if 当前列 > 目标列:
    向左转发
else if 当前行 < 目标行:
    向下转发
else if 当前行 > 目标行:
    向上转发
else:
    交给本地节点
```

### ④ 自适应路由

除了固定 XY 路由，还可以使用自适应路由：

```text
根据链路拥塞情况选择路径。
避开繁忙区域。
避开故障节点。
选择更低延迟路径。
```

优点：

```text
灵活。
可以缓解拥塞。
容错能力更强。
```

缺点：

```text
硬件更复杂。
可能引入乱序。
可能更难避免死锁。
```

> [!tip] 记忆口诀
> **XY 路由先横后竖，规则简单不迷路；自适应路由更聪明，但控制更复杂。**

# 05 | 跳数 Hop：Mesh 延迟的核心来源之一

> [!abstract] 核心结论
> 在 Mesh 中，数据包从一个节点到相邻节点叫一跳。源节点到目标节点之间跳数越多，基础传输延迟越高。

### ① 什么是一跳

```text
一跳 Hop = 数据包从一个路由器转发到相邻路由器。
```

例如：

```text
N00 → N01
```

就是一跳。

```text
N00 → N01 → N02 → N12 → N22
```

就是四跳。

### ② 跳数计算

二维网格中，如果不考虑绕路，最短跳数通常是：

```text
跳数 = |源行 - 目标行| + |源列 - 目标列|
```

例如：

```text
源节点 N00 = (0,0)
目标节点 N22 = (2,2)

跳数 = |0 - 2| + |0 - 2|
     = 2 + 2
     = 4
```

### ③ 延迟组成

一次 Mesh 传输延迟大致来自：

```text
源节点注入延迟。
每一跳链路延迟。
每个路由器处理延迟。
排队等待延迟。
目标节点接收延迟。
协议处理延迟。
```

粗略理解：

```text
总延迟 ≈ 固定开销 + 跳数 × 每跳延迟 + 排队延迟
```

### ④ 为什么节点越多，延迟可能越大

因为网格变大后：

```text
最远节点之间距离变长。
平均跳数增加。
中间路由器数量增加。
排队和拥塞概率增加。
```

所以 Mesh 虽然比单一环更适合大规模扩展，但也不是无限免费扩展。

> [!tip] 记忆口诀
> **Mesh 传输看跳数，跳数越多路越远；路由器越多，基础延迟越高。**

# 06 | 链路、带宽与拥塞

> [!abstract] 核心结论
> Mesh 的每条链路都有固定宽度和频率，决定单条链路的理论带宽。但真实性能还会受路由、排队、热点访问和协议开销影响。

### ① 链路宽度

```text
链路宽度 = 相邻两个路由器之间一次能传多少 bit。
```

例如：

```text
128 bit 链路 = 一次传 16 字节
256 bit 链路 = 一次传 32 字节
512 bit 链路 = 一次传 64 字节
```

### ② 理论带宽

粗略公式：

```text
单链路理论带宽 = 每拍传输字节数 × 时钟频率
```

例如：

```text
链路宽度 = 256 bit = 32 字节
频率 = 1 GHz

理论带宽 = 32 字节 × 1,000,000,000
        = 32 GB/s
```

### ③ 实际带宽为什么更低

因为存在：

```text
包头开销。
路由控制信息。
流控信号。
空闲周期。
缓存一致性消息。
链路竞争。
路由器缓冲区满。
目标节点处理不及时。
```

### ④ 拥塞是什么

```text
拥塞 = 某些链路或路由器收到的数据包太多，处理不过来。
```

表现：

```text
数据包排队。
延迟上升。
吞吐下降。
局部区域变成瓶颈。
```

### ⑤ 热点访问

如果很多核心同时访问同一个目标：

```text
同一个内存控制器。
同一个缓存切片。
同一个锁变量。
同一个共享队列。
```

就会产生热点。

热点会导致：

```text
通往目标节点的链路拥塞。
目标路由器队列变长。
远端核心访问延迟明显上升。
```

### ⑥ 缓解拥塞的方法

```text
增加链路宽度。
提高 Mesh 频率。
增加多个内存控制器。
让缓存切片分布更均匀。
优化数据放置。
减少共享热点。
使用自适应路由。
使用更好的流控机制。
```

> [!tip] 记忆口诀
> **链路越宽路越大，频率越高车越快；热点一多就堵车，排队越长延迟越高。**

# 07 | 流控与缓冲区：避免数据包把路由器挤爆

> [!abstract] 核心结论
> Mesh 中每个路由器通常有输入缓冲区和流控机制。流控的作用是：当下游接收不了时，上游不能继续无脑发送，否则会丢包或堵死。

### ① 为什么需要缓冲区

数据包到达路由器时，下一条链路可能正在忙。

此时路由器需要：

```text
先把数据包暂存在缓冲区。
等链路空出来后再转发。
```

如果没有缓冲区：

```text
数据包可能直接丢失。
```

### ② 输入缓冲与输出缓冲

```text
输入缓冲：
数据包刚进入路由器时暂存。

输出缓冲：
数据包准备从某个方向发出前暂存。
```

### ③ 流控的作用

```text
流控 = 控制上游什么时候可以继续发送。
```

如果下游缓冲区快满：

```text
下游通知上游暂停。
```

如果下游缓冲区有空位：

```text
下游允许上游继续发送。
```

### ④ Credit-based 流控

一种常见思想：

```text
下游告诉上游自己还有多少空位。
每发送一个包，credit 减少。
下游释放缓冲后，credit 增加。
```

如果：

```text
credit = 0
```

说明：

```text
下游没有空位。
上游不能继续发送。
```

### ⑤ 背压 Backpressure

当某个节点堵住时：

```text
它会让上游暂停。
上游又可能让更上游暂停。
拥塞压力会沿路径反向传播。
```

这叫：

```text
背压 Backpressure。
```

### ⑥ 流控不好会怎样

```text
缓冲区溢出。
数据包丢失。
延迟暴涨。
全网吞吐下降。
严重时可能形成死锁。
```

> [!tip] 记忆口诀
> **路由器要有缓冲，缓冲满了要流控；下游没空位，上游别硬冲。**

# 08 | 死锁：数据包互相等待导致全网卡住

> [!abstract] 核心结论
> Mesh 网络中如果多个数据包各自占有资源，又互相等待对方释放资源，就可能形成死锁。路由算法和虚拟通道常用于避免死锁。

### ① 什么是死锁

死锁可以理解为：

```text
A 等 B。
B 等 C。
C 又等 A。
大家都不释放资源。
系统卡住。
```

在 Mesh 中表现为：

```text
多个数据包占住不同路由器缓冲。
每个包都需要下一个缓冲区才能继续。
但下一个缓冲区又被其他包占住。
```

### ② Mesh 中的死锁示意

```text
包 1 占用 A，等待 B。
包 2 占用 B，等待 C。
包 3 占用 C，等待 D。
包 4 占用 D，等待 A。
```

形成环形等待：

```text
A → B → C → D → A
```

### ③ 为什么会发生

常见原因：

```text
路由路径形成循环依赖。
缓冲资源有限。
流控导致上游暂停。
多个包互相占用通道。
```

### ④ 避免死锁的方法

```text
使用无死锁路由算法。
限制转向规则。
使用 XY 维序路由。
增加虚拟通道。
协议层避免循环等待。
合理设计请求 / 响应通道。
```

### ⑤ XY 路由为什么能帮助避免死锁

XY 路由固定顺序：

```text
先 X 方向，再 Y 方向。
```

它避免了随意转向造成的循环依赖。

例如不允许：

```text
先 Y 后 X 再回到 Y。
```

这样可以减少形成环形等待的机会。

### ⑥ 虚拟通道

虚拟通道 Virtual Channel：

```text
在同一条物理链路上划分多个逻辑通道。
不同类型的数据包使用不同虚拟通道。
```

作用：

```text
减少头阻塞。
打破循环依赖。
提高吞吐能力。
帮助协议避免死锁。
```

> [!tip] 记忆口诀
> **死锁就是大家互相等；路由定规矩，虚拟通道分车道。**

# 09 | Mesh 与缓存一致性：多核共享数据怎么保持一致

> [!abstract] 核心结论
> 在多核 CPU 中，Mesh 常用来传递缓存一致性消息。多个核心都有私有缓存时，必须通过一致性协议保证同一份数据不会出现互相矛盾的副本。

### ① 为什么需要缓存一致性

假设变量 `x` 在内存中：

```text
x = 10
```

Core 0 缓存了它：

```text
Core 0 Cache：x = 10
```

Core 1 也缓存了它：

```text
Core 1 Cache：x = 10
```

如果 Core 0 修改：

```text
x = 20
```

Core 1 如果不知道，就可能继续看到：

```text
x = 10
```

这就不一致。

### ② Mesh 上传递的一致性消息

常见消息：

```text
Read Request：读请求
Read Response：读响应
Invalidate：失效请求
Writeback：写回
Ownership Request：所有权请求
Snoop：监听请求
Ack：确认
```

### ③ 失效流程示例

```text
1. Core 0 想写某个缓存行。
2. Core 0 发送所有权请求。
3. 请求包通过 Mesh 到达目录节点或相关缓存节点。
4. 其他核心收到失效请求。
5. 其他核心把自己的副本标记为无效。
6. Core 0 获得写权限。
7. Core 0 修改数据。
```

### ④ 目录式一致性

大规模 Mesh 中常用目录思想：

```text
目录记录某个缓存行被哪些核心缓存。
```

当某个核心要写：

```text
不一定广播给所有核心。
只通知真正持有副本的核心。
```

好处：

```text
减少全网广播。
降低 Mesh 流量。
提高扩展性。
```

### ⑤ 缓存行粒度

一致性通常按缓存行管理，而不是按单个变量管理。

```text
缓存行常见大小：64 字节
```

这意味着：

```text
两个不同变量如果落在同一缓存行，
可能互相影响一致性流量。
```

这也会导致：

```text
伪共享 False Sharing。
```

### ⑥ 伪共享

两个核心修改不同变量：

```c
core0 修改 a
core1 修改 b
```

如果 `a` 和 `b` 在同一个缓存行：

```text
两个核心会反复让对方的缓存行失效。
即使它们没有真正共享同一个变量。
```

结果：

```text
Mesh 上一致性消息变多。
程序变慢。
```

> [!tip] 记忆口诀
> **多核各有缓存，修改要通知；Mesh 传一致性消息，目录减少全网广播。**

---

# 10 | Mesh 总结

> [!abstract] 核心结论
> 网格互联架构的底层核心是“节点 + 路由器 + 链路 + 数据包 + 路由算法 + 流控机制”。它比环形互联更适合大规模多核扩展，但也会带来路由、拥塞、死锁和一致性协议复杂度。

### ① 核心总结

<div style="background-color: #F7F9FC; padding: 15px; border-radius: 8px; border-left: 5px solid #27AE60; color: #333; margin-bottom: 10px;">
<strong>网格互联架构底层总结：</strong>
<ul>
  <li><strong>本质：</strong> 芯片内部的二维片上网络。</li>
  <li><strong>节点：</strong> CPU 核心、缓存切片、内存控制器、I/O 控制器等。</li>
  <li><strong>路由器：</strong> 每个节点附近负责收包、判断、转发。</li>
  <li><strong>链路：</strong> 相邻节点之间的数据通道。</li>
  <li><strong>数据包：</strong> 携带源节点、目标节点、操作类型、地址和数据。</li>
  <li><strong>路由算法：</strong> 决定数据包每一跳往哪里走。</li>
  <li><strong>跳数：</strong> 影响传输延迟。</li>
  <li><strong>流控：</strong> 防止下游缓冲区被挤爆。</li>
  <li><strong>死锁：</strong> 多个包互相等待时可能全网卡住。</li>
  <li><strong>缓存一致性：</strong> 多核共享数据时需要通过 Mesh 传递一致性消息。</li>
</ul>
</div>

### ② 最底层一句话

```text
Mesh 互联 = 芯片内部由路由器和链路组成的二维网络，数据包按路由规则逐跳转发。
```

### ③ 最终记忆口诀

> [!tip] 记忆口诀
> **Mesh 像城市路网，节点是路口，链路是道路；数据包带着目的地，路由器逐跳转发。路多能扩展，车多也会堵，规则不好还会死锁。**
