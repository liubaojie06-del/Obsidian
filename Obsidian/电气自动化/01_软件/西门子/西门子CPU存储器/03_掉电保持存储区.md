# 01 掉电保持存储区总览

> [!info] 核心结论
> **掉电保持存储区** 是PLC中专门用于保存断电后仍需要保留数据的区域。  
> 简单理解：设备断电再上电后，某些重要数值不会清零，而是从掉电前的值继续使用。

## ① 分类/公式/规则

> [!warning] 基本公式
> **掉电保持存储区 = 断电后仍保留的变量区域**

## ② 参考表

| 项目 | 说明 |
|---|---|
| **核心作用** | 断电后保存关键变量值 |
| **常见名称** | Retain区、保持区、掉电保持区 |
| **保存对象** | 累计值、配方号、参数、运行小时、保养计时 |
| **不适合保存** | 启动命令、点动命令、输出控制位、临时变量 |
| **常见区域** | 保持DB、保持M区、保持变量 |
| **风险点** | 启动清零、下载覆盖、容量不足、误保持命令位 |
| **验证方法** | 修改变量值 → 断电 → 上电 → 检查是否保持 |

## ③ 计算/选型示例

- 已知：  
  累计产量为3800件，PLC断电后重新上电。
- 公式：  
  **累计产量需要保持 → 放入掉电保持存储区**
- 计算：  
  若设置正确，上电后累计产量仍为3800。
- 结论：  
  累计产量这类数据应配置到掉电保持存储区。

> [!tip] 记忆口诀
> 掉电后还要记住的数据，就放掉电保持区。

# 02 掉电保持存储区的作用

> [!info] 核心结论
> 掉电保持存储区用于保证设备断电、重启或短暂停电后，关键生产数据和工艺状态不会丢失。

## ① 分类/公式/规则

> [!warning] 作用公式
> **设备恢复可靠 = 关键数据掉电保持 + 上电后正确读取**

## ② 参考表

| 作用 | 示例 |
|---|---|
| **保存累计产量** | 总产量、班次产量 |
| **保存当前配方** | 当前产品号、配方编号 |
| **保存工艺参数** | 速度、温度、压力、位置设定 |
| **保存运行小时** | 电机运行时间、设备运行时间 |
| **保存保养计时** | 到期提醒、维护周期 |
| **保存用户设置** | 语言、单位、界面选择 |
| **支持断电恢复** | 上电后恢复到合理状态 |

## ③ 计算/选型示例

- 已知：  
  电机运行小时为999小时，1000小时需要保养。
- 公式：  
  **运行小时必须掉电保持**
- 计算：  
  若断电后清零，保养提醒会失效。
- 结论：  
  运行小时和保养计时必须放入掉电保持存储区。

> [!tip] 记忆口诀
> 统计、配方、参数、维护时间，通常都要掉电保持。

# 03 掉电保持存储区与保持存储器

> [!info] 核心结论
> **保持存储器** 是功能概念，**掉电保持存储区** 是具体被配置为保持的变量范围或数据区域。

## ① 分类/公式/规则

> [!warning] 区分公式
> **保持存储器 = 能保持数据的存储资源**  
> **掉电保持存储区 = 被指定为保持的具体区域**

## ② 参考表

| 对比项 | 保持存储器 | 掉电保持存储区 |
|---|---|---|
| 含义 | 断电保持能力或资源 | 具体配置出来的保持范围 |
| 关注点 | 有没有保持能力 | 哪些变量被保持 |
| 示例 | CPU支持Retain | DB10、M20~M50设置保持 |
| 风险 | 容量有限 | 配置错误会导致数据丢失 |
| 调试重点 | 看CPU能力 | 看变量属性和保持范围 |
| 常见问题 | 保持容量不足 | 变量未放入保持区 |

## ③ 计算/选型示例

- 已知：  
  CPU支持保持功能，但某个参数断电后仍清零。
- 公式：  
  **有保持能力 ≠ 该变量已保持**
- 计算：  
  需要检查该变量是否被放入掉电保持存储区。
- 结论：  
  CPU支持保持只是前提，变量必须实际配置保持才有效。

> [!tip] 记忆口诀
> 有保持存储器，不代表所有变量都会保持。

# 04 掉电保持存储区保存什么

> [!info] 核心结论
> 掉电保持存储区应保存断电后仍有意义、丢失会影响生产、维护、追溯或工艺的数据。

## ① 分类/公式/规则

> [!warning] 保存判断规则
> **断电后仍要继续使用的数据 → 保存到掉电保持区**

## ② 参考表

| 建议保存 | 原因 |
|---|---|
| **累计产量** | 生产统计需要 |
| **累计运行时间** | 设备寿命和保养需要 |
| **当前配方号** | 上电后知道当前产品 |
| **工艺设定参数** | 防止参数恢复默认 |
| **保养计数** | 防止维护周期丢失 |
| **动作累计次数** | 判断气缸、阀门寿命 |
| **用户设置** | 语言、单位、显示偏好 |

## ③ 计算/选型示例

- 已知：  
  当前配方号为12，断电后设备需要继续生产同一产品。
- 公式：  
  **当前配方号需要保持**
- 计算：  
  上电后仍显示当前配方号12。
- 结论：  
  当前配方号适合放入掉电保持存储区。

> [!tip] 记忆口诀
> 断电后还要用，保持区里放。

# 05 不建议放入掉电保持区的数据

> [!info] 核心结论
> 会导致误动作、误启动、误复位或状态混乱的数据，不建议放入掉电保持存储区。

## ① 分类/公式/规则

> [!warning] 禁忌规则
> **命令位、输出位、临时位通常不保持**

## ② 参考表

| 不建议保持 | 风险 |
|---|---|
| **启动命令** | 上电后可能误启动 |
| **停止命令** | 状态逻辑混乱 |
| **复位命令** | 上电后可能误清报警 |
| **点动命令** | 上电后可能误动作 |
| **电磁阀输出位** | 可能误动作 |
| **临时中间变量** | 重启后无实际意义 |
| **一次性触发脉冲** | 应上电清零 |

## ③ 计算/选型示例

- 已知：  
  气缸伸出命令断电前为1，且被设置为掉电保持。
- 公式：  
  **动作命令保持 → 上电误动作风险**
- 计算：  
  上电后命令仍为1，气缸可能继续伸出。
- 结论：  
  动作命令位应上电清零，不应掉电保持。

> [!tip] 记忆口诀
> 参数可保持，命令要清零。

# 06 掉电保持DB

> [!info] 核心结论
> 掉电保持DB是把数据块中的变量设置为断电保持，适合集中保存参数、累计值、配方状态和维护数据。

## ① 分类/公式/规则

> [!warning] 保持DB公式
> **掉电保持DB = DB变量 + Retain属性 + 启动不覆盖**

## ② 参考表

| 保持DB内容 | 示例 |
|---|---|
| **配方状态** | CurrentRecipeNo |
| **工艺参数** | Speed_SP、Temp_SP |
| **累计值** | TotalCount、RunHour |
| **保养数据** | MaintenanceTime、ActionCount |
| **生产状态** | BatchNo、ProductType |
| **用户设置** | LanguageIndex |
| **系统设置** | UnitType、DisplayMode |

## ③ 计算/选型示例

- 已知：  
  保持DB中有60个Real参数，每个Real占4字节。
- 公式：  
  **容量 = 60 × 4**
- 计算：  
  容量 = **240字节**
- 结论：  
  使用保持DB时要计算保持容量，并保留余量。

> [!tip] 记忆口诀
> 参数多、结构复杂，用保持DB集中管理。

# 07 掉电保持M区

> [!info] 核心结论
> 掉电保持M区适合保存少量简单标志和数值，但不建议把复杂参数全部堆在M区。

## ① 分类/公式/规则

> [!warning] M区规则
> **少量简单状态可用保持M区，复杂数据优先用DB**

## ② 参考表

| 保持M区用途 | 示例 |
|---|---|
| **简单状态位** | 上次模式选择 |
| **少量计数值** | 小型计数 |
| **语言编号** | 当前语言索引 |
| **设备选择号** | 当前工位选择 |
| **简单锁存位** | 需要保持的提示状态 |
| **调试选择状态** | 上次选择页面或对象 |
| **小项目参数** | 少量简单设定 |

## ③ 计算/选型示例

- 已知：  
  M30.0表示上次选择自动模式，需要断电后保留。
- 公式：  
  **M30.0设置为保持 → 上电后仍保留状态**
- 计算：  
  PLC重启后仍可读取该模式标志。
- 结论：  
  少量简单状态可使用掉电保持M区。

> [!tip] 记忆口诀
> M区适合小标志，DB适合大数据。

# 08 掉电保持与初始化

> [!info] 核心结论
> 掉电保持区的数据可能被启动程序覆盖，所以启动初始化逻辑必须谨慎设计。

## ① 分类/公式/规则

> [!warning] 初始化规则
> **保持数据不能在每次启动时无条件清零或赋默认值**

## ② 参考表

| 启动处理 | 结果 |
|---|---|
| **无条件清零保持变量** | 保持失效 |
| **无条件写默认参数** | 掉电值被覆盖 |
| **首次运行初始化** | 合理 |
| **人工恢复默认** | 合理，但要权限确认 |
| **命令位上电清零** | 合理 |
| **累计值上电清零** | 通常不合理 |
| **参数上电恢复上次值** | 通常合理 |

## ③ 计算/选型示例

- 已知：  
  TotalCount设置了掉电保持，但启动程序每次执行TotalCount := 0。
- 公式：  
  **保持变量 + 启动清零 = 数据仍会丢**
- 计算：  
  上电后TotalCount被程序清为0。
- 结论：  
  掉电保持区必须避免被启动逻辑误覆盖。

> [!tip] 记忆口诀
> 保持区能记住，但程序清零照样忘。

# 09 掉电保持与下载覆盖

> [!info] 核心结论
> PLC下载程序或DB时，可能会把当前保持数据覆盖为初始值，特别是下载数据块时要注意。

## ① 分类/公式/规则

> [!warning] 下载规则
> **下载DB初始值可能覆盖当前实际值**

## ② 参考表

| 场景 | 风险 |
|---|---|
| **下载保持DB** | 当前值可能被初始值覆盖 |
| **修改DB结构** | 数据可能重新初始化 |
| **恢复默认工程** | 参数可能回到默认值 |
| **在线修改变量结构** | 变量偏移可能变化 |
| **下载前未上传当前值** | 现场参数丢失 |
| **配方参数在PLC中保持** | 下载时尤其要谨慎 |
| **累计值在DB中保持** | 可能被覆盖清零 |

## ③ 计算/选型示例

- 已知：  
  现场Speed_SP为500，工程DB初始值为300，下载DB后Speed_SP变为300。
- 公式：  
  **下载初始值覆盖当前值**
- 计算：  
  500被工程初始值300替换。
- 结论：  
  下载前应确认是否保留当前值，必要时先上传或备份参数。

> [!tip] 记忆口诀
> 断电不丢，不代表下载不覆盖。

# 10 掉电保持容量

> [!info] 核心结论
> 掉电保持存储区容量有限，保持变量过多或大数组放入保持区，可能导致容量超限。

## ① 分类/公式/规则

> [!warning] 容量公式
> **保持区剩余容量 = 保持总容量 - 已配置保持数据容量**

## ② 参考表

| 占用来源 | 说明 |
|---|---|
| **保持DB** | 参数、累计值、结构体 |
| **保持M区** | 标志位、简单状态 |
| **保持数组** | 元素多时占用明显 |
| **配方数据** | 产品和参数越多，占用越大 |
| **运行小时数据** | 多设备分别计时会增加占用 |
| **动作次数统计** | 多机构寿命统计会增加占用 |
| **历史缓存** | 不建议大量放保持区 |

## ③ 计算/选型示例

- 已知：  
  掉电保持区总容量8KB，计划保持两个DB，每个5KB。
- 公式：  
  **总占用 = 5KB + 5KB**
- 计算：  
  总占用 = **10KB > 8KB**
- 结论：  
  保持容量不足，应减少保持数据或更换更大容量CPU。

> [!tip] 记忆口诀
> 保持区不是仓库，关键数据才进去。

# 11 掉电保持与累计值

> [!info] 核心结论
> 累计产量、累计运行小时、动作次数、故障次数等累计值通常需要放入掉电保持存储区。

## ① 分类/公式/规则

> [!warning] 累计公式
> **新累计值 = 原累计值 + 本次增量**

## ② 参考表

| 累计数据 | 用途 |
|---|---|
| **总产量** | 生产统计 |
| **班次产量** | 班次报表 |
| **设备运行小时** | 维护保养 |
| **电机运行小时** | 寿命管理 |
| **气缸动作次数** | 易损件维护 |
| **报警发生次数** | 故障分析 |
| **累计能耗** | 能源统计 |

## ③ 计算/选型示例

- 已知：  
  断电前动作次数为30000次，本次运行新增120次。
- 公式：  
  **新动作次数 = 30000 + 120**
- 计算：  
  新动作次数 = **30120次**
- 结论：  
  动作次数应掉电保持，否则寿命统计不准确。

> [!tip] 记忆口诀
> 累计值不保持，统计就断档。

# 12 掉电保持与配方

> [!info] 核心结论
> 当前配方号、当前产品型号、PLC当前运行参数通常需要保持，但完整配方库是否放入PLC保持区要根据系统架构决定。

## ① 分类/公式/规则

> [!warning] 配方保持规则
> **当前使用状态要保持，完整配方库可由HMI或数据库管理**

## ② 参考表

| 配方数据 | 是否建议保持 |
|---|---|
| **当前配方号** | 建议保持 |
| **当前产品型号** | 建议保持 |
| **当前运行参数** | 通常建议保持 |
| **临时选中配方号** | 不一定需要 |
| **完整配方库** | 视容量和架构决定 |
| **下载完成标志** | 视逻辑决定 |
| **配方错误状态** | 视报警要求决定 |

## ③ 计算/选型示例

- 已知：  
  设备断电前正在生产产品B，配方号为2。
- 公式：  
  **当前配方号保持 → 上电后仍知道产品B**
- 计算：  
  上电后CurrentRecipeNo仍为2。
- 结论：  
  当前配方号应放入掉电保持存储区。

> [!tip] 记忆口诀
> 上次生产什么产品，PLC最好能记住。

# 13 掉电保持与报警锁存

> [!info] 核心结论
> 某些重要报警锁存可以掉电保持，但普通状态报警通常可以根据当前条件重新判断。

## ① 分类/公式/规则

> [!warning] 报警保持规则
> **重要安全事件可保持，普通当前状态可重新判断**

## ② 参考表

| 报警类型 | 建议 |
|---|---|
| **急停记录** | 可保持或由HMI历史记录保存 |
| **安全门报警** | 视安全要求决定 |
| **伺服故障锁存** | 常需要保留到人工复位 |
| **气压低报警** | 可按当前气压重新判断 |
| **通讯中断报警** | 通常按当前连接状态判断 |
| **普通提示报警** | 通常不需要保持 |
| **报警历史记录** | 建议HMI/SCADA归档 |

## ③ 计算/选型示例

- 已知：  
  伺服故障发生后断电，上电后希望仍提示需要检查驱动器。
- 公式：  
  **伺服故障锁存可保持**
- 计算：  
  故障锁存位保持为1，直到维修人员复位。
- 结论：  
  重要故障锁存是否保持，要根据工艺和安全要求决定。

> [!tip] 记忆口诀
> 当前状态可重算，重要故障要留痕。

# 14 掉电保持与上电安全

> [!info] 核心结论
> 掉电保持区能保存数据，但上电后设备是否允许继续运行，必须由PLC安全逻辑判断，不能只依赖保持状态。

## ① 分类/公式/规则

> [!warning] 上电安全公式
> **安全恢复 = 保持数据 + 当前输入状态 + 安全联锁 + 人工确认**

## ② 参考表

| 上电检查 | 说明 |
|---|---|
| **急停状态** | 是否释放 |
| **安全门状态** | 是否关闭 |
| **气压液压** | 是否正常 |
| **伺服状态** | 是否无故障且已回零 |
| **当前配方** | 是否有效 |
| **动作命令** | 上电应清零 |
| **运行允许** | 需要重新判断 |

## ③ 计算/选型示例

- 已知：  
  断电前设备自动运行标志为1，上电后该标志仍保持为1。
- 公式：  
  **自动运行标志保持 ≠ 允许直接自动启动**
- 计算：  
  上电后仍需检查安全门、急停、气压、伺服和人工启动确认。
- 结论：  
  掉电保持不能绕过上电安全流程。

> [!tip] 记忆口诀
> 数据可以保持，动作不能自动乱恢复。

# 15 掉电保持常见故障

> [!info] 核心结论
> 掉电保持常见问题包括变量未保持、启动程序清零、下载覆盖、保持容量不足、保持范围配置错误、硬件电池或存储介质异常。

## ① 分类/公式/规则

> [!warning] 故障公式

# 18 掉电保持存储区快速检查表

> 掉电保持存储区检查的目标是确保关键数据断电不丢、启动不清、下载不覆盖、命令不误保持、容量不超限。

> [!warning] 最终检查公式
