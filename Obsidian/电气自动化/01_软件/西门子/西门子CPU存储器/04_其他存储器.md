# 01 其他存储器总览

> [!info] 核心结论
> **其他存储器** 是指PLC中除装载存储器、工作存储器、保持存储器、系统存储器以外，用于 **固件保存、诊断记录、通讯缓存、数据日志、参数备份、存储卡扩展和特殊功能支持** 的存储区域或存储介质。

## ① 分类/公式/规则

> [!warning] 基本公式
> **其他存储器 = 固件存储 + 诊断缓冲 + 通讯缓冲 + 数据日志 + 存储卡 + 参数备份 + 特殊功能区**

## ② 参考表

| 类型 | 作用 |
|---|---|
| **固件存储器** | 保存CPU系统固件 |
| **存储卡** | 保存工程、程序、数据、备份 |
| **诊断缓冲区** | 保存CPU报警、错误、事件记录 |
| **通讯缓冲区** | 临时保存通讯收发数据 |
| **数据日志区** | 保存采集数据、历史记录 |
| **参数存储区** | 保存模块参数、设备参数 |
| **系统备份区** | 保存恢复、升级或系统配置数据 |

## ③ 计算/选型示例

- 已知：  
  PLC运行时出现通讯中断、模块报警和程序下载失败。
- 公式：  
  **故障排查 = 诊断缓冲区 + 通讯状态 + 存储空间检查**
- 计算：  
  先查看诊断缓冲区，再检查存储卡和通讯缓冲状态。
- 结论：  
  其他存储器虽然不直接等同于用户程序区，但对诊断、维护和可靠运行很重要。

> [!tip] 记忆口诀
> 程序靠装载，运行靠工作，数据靠保持，诊断和缓存就看其他存储区。

# 02 固件存储器

> [!info] 核心结论
> 固件存储器用于保存PLC CPU自身运行所需的系统程序、固件和底层功能，是CPU能正常启动和管理硬件的基础。

## ① 分类/公式/规则

> [!warning] 固件规则
> **CPU能启动 = 固件完整 + 硬件正常 + 系统配置有效**

## ② 参考表

| 项目 | 说明 |
|---|---|
| **固件 Firmware** | CPU内部系统软件 |
| **启动管理** | 支持CPU上电启动 |
| **通讯服务** | 支持以太网、PROFINET、Modbus等 |
| **诊断服务** | 支持故障记录和诊断 |
| **系统功能** | 支持时间、任务、中断、存储管理 |
| **升级风险** | 升级中断可能导致CPU异常 |
| **版本兼容** | 工程软件和硬件模块需兼容固件版本 |

## ③ 计算/选型示例

- 已知：  
  工程软件要求CPU固件版本不低于V4.5，但现场CPU为V4.2。
- 公式：  
  **固件版本不足 → 功能可能不可用**
- 计算：  
  某些新指令、新模块或通讯功能可能无法下载或运行。
- 结论：  
  需要确认固件版本是否满足项目要求。

> [!tip] 记忆口诀
> 固件是CPU自己的系统程序，不是用户写的PLC程序。

# 03 存储卡

> [!info] 核心结论
> 存储卡是PLC常见的外部存储介质，可用于保存程序、工程数据、系统数据、配方、日志或固件升级文件。

## ① 分类/公式/规则

> [!warning] 存储卡规则
> **存储卡可用 = 型号匹配 + 容量足够 + 文件完整 + 格式正确**

## ② 参考表

| 用途 | 说明 |
|---|---|
| **保存用户程序** | 作为装载存储器载体 |
| **保存系统数据** | 保存CPU配置和系统信息 |
| **保存配方数据** | 部分PLC可保存工艺参数 |
| **保存数据日志** | 记录采集数据或事件 |
| **固件升级** | 存放升级文件 |
| **备份恢复** | 支持工程恢复或CPU更换 |
| **扩展容量** | 弥补内部存储不足 |

## ③ 计算/选型示例

- 已知：  
  工程占用12MB，存储卡容量8MB。
- 公式：  
  **工程容量 > 存储卡容量**
- 计算：  
  12MB > 8MB，程序或数据无法完整保存。
- 结论：  
  应选择更大容量、型号兼容的存储卡。

> [!tip] 记忆口诀
> 存储卡不是随便插，大容量和型号兼容都要看。

# 04 诊断缓冲区

> [!info] 核心结论
> 诊断缓冲区用于保存PLC运行过程中发生的系统事件、故障、报警、模块异常、通讯异常和启动停止记录。

## ① 分类/公式/规则

> [!warning] 诊断公式
> **故障追溯 = 诊断缓冲区事件 + 时间戳 + 错误代码 + 相关模块**

## ② 参考表

| 诊断内容 | 示例 |
|---|---|
| **CPU启动/停止** | RUN、STOP、重启记录 |
| **模块故障** | I/O模块掉站、模块错误 |
| **通讯异常** | HMI断开、网络错误 |
| **程序错误** | 访问错误、块调用异常 |
| **存储器错误** | 装载或工作存储器不足 |
| **电源事件** | 掉电、上电 |
| **系统报警** | 诊断中断、硬件报警 |

## ③ 计算/选型示例

- 已知：  
  设备上午10:15突然停机。
- 公式：  
  **停机分析 = 10:15附近诊断事件**
- 计算：  
  查看诊断缓冲区中该时间段是否有CPU STOP、模块掉站或程序错误。
- 结论：  
  诊断缓冲区是PLC故障排查的重要入口。

> [!tip] 记忆口诀
> PLC出过什么事，先看诊断缓冲区。

# 05 通讯缓冲区

> [!info] 核心结论
> 通讯缓冲区用于临时保存PLC与HMI、SCADA、网关、伺服、变频器、MES等设备之间收发的数据。

## ① 分类/公式/规则

> [!warning] 通讯负载公式
> **通讯压力 = 连接数量 × 数据量 × 刷新频率**

## ② 参考表

| 通讯对象 | 缓冲作用 |
|---|---|
| **HMI** | 保存画面读写变量数据 |
| **SCADA** | 保存大量采集点数据 |
| **工业网关** | 保存协议转换数据 |
| **变频器** | 保存频率、状态、故障码 |
| **伺服驱动器** | 保存位置、速度、报警数据 |
| **MES系统** | 保存生产数据上传下载 |
| **远程I/O** | 保存输入输出刷新数据 |

## ③ 计算/选型示例

- 已知：  
  HMI每秒读取1000个变量，SCADA每秒读取2000个变量。
- 公式：  
  **总通讯读取量 = 1000 + 2000**
- 计算：  
  每秒约3000个变量请求，通讯压力较高。
- 结论：  
  应优化变量刷新周期和通讯分组，避免通讯缓冲压力过大。

> [!tip] 记忆口诀
> 通讯变量越多越快，缓冲和CPU压力越大。

# 06 数据日志存储区

> [!info] 核心结论
> 数据日志存储区用于保存PLC或控制系统采集到的过程数据、报警数据、事件记录或生产数据。

## ① 分类/公式/规则

> [!warning] 日志容量公式
> **日志容量 = 每条记录大小 × 记录条数 × 保存天数**

## ② 参考表

| 日志类型 | 示例 |
|---|---|
| **过程数据** | 温度、压力、流量、液位 |
| **报警记录** | 报警发生、恢复、确认 |
| **操作事件** | 启动、停止、复位 |
| **生产数据** | 产量、批次、良率 |
| **维护数据** | 运行小时、动作次数 |
| **通讯事件** | 连接断开、恢复 |
| **系统事件** | CPU启动、停止、错误 |

## ③ 计算/选型示例

- 已知：  
  每条记录100字节，每分钟记录60条，每天运行24小时。
- 公式：  
  **每日容量 = 100 × 60 × 60 × 24**
- 计算：  
  每日容量 = **8,640,000字节 ≈ 8.64MB**
- 结论：  
  数据日志需要规划保存周期和清理策略。

> [!tip] 记忆口诀
> 记录能帮追溯，但存储空间也要算。

# 07 参数存储区

> [!info] 核心结论
> 参数存储区用于保存PLC、模块、通讯设备或工艺功能需要的参数，例如IP地址、模块设置、工艺设定和设备配置。

## ① 分类/公式/规则

> [!warning] 参数规则
> **参数有效 = 参数值正确 + 下载成功 + 启动后加载正确**

## ② 参考表

| 参数类型 | 示例 |
|---|---|
| **CPU参数** | 启动方式、周期设置 |
| **通讯参数** | IP地址、端口、站号 |
| **模块参数** | 模拟量量程、滤波、诊断 |
| **工艺参数** | 速度、温度、压力、位置 |
| **运动参数** | 轴参数、限位、加速度 |
| **安全参数** | 安全输入输出配置 |
| **用户设置** | 语言、单位、显示模式 |

## ③ 计算/选型示例

- 已知：  
  模拟量输入模块设置为0~10V，但现场传感器是4~20mA。
- 公式：  
  **模块参数错误 → 采集数据错误**
- 计算：  
  输入信号类型不匹配，PLC读取值会异常。
- 结论：  
  参数存储区中的模块参数必须与现场硬件一致。

> [!tip] 记忆口诀
> 参数存得对，硬件才按预期工作。

# 08 备份存储区

> [!info] 核心结论
> 备份存储区用于保存工程、参数、配方、日志或系统恢复数据，便于故障后快速恢复设备。

## ① 分类/公式/规则

> [!warning] 备份公式
> **可靠恢复 = 程序备份 + 参数备份 + 配方备份 + 版本说明**

## ② 参考表

| 备份内容 | 作用 |
|---|---|
| **PLC程序** | 恢复控制逻辑 |
| **硬件组态** | 恢复模块和网络配置 |
| **工艺参数** | 恢复现场设定 |
| **配方数据** | 恢复产品参数 |
| **用户权限** | 恢复账号和角色 |
| **报警/日志** | 保留追溯数据 |
| **版本说明** | 确认当前工程版本 |

## ③ 计算/选型示例

- 已知：  
  CPU损坏需要更换，但没有参数和配方备份。
- 公式：  
  **无备份 → 恢复困难**
- 计算：  
  即使程序可重新下载，现场工艺参数可能需要重新调试。
- 结论：  
  备份存储区或外部备份必须包含程序、参数和配方。

> [!tip] 记忆口诀
> 备份不能只备程序，参数和配方一样重要。

# 09 闪存 Flash

> [!info] 核心结论
> Flash是一种非易失存储器，常用于保存固件、程序、参数或长期数据，断电后数据不丢失。

## ① 分类/公式/规则

> [!warning] Flash规则
> **Flash适合长期保存，不适合高频反复写入**

## ② 参考表

| 特点 | 说明 |
|---|---|
| **非易失** | 断电后数据保留 |
| **可擦写** | 可写入和更新 |
| **有寿命限制** | 写入次数有限 |
| **适合保存程序** | 用户程序、固件、配置 |
| **适合保存参数** | 工艺设定、系统配置 |
| **不适合高频日志** | 高频写入会影响寿命 |
| **常见载体** | CPU内部Flash、存储卡 |

## ③ 计算/选型示例

- 已知：  
  每秒向Flash写入一次数据日志。
- 公式：  
  **高频写入 → Flash寿命风险**
- 计算：  
  每天写入86400次，长期运行会增加磨损。
- 结论：  
  高频数据建议先缓存，再批量写入或使用更适合的数据记录方案。

> [!tip] 记忆口诀
> Flash能断电保存，但不要当高速记事本乱写。

# 10 EEPROM

> [!info] 核心结论
> EEPROM也是一种非易失存储器，常用于保存少量参数或校准数据，特点是断电保存、可擦写但容量较小、写入寿命有限。

## ① 分类/公式/规则

> [!warning] EEPROM规则
> **EEPROM适合少量关键参数，不适合大量频繁写入**

## ② 参考表

| 用途 | 示例 |
|---|---|
| **设备参数** | 地址、站号、模式 |
| **校准数据** | 传感器校准系数 |
| **少量设定** | 用户配置 |
| **设备识别** | 序列号、型号信息 |
| **维护信息** | 少量累计值 |
| **掉电保存** | 少量关键状态 |
| **不适合** | 高频趋势和大量日志 |

## ③ 计算/选型示例

- 已知：  
  每个产品周期都写一次EEPROM保存产量。
- 公式：  
  **频繁写EEPROM → 寿命风险**
- 计算：  
  如果每分钟写60次，长期运行写入次数很高。
- 结论：  
  产量累计应合理设计保存策略，不要过度频繁写EEPROM。

> [!tip] 记忆口诀
> EEPROM能记住，但不能太频繁写。

# 11 RAM临时存储

> [!info] 核心结论
> RAM用于PLC运行过程中的临时数据、缓存和计算，速度快，但断电后通常不保留。

## ① 分类/公式/规则

> [!warning] RAM规则
> **RAM适合运行时临时处理，断电长期保存要用保持或非易失存储**

## ② 参考表

| RAM用途 | 说明 |
|---|---|
| **运行数据** | 程序当前处理的数据 |
| **临时变量** | 计算中间值 |
| **通讯缓存** | 临时收发数据 |
| **任务缓冲** | 临时任务状态 |
| **高速处理** | 快速读写 |
| **断电风险** | 普通RAM断电丢失 |
| **保持需求** | 关键数据需转入保持区 |

## ③ 计算/选型示例

- 已知：  
  一个临时计算结果只用于本次扫描周期。
- 公式：  
  **临时计算 → RAM或局部变量**
- 计算：  
  不需要保存到保持存储区。
- 结论：  
  临时数据用RAM即可，关键累计值不能只放普通RAM。

> [!tip] 记忆口诀
> RAM跑得快，但断电通常记不住。

# 12 ROM只读存储

> [!info] 核心结论
> ROM用于保存不可随意修改的系统基础程序或出厂数据，普通工程师通常不能直接改写。

## ① 分类/公式/规则

> [!warning] ROM规则
> **ROM保存系统基础内容，用户程序一般不直接写ROM**

## ② 参考表

| 内容 | 说明 |
|---|---|
| **启动程序** | CPU底层启动功能 |
| **出厂数据** | 设备基本信息 |
| **基础固件** | 系统级功能 |
| **只读保护** | 防止误改 |
| **稳定性高** | 不易被用户程序破坏 |
| **不可随意修改** | 通常需厂商工具或固件升级 |
| **用户程序区别** | 用户程序通常在装载存储器中 |

## ③ 计算/选型示例

- 已知：  
  用户想通过PLC程序修改CPU底层启动程序。
- 公式：  
  **用户程序不能直接改ROM系统内容**
- 计算：  
  这类内容由厂家固件和系统机制管理。
- 结论：  
  ROM属于系统基础存储，不是普通用户逻辑存储区。

> [!tip] 记忆口诀
> ROM像出厂底座，不是用户随便写的区域。

# 13 时钟与系统时间存储

> [!info] 核心结论
> PLC通常具有系统时钟和时间相关存储，用于记录报警时间、数据时间戳、定时任务和生产追溯。

## ① 分类/公式/规则

> [!warning] 时间规则
> **有效追溯 = 准确系统时间 + 正确时间戳**

## ② 参考表

| 用途 | 示例 |
|---|---|
| **报警时间戳** | 报警发生、恢复时间 |
| **数据日志时间** | 过程数据采样时间 |
| **定时任务** | 每天、每班执行 |
| **生产批次** | 批次开始和结束时间 |
| **维护记录** | 保养时间、运行小时 |
| **NTP同步** | 与服务器校时 |
| **掉电保持时间** | RTC电池或超级电容支持 |

## ③ 计算/选型示例

- 已知：  
  PLC时间错误，报警记录时间比实际晚2小时。
- 公式：  
  **时间错误 → 追溯错误**
- 计算：  
  故障分析时无法准确对应现场事件。
- 结论：  
  系统时间和时钟存储必须定期校验或同步。

> [!tip] 记忆口诀
> 记录有没有价值，先看时间准不准。

# 14 高速计数与专用缓冲存储

> [!info] 核心结论
> 高速计数、运动控制、脉冲输出、通信模块等特殊功能可能使用专用缓冲区或模块内部存储区。

## ① 分类/公式/规则

> [!warning] 专用区规则
> **特殊模块数据 = 模块内部缓冲 + PLC读取/写入接口**

## ② 参考表

| 专用功能 | 存储内容 |
|---|---|
| **高速计数** | 当前计数值、锁存值 |
| **脉冲输出** | 目标频率、脉冲数 |
| **运动控制** | 轴状态、位置、速度 |
| **模拟量模块** | 原始值、诊断信息 |
| **通信模块** | 收发缓冲、状态码 |
| **称重模块** | 重量值、滤波参数 |
| **定位模块** | 坐标、回零状态 |

## ③ 计算/选型示例

- 已知：  
  高速计数器每秒接收5000个脉冲。
- 公式：  
  **高速信号 → 需要专用高速计数缓冲**
- 计算：  
  普通扫描周期可能来不及逐个脉冲处理。
- 结论：  
  高速信号应使用高速计数模块或专用存储/缓冲机制。

> [!tip] 记忆口诀
> 普通信号进I区，高速信号靠专用计数和缓冲。

# 15 其他存储器常见故障

> [!info] 核心结论
> 其他存储器相关故障常见于存储卡不兼容、诊断记录溢出、通讯缓冲拥堵、日志空间不足、参数丢失和Flash写入过频。

## ① 分类/公式/规则

> [!warning] 故障公式
> **其他存储器异常 = 介质异常 + 容量不足 + 写入过频 + 参数错误 + 缓冲拥堵**

## ② 参考表

| 现象 | 可能原因 |
|---|---|
| **存储卡识别失败** | 型号不匹配、损坏、格式错误 |
| **诊断记录看不到** | 缓冲区被覆盖或未保存 |
| **通讯频繁异常** | 缓冲压力大、刷新过快 |
| **日志写入失败** | 空间不足、路径错误 |
| **参数丢失** | 未备份、下载覆盖 |
| **Flash寿命风险** | 高频写入 |
| **CPU启动异常** | 固件或存储介质异常 |

## ③ 计算/选型示例

- 已知：  
  数据日志写入失败，同时存储卡剩余空间为0。
- 公式：  
  **剩余空间=0 → 无法继续写日志**
- 计算：  
  新记录无法保存，可能导致历史数据缺失。
- 结论：  
  数据日志必须设置容量监控、自动清理或备份策略。

> [!tip] 记忆口诀
> 存储空间满了，记录再重要也写不进去。

# 16 其他存储器排查步骤

> [!info] 核心结论
> 排查其他存储器问题时，应按存储介质、容量、文件完整性、诊断记录、通讯负载、参数版本和写入频率逐项检查。

## ① 分类/公式/规则

> [!warning] 排查公式
> **其他存储器排查 = 查介质 + 查容量 + 查诊断 + 查缓冲 + 查参数 + 查写入频率**

## ② 参考表

| 步骤 | 检查内容 |
|---|---|
| **1** | 检查存储卡或介质是否正常 |
| **2** | 检查剩余容量是否足够 |
| **3** | 查看CPU诊断缓冲区 |
| **4** | 检查通讯连接和刷新周期 |
| **5** | 检查日志保存路径和权限 |
| **6** | 检查参数是否被覆盖或丢失 |
| **7** | 检查Flash/EEPROM是否高频写入 |
| **8** | 必要时备份后更换存储介质 |

## ③ 计算/选型示例

- 已知：  
  PLC偶发通讯卡顿，HMI变量刷新很慢。
- 公式：  
  **通讯慢 = 通讯负载高 / 缓冲拥堵 / 网络异常**
- 计算：  
  检查HMI变量数量、刷新周期、通讯诊断和网络状态。
- 结论：  
  通讯缓冲问题要结合变量刷新策略一起排查。

> [!tip] 记忆口诀
> 存储问题先看容量，通讯问题先看负载。

# 17 其他存储器快速检查表

> [!info] 核心结论
> 其他存储器检查的目标是确保固件可靠、存储卡正常、诊断可查、通讯稳定、日志不丢、参数可恢复。

## ① 分类/公式/规则

> [!note] 十步检查法
> ① 检查存储卡型号、容量和状态  
> ② 检查固件版本是否满足工程要求  
> ③ 检查诊断缓冲区是否有关键报警  
> ④ 检查通讯变量数量和刷新周期  
> ⑤ 检查日志存储空间和清理策略  
> ⑥ 检查参数、配方和工程是否备份  
> ⑦ 检查Flash/EEPROM是否存在高频写入  
> ⑧ 检查系统时间是否准确  
> ⑨ 检查专用模块缓冲和状态数据  
> ⑩ 上线前做断电、重启、通讯中断、存储满测试

> [!warning] 最终检查公式
> **合格其他存储器 = 介质正常 + 容量足够 + 诊断可查 + 通讯稳定 + 数据可恢复**

## ② 参考表

| 检查项 | 目标 |
|---|---|
| **固件存储** | CPU系统稳定 |
| **存储卡** | 工程和数据可靠保存 |
| **诊断缓冲** | 故障可追溯 |
| **通讯缓冲** | 数据交换稳定 |
| **数据日志** | 历史记录不丢 |
| **参数存储** | 配置正确可恢复 |
| **Flash/EEPROM** | 避免过频写入 |
| **系统时间** | 记录时间准确 |

## ③ 计算/选型示例

- 已知：  
  项目需要长期保存报警、趋势、配方和生产记录，还要支持CPU快速恢复。
- 公式：  
  **可靠存储方案 = 存储卡 + 日志清理 + 参数备份 + 诊断记录**
- 计算：  
  不仅要看PLC程序容量，还要规划记录容量、备份方式和恢复流程。
- 结论：  
  其他存储器是系统可靠性和维护性的补充，不能只关注用户程序存储区。

> [!tip] 选型口诀
> 固件管系统，卡管保存；诊断查故障，缓冲管通讯；日志要清理，参数要备份。
