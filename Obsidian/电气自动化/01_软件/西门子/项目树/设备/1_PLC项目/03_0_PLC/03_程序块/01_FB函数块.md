# 01｜函数块 FB 是什么

> [!info] 核心结论
> **函数块 FB 是带“记忆能力”的程序块，适合封装电机、阀门、气缸、轴控、通信等需要保存状态的设备逻辑。**

<div class="card blue" style="background:#F7F9FC;border-left:6px solid #173B63;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>一句话理解</strong><br>
FB 就像一个“带记忆的功能模块”，不仅能执行逻辑，还能把自己的状态保存下来。
</div>

## ① 分类/公式/规则

| 项目 | 说明 |
|---|---|
| **英文名称** | Function Block |
| **中文名称** | 函数块 |
| **核心特点** | 有实例 DB，能保存状态 |
| **适合场景** | 电机控制、阀门控制、通信、状态机 |
| **调用方式** | 由 OB / FC / FB 调用 |

<div class="card orange" style="background:#FFF7F3;border-left:6px solid #FF6B35;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>核心规则</strong><br>
FB 必须配合实例 DB 使用，否则它无法保存自己的静态数据。
</div>

## ② 参考表

| 块类型 | 是否带记忆 | 典型用途 |
|---|---|---|
| **OB** | 否/系统调度 | 程序入口 |
| **FB** | **是** | 设备控制、状态机 |
| **FC** | 否 | 计算、转换、通用函数 |
| **DB** | 是 | 数据存储 |

## ③ 计算/选型示例

- **已知：** 要写一个电机启停控制块，需要保存运行状态、故障锁存、启动延时  
- **公式：** 需要保存状态 → 优先用 FB  
- **计算：**
  - 运行状态需要记住
  - 故障锁存需要记住
  - 定时器状态需要保存
  - 所以需要实例数据
- **结论：** 电机控制逻辑适合写成 FB。

> [!tip] 记忆口诀
> **需要记状态，用 FB；只算一次值，用 FC。**

# 02｜FB 与实例 DB

> [!info] 核心结论
> **FB 的“记忆”来自实例 DB，每调用一次 FB，就需要一个对应的数据存储空间。**

<div class="card green" style="background:#F1FBF5;border-left:6px solid #27AE60;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>一句话理解</strong><br>
FB 是逻辑模板，实例 DB 是它的专属记事本。
</div>

## ① 分类/公式/规则

| 名称 | 作用 |
|---|---|
| **FB** | 存放控制逻辑 |
| **实例 DB** | 存放该 FB 的静态变量和运行状态 |
| **单实例** | 一个 FB 调用对应一个实例 DB |
| **多实例** | 多个 FB 实例放在一个上级 FB 的实例 DB 中 |
| **Static 变量** | 会保存在实例 DB 中 |

<div class="card orange" style="background:#FFF7F3;border-left:6px solid #FF6B35;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>重点公式</strong><br>
FB 调用 = FB 逻辑 + 实例 DB 数据
</div>

## ② 参考表

| 调用方式 | 数据存放位置 | 适合场景 |
|---|---|---|
| **单实例 DB** | 独立实例 DB | 单个设备清晰管理 |
| **多实例 DB** | 上级 FB 的实例 DB | 多个设备集中封装 |
| **全局 DB 配合 FB** | 参数/状态分开管理 | 大型项目 |

## ③ 计算/选型示例

- **已知：** 有 3 台电机，使用同一个 `FB_Motor` 控制  
- **公式：** 每台设备需要独立状态 → 每台需要独立实例  
- **计算：**
  - 电机 1：`DB_Motor_1`
  - 电机 2：`DB_Motor_2`
  - 电机 3：`DB_Motor_3`
- **结论：** 同一个 FB 可以重复调用，但每个设备要有自己的实例数据。

> [!tip] 记忆口诀
> **FB 是模具，实例 DB 是每个产品自己的档案。**

# 03｜FB 接口区

> [!info] 核心结论
> **FB 接口区用于定义输入、输出、输入输出、静态变量和临时变量，是 FB 对外连接和内部存储的核心。**

<div class="card blue" style="background:#F7F9FC;border-left:6px solid #173B63;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>接口口诀</strong><br>
Input 接命令，Output 给结果，InOut 双向传，Static 记状态，Temp 临时算。
</div>

## ① 分类/公式/规则

| 接口类型 | 中文含义 | 是否保存 |
|---|---|---|
| **Input** | 输入参数 | 否 |
| **Output** | 输出参数 | 通常输出结果 |
| **InOut** | 输入输出参数 | 外部变量双向传递 |
| **Static** | 静态变量 | **保存到实例 DB** |
| **Temp** | 临时变量 | 不保存 |

## ② 参考表

| 接口 | 适合放什么 | 注意 |
|---|---|---|
| **Input** | 启动、停止、复位、设定值 | 外部给 FB |
| **Output** | 运行、报警、完成、状态码 | FB 输出给外部 |
| **InOut** | 需要读写的外部数据 | 传地址，需谨慎 |
| **Static** | 状态机步骤、锁存位、定时器 | 断电保持需另行设置 |
| **Temp** | 中间计算变量 | 扫描结束不保留 |

## ③ 计算/选型示例

- **已知：** 电机 FB 需要保存当前步骤号 `Step`  
- **公式：** 需要跨扫描周期保存 → 用 Static  
- **计算：**
  - `Step` 每个扫描周期都要记住
  - 放 Temp 会丢失
  - 放 Static 可保存在实例 DB
- **结论：** 状态机步骤号应放在 Static 区。

> [!tip] 记忆口诀
> **要记住放 Static，用完就丢放 Temp。**

# 04｜FB 与 FC 的区别

> [!info] 核心结论
> **FB 有实例数据，适合做设备对象；FC 没有固定实例数据，适合做通用计算。**

<div class="card orange" style="background:#FFF7F3;border-left:6px solid #FF6B35;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>选型口诀</strong><br>
设备控制用 FB，数学计算用 FC；要记忆用 FB，不记忆用 FC。
</div>

## ① 分类/公式/规则

| 对比项 | FB | FC |
|---|---|---|
| **是否有实例 DB** | **有** | 无 |
| **是否保存状态** | **可以** | 不适合 |
| **适合对象** | 电机、阀门、气缸、轴 | 计算、转换、判断 |
| **调用复杂度** | 稍高 | 简单 |
| **复用方式** | 多实例/多 DB | 直接调用 |

## ② 参考表

| 需求 | 推荐块 |
|---|---|
| **电机启停控制** | FB |
| **阀门开关控制** | FB |
| **模拟量工程量换算** | FC |
| **温度单位转换** | FC |
| **通信状态机** | FB |
| **报警锁存逻辑** | FB |

## ③ 计算/选型示例

- **已知：** 要把 `0-27648` 的模拟量换算成 `0-100.0℃`  
- **公式：** 纯计算，不需要保存状态 → FC  
- **计算：**
  - 输入原始值
  - 按比例计算输出工程量
  - 不需要记住上一次状态
- **结论：** 这种换算函数更适合写成 FC，而不是 FB。

> [!tip] 记忆口诀
> **FB 像设备，FC 像公式。**

# 05｜FB 的典型用途

> [!info] 核心结论
> **凡是有状态、有步骤、有报警、有定时、有保持需求的控制逻辑，都很适合封装成 FB。**

<div class="card green" style="background:#F1FBF5;border-left:6px solid #27AE60;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>工程理解</strong><br>
大型 PLC 项目中，FB 常用来做“标准设备模板”，让程序更整齐、更容易复用。
</div>

## ① 分类/公式/规则

| 对象 | 为什么适合 FB |
|---|---|
| **电机** | 运行、故障、联锁、延时 |
| **阀门** | 开到位、关到位、超时报警 |
| **气缸** | 伸出、缩回、到位检测 |
| **伺服轴** | 使能、定位、报警、状态 |
| **通信块** | 步骤、超时、重试、状态码 |
| **设备单元** | 多状态、多条件、多输出 |

## ② 参考表

| FB 模板 | 常见接口 |
|---|---|
| **FB_Motor** | Start、Stop、Fault、Running |
| **FB_Valve** | OpenCmd、CloseCmd、OpenFB、CloseFB |
| **FB_Cylinder** | Extend、Retract、Home、Work |
| **FB_Axis** | Enable、MoveAbs、Done、Error |
| **FB_Comm** | SendReq、Busy、Done、Status |

## ③ 计算/选型示例

- **已知：** 一条产线有 20 个气缸，动作逻辑类似  
- **公式：** 相同对象重复出现 → 写成 FB 模板  
- **计算：**
  - 写一个 `FB_Cylinder`
  - 每个气缸一个实例 DB
  - OB1 中按设备调用
- **结论：** FB 可以大幅减少重复代码，提高维护效率。

> [!tip] 记忆口诀
> **设备越多，越要模板；逻辑越重复，越要 FB。**

# 06｜FB 调用方式

> [!info] 核心结论
> **FB 必须被 OB、FC 或其他 FB 调用才会执行，最常见是在 OB1 中调用各个设备 FB。**

<div class="card blue" style="background:#F7F9FC;border-left:6px solid #173B63;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>调用链理解</strong><br>
CPU 自动执行 OB，OB 再调用 FB，FB 内部执行设备逻辑。
</div>

## ① 分类/公式/规则

| 调用位置 | 说明 |
|---|---|
| **OB1 调用 FB** | 最常见主循环调用 |
| **FC 调用 FB** | 结构化封装 |
| **FB 调用 FB** | 多实例常见 |
| **中断 OB 调用 FB** | 需谨慎，逻辑要短 |
| **启动 OB 调用 FB** | 初始化类任务 |

## ② 参考表

| 调用方式 | 优点 | 注意 |
|---|---|---|
| **单实例调用** | 清晰直观 | DB 数量多 |
| **多实例调用** | 结构紧凑 | 层级更深 |
| **OB1 直接调用** | 简单 | 项目大时 OB1 会臃肿 |
| **分区 FC 调用** | 结构清晰 | 需规划调用层级 |

## ③ 计算/选型示例

- **已知：** 有 5 台电机，主程序需要周期执行它们的逻辑  
- **公式：** 每个扫描周期调用一次设备 FB  
- **计算：**
  - `OB1` 调用 `FB_Motor_1`
  - `OB1` 调用 `FB_Motor_2`
  - ...
  - `OB1` 调用 `FB_Motor_5`
- **结论：** 如果某个 FB 没被调用，它的逻辑不会执行。

> [!tip] 记忆口诀
> **FB 写好了不调用，等于没写。**

# 07｜单实例与多实例

> [!info] 核心结论
> **单实例是每个 FB 调用单独生成一个实例 DB；多实例是把多个 FB 实例集中放进上级 FB 的实例 DB 中。**

<div class="card purple" style="background:#F8F1FF;border-left:6px solid #9B59B6;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>一句话理解</strong><br>
单实例像“每台设备一本档案”，多实例像“一个设备组共用一本大档案”。
</div>

## ① 分类/公式/规则

| 类型 | 数据位置 | 适合场景 |
|---|---|---|
| **单实例** | 独立实例 DB | 少量设备、调试直观 |
| **多实例** | 上级 FB 的实例 DB | 多设备封装、结构整齐 |
| **嵌套多实例** | 多层 FB 内部 | 复杂设备单元 |

<div class="card orange" style="background:#FFF7F3;border-left:6px solid #FF6B35;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>选型规则</strong><br>
新手调试优先单实例；大型设备单元优先考虑多实例。
</div>

## ② 参考表

| 对比项 | 单实例 | 多实例 |
|---|---|---|
| **直观性** | 高 | 中 |
| **DB 数量** | 多 | 少 |
| **封装性** | 一般 | 强 |
| **维护方式** | 单个设备清楚 | 设备组更整齐 |
| **适合项目** | 小项目/调试 | 中大型项目 |

## ③ 计算/选型示例

- **已知：** 一个工站内有 4 个气缸、2 个电机，想封装成一个工站 FB  
- **公式：** 设备组封装 → 多实例  
- **计算：**
  - 创建 `FB_Station`
  - 在 `FB_Station` 的 Static 中声明多个电机/气缸 FB 实例
  - 只给 `FB_Station` 一个实例 DB
- **结论：** 多实例适合把多个设备封装成一个完整功能单元。

> [!tip] 记忆口诀
> **单设备单实例，设备组多实例。**

# 08｜FB 变量区怎么用

> [!info] 核心结论
> **FB 变量区设计是否合理，直接决定程序是否清晰、可复用、可维护。**

<div class="card blue" style="background:#F7F9FC;border-left:6px solid #173B63;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>变量设计口诀</strong><br>
外部给的放 Input，外部看的放 Output，内部记忆放 Static，临时计算放 Temp。
</div>

## ① 分类/公式/规则

| 变量区 | 适合内容 |
|---|---|
| **Input** | 命令、设定值、联锁条件 |
| **Output** | 状态、报警、完成信号 |
| **InOut** | 双向数据、外部结构体 |
| **Static** | 步骤号、锁存、定时器、内部状态 |
| **Temp** | 中间计算、临时判断 |

## ② 参考表

| 变量名示例 | 建议变量区 | 原因 |
|---|---|---|
| **StartCmd** | Input | 外部启动命令 |
| **Running** | Output | 给外部显示 |
| **Step** | Static | 状态机需保存 |
| **TimerStartDelay** | Static | 定时器需保存状态 |
| **TempSpeedLimit** | Temp | 临时计算 |
| **ParameterStruct** | InOut | 双向读写结构 |

## ③ 计算/选型示例

- **已知：** FB 内部有一个启动延时定时器  
- **公式：** 定时器实例需要保存状态 → 放 Static  
- **计算：**
  - 定时器需要保存当前计时状态
  - 每个 FB 实例要有自己的定时器
  - 所以定时器应声明在 Static 中
- **结论：** FB 内部定时器通常放 Static 区。

> [!tip] 记忆口诀
> **状态放 Static，计算放 Temp；接口要清楚，调用才舒服。**

# 09｜FB 编程模板

> [!info] 核心结论
> **一个好的 FB 模板通常包含：命令输入、联锁判断、状态输出、报警处理、复位逻辑和状态保持。**

<div class="card green" style="background:#F1FBF5;border-left:6px solid #27AE60;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>模板思路</strong><br>
先判断能不能动，再执行动作；动作后给状态，异常时给报警。
</div>

## ① 分类/公式/规则

| 功能区 | 内容 |
|---|---|
| **命令输入** | Start、Stop、Reset |
| **条件判断** | Auto、Manual、Interlock |
| **动作输出** | RunCmd、OpenCmd、CloseCmd |
| **状态反馈** | Running、Done、Ready |
| **报警处理** | Fault、Timeout、ErrorCode |
| **复位逻辑** | Reset 清故障 |

## ② 参考表

| 设备 FB | 必备逻辑 |
|---|---|
| **电机 FB** | 启停、运行反馈、故障、复位 |
| **阀门 FB** | 开关命令、到位、超时 |
| **气缸 FB** | 伸缩命令、两端到位、超时 |
| **通信 FB** | 请求、忙、完成、错误、状态码 |
| **轴控 FB** | 使能、定位、完成、报警 |

## ③ 计算/选型示例

- **已知：** 设计一个阀门 FB，需要开阀超时报警  
- **公式：** 发出开阀命令后，在规定时间内未到位 → 报警  
- **计算：**
  - `OpenCmd = TRUE`
  - 启动开阀定时器
  - 如果 `OpenFB` 未在设定时间内变为 TRUE
  - 置位 `OpenTimeout`
- **结论：** FB 中适合封装超时、报警和复位逻辑。

> [!tip] 记忆口诀
> **命令进去，状态出来；异常锁存，复位清除。**

# 10｜FB 常见错误

> [!info] 核心结论
> **FB 常见问题集中在：没调用、实例 DB 错、变量区用错、Static 没理解、多设备共用同一实例。**

<div class="card red" style="background:#FFF4F3;border-left:6px solid #E74C3C;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>避坑提醒</strong><br>
同一个实例 DB 不能随便给多台独立设备共用，否则状态会互相串。
</div>

## ① 分类/公式/规则

| 问题 | 原因 |
|---|---|
| **FB 不执行** | 没被 OB/FC/FB 调用 |
| **状态混乱** | 多台设备共用一个实例 DB |
| **数据不保持** | 应放 Static 却放 Temp |
| **输出不更新** | 调用条件不成立 |
| **下载后异常** | 接口修改后未更新调用 |
| **实例 DB 报错** | FB 接口变化后未同步 |

## ② 参考表

| 现象 | 排查方向 | 解决建议 |
|---|---|---|
| **逻辑没反应** | 看 FB 是否被调用 | 在线监控调用网络 |
| **两台设备互相影响** | 查实例 DB | 每台设备独立实例 |
| **步骤号每周期归零** | 查变量区 | Step 放 Static |
| **报警无法复位** | 查锁存逻辑 | 增加 Reset 条件 |
| **接口红色报错** | FB 接口变更 | 更新块调用 |
| **DB 数据异常** | 多实例/单实例混用 | 检查调用结构 |

## ③ 计算/选型示例

- **已知：** 两台电机调用同一个 `FB_Motor`，但共用了同一个实例 DB  
- **公式：** 独立设备 = 独立实例数据  
- **计算：**
  - 电机 1 的状态写入实例 DB
  - 电机 2 也写入同一个实例 DB
  - 两台电机状态互相覆盖
- **结论：** 两台独立电机必须使用不同实例 DB，或用正确的多实例结构。

> [!tip] 记忆口诀
> **FB 可以共用，实例不能乱共用。**

# 11｜一页速记总卡

> [!info] 核心结论
> **FB 的核心价值是“封装可复用设备逻辑，并通过实例 DB 保存每个对象自己的状态”。**

<div class="card blue" style="background:#F7F9FC;border-left:6px solid #173B63;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>总口诀</strong><br>
FB 管逻辑，DB 管记忆；一个设备，一个实例；多个设备，多份状态。
</div>

## ① 分类/公式/规则

| 主轴 | 关键词 |
|---|---|
| **本质** | 带记忆的功能块 |
| **依赖** | 实例 DB |
| **适合** | 设备控制、状态机、通信 |
| **变量区** | Input / Output / InOut / Static / Temp |
| **调用** | 由 OB / FC / FB 调用 |
| **复用** | 单实例 / 多实例 |

## ② 参考表

| 需求 | 推荐做法 |
|---|---|
| **普通计算** | FC |
| **电机控制** | FB |
| **气缸控制** | FB |
| **单位换算** | FC |
| **通信状态机** | FB |
| **保存状态** | Static + 实例 DB |
| **多设备复用** | 多实例或多个实例 DB |

## ③ 计算/选型示例

- **已知：** 项目中有 10 个类似阀门，需要统一控制逻辑  
- **公式：** 相同设备逻辑 → 一个 FB 模板 + 多个实例  
- **计算：**
  - 创建 `FB_Valve`
  - 每个阀门一个实例
  - OB1 中分别调用
  - 每个实例保存自己的到位、超时和报警状态
- **结论：** FB 让程序模块化、标准化、可复用，特别适合中大型 PLC 项目。
