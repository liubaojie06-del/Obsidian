# 01｜程序信息是什么

> [!info] 核心结论
> **程序信息是 TIA Portal 中用于查看 PLC 程序结构、块调用关系、变量引用关系和资源使用情况的分析入口。**

<div class="card blue" style="background:#F7F9FC;border-left:6px solid #173B63;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>图片识别</strong><br>
截图中的 <strong>程序信息</strong> 是 PLC 项目树里的程序分析工具，不是 OB / FB / FC / DB 这种直接执行的程序块。
</div>

## ① 分类/公式/规则

| 项目 | 说明 |
|---|---|
| **中文名称** | 程序信息 |
| **英文理解** | Program Information |
| **核心作用** | 查看程序结构和引用关系 |
| **常见用途** | 查调用链、查变量引用、查块关系、查资源 |
| **是否参与 PLC 执行** | 不参与执行 |
| **适合场景** | 程序维护、排故、交接、重构、下载前检查 |

<div class="card orange" style="background:#FFF7F3;border-left:6px solid #FF6B35;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>一句话理解</strong><br>
程序信息就是 PLC 程序的“结构透视图”，帮你看清谁调用谁、谁用了谁。
</div>

## ② 参考表

| 功能方向 | 作用 |
|---|---|
| **调用结构** | 查看 OB、FB、FC 的调用层级 |
| **从属关系** | 查看块之间的依赖关系 |
| **交叉引用** | 查看变量、块、地址在哪里被使用 |
| **资源信息** | 查看程序块、存储区、资源占用 |
| **一致性检查** | 辅助发现未调用、未使用、引用异常 |

## ③ 计算/选型示例

- **已知：** 一个电机 FB 不执行，不知道是否被 OB1 调用  
- **公式：** 程序不执行 → 先查调用结构  
- **计算：**
  - 打开程序信息
  - 查看 `FB_Motor`
  - 检查是否被 `OB1` 或其它块调用
  - 如果没有调用，程序逻辑不会运行
- **结论：** 程序信息可以快速判断某个块是否真正进入 PLC 扫描流程。

> [!tip] 记忆口诀
> **程序写了不一定跑，被调用了才会跑。**

# 02｜调用结构

> [!info] 核心结论
> **调用结构用于查看 PLC 程序从 OB 入口开始，逐层调用哪些 FB、FC 和实例 DB。**

<div class="card green" style="background:#F1FBF5;border-left:6px solid #27AE60;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>工程理解</strong><br>
调用结构就像程序的“树状地图”，从 OB1 往下看，能看清主程序到底调了哪些功能块。
</div>

## ① 分类/公式/规则

| 层级 | 说明 |
|---|---|
| **OB** | 程序入口，如 `OB1`、启动 OB、中断 OB |
| **FB** | 带实例数据的功能块 |
| **FC** | 无实例数据的函数 |
| **实例 DB** | FB 调用时对应的数据块 |
| **多实例** | 上级 FB 内部调用下级 FB |

<div class="card orange" style="background:#FFF7F3;border-left:6px solid #FF6B35;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>重点公式</strong><br>
PLC 执行路径 = OB 入口 → FB / FC 调用链 → 具体逻辑
</div>

## ② 参考表

| 查看内容 | 用途 |
|---|---|
| **OB1 调用了哪些块** | 查看主循环结构 |
| **某个 FB 是否被调用** | 判断逻辑是否执行 |
| **调用层级是否过深** | 优化程序结构 |
| **实例 DB 是否对应正确** | 避免设备状态串台 |
| **某块是否孤立** | 找未使用块 |

## ③ 计算/选型示例

- **已知：** `FC_ScaleAnalog` 已经写好，但模拟量工程值一直不变  
- **公式：** FC 必须被调用才会执行  
- **计算：**
  - 在调用结构中查 `FC_ScaleAnalog`
  - 如果没有出现在 OB1 或其它调用链下
  - 说明该 FC 没有被执行
- **结论：** 需要在 OB1、FB 或其它 FC 中调用它。

> [!tip] 记忆口诀
> **先看入口 OB，再查调用链。**

# 03｜从属关系

> [!info] 核心结论
> **从属关系用于查看一个块依赖哪些对象，或被哪些对象使用，适合排查删除块、修改接口、重构程序时的影响范围。**

<div class="card blue" style="background:#F7F9FC;border-left:6px solid #173B63;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>一句话理解</strong><br>
从属关系就是看“这个块和谁有关”，防止你改一个地方，影响一大片。
</div>

## ① 分类/公式/规则

| 关系 | 含义 |
|---|---|
| **调用者** | 哪些块调用了它 |
| **被调用对象** | 它内部调用了哪些块 |
| **使用的数据块** | 它引用了哪些 DB |
| **使用的数据类型** | 它依赖哪些 UDT |
| **相关工艺对象** | 是否关联轴、PID、通信对象 |

## ② 参考表

| 操作 | 为什么要查从属关系 |
|---|---|
| **删除 FB** | 防止其它程序还在调用 |
| **修改 FC 接口** | 防止调用点参数报错 |
| **修改 UDT** | 防止相关 DB / FB 全部受影响 |
| **改 DB 结构** | 防止 HMI、通信、程序引用异常 |
| **重构程序** | 先确认影响范围 |

## ③ 计算/选型示例

- **已知：** 准备修改 `UDT_Motor` 的结构  
- **公式：** UDT 变化 → 所有使用该 UDT 的 DB / FB 都可能受影响  
- **计算：**
  - 查 `UDT_Motor` 的从属关系
  - 找到所有使用它的 DB 和 FB
  - 修改前先确认是否需要同步更新
- **结论：** 修改公共 UDT 前必须先查影响范围。

> [!tip] 记忆口诀
> **改公共结构前，先查谁在用。**

# 04｜交叉引用

> [!info] 核心结论
> **交叉引用用于查看某个变量、地址、块、DB 成员在哪里被读取、写入或调用。**

<div class="card purple" style="background:#F8F1FF;border-left:6px solid #9B59B6;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>工程理解</strong><br>
交叉引用是 PLC 排故神器：一个变量为什么变了，先查是谁写了它。
</div>

## ① 分类/公式/规则

| 查询对象 | 能查什么 |
|---|---|
| **PLC 变量** | 哪些网络读取/写入 |
| **DB 变量** | 哪些块访问它 |
| **I / Q / M 地址** | 地址在哪里使用 |
| **FB / FC** | 哪些地方调用它 |
| **UDT** | 哪些结构引用它 |
| **常量/枚举** | 哪些逻辑使用它 |

<div class="card orange" style="background:#FFF7F3;border-left:6px solid #FF6B35;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>重点规则</strong><br>
变量值异常时，不要只看变量本身，要查所有写入位置。
</div>

## ② 参考表

| 现象 | 交叉引用排查 |
|---|---|
| **变量被自动清零** | 查哪里写 FALSE / 0 |
| **输出莫名动作** | 查谁写了输出命令 |
| **报警无法复位** | 查报警锁存和复位条件 |
| **DB 数据被覆盖** | 查所有写入该 DB 的块 |
| **M 点冲突** | 查同一地址是否多处使用 |
| **块不知道谁调用** | 查块调用引用 |

## ③ 计算/选型示例

- **已知：** `Motor1_RunCmd` 在监控表中刚改成 TRUE，下一扫描又变 FALSE  
- **公式：** 变量被程序覆盖 → 查写入来源  
- **计算：**
  - 对 `Motor1_RunCmd` 做交叉引用
  - 查看所有写入点
  - 找到某个联锁逻辑在条件不满足时写 FALSE
- **结论：** 变量自动变化通常是因为程序中有其它地方在写它。

> [!tip] 记忆口诀
> **变量乱变，查谁在写；输出乱动，查谁在控。**

# 05｜资源信息

> [!info] 核心结论
> **资源信息用于查看程序块、数据块、存储区和 CPU 资源占用情况，适合下载前、扩展前和项目优化时检查。**

<div class="card green" style="background:#F1FBF5;border-left:6px solid #27AE60;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>工程理解</strong><br>
资源信息就像 PLC 的“体检表”，看程序占了多少空间、还有多少余量。
</div>

## ① 分类/公式/规则

| 资源项 | 说明 |
|---|---|
| **装载存储器** | 存放下载到 CPU 的程序和数据 |
| **工作存储器** | CPU 运行时使用的内存 |
| **块数量** | OB / FB / FC / DB 数量 |
| **数据大小** | DB、实例 DB、数组等占用 |
| **编译结果** | 是否存在错误或警告 |
| **CPU 余量** | 后续扩展能力 |

## ② 参考表

| 检查场景 | 看什么 |
|---|---|
| **下载前** | 编译是否通过、资源是否超限 |
| **项目变大** | 工作存储器余量 |
| **DB 很多** | 数据存储占用 |
| **新增功能前** | CPU 是否还有资源 |
| **程序优化** | 哪些块或 DB 占用大 |
| **换 CPU 前** | 新 CPU 资源是否足够 |

## ③ 计算/选型示例

- **已知：** 程序新增大量报警历史 DB 后，下载提示资源不足  
- **公式：** 大量历史数据 → DB 占用增加  
- **计算：**
  - 查资源信息
  - 找到新增 DB 占用较大
  - 减少记录数量或优化数据类型
- **结论：** 资源不足时，要先查哪些 DB 或块占用最大。

> [!tip] 记忆口诀
> **程序能不能装进去，先看资源够不够。**

# 06｜程序信息适合什么时候用

> [!info] 核心结论
> **程序信息适合在排故、交接、重构、下载前检查、变量追踪和项目审查时使用。**

<div class="card orange" style="background:#FFF7F3;border-left:6px solid #FF6B35;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>使用口诀</strong><br>
查结构看调用，查变量看引用，查容量看资源。
</div>

## ① 分类/公式/规则

| 场景 | 推荐查看 |
|---|---|
| **程序不执行** | 调用结构 |
| **变量异常变化** | 交叉引用 |
| **准备删除块** | 从属关系 |
| **准备修改 UDT** | 从属关系 |
| **下载前检查** | 编译和资源信息 |
| **项目交接** | 调用结构 + 资源信息 |

## ② 参考表

| 问题 | 程序信息怎么帮你 |
|---|---|
| **FB 没反应** | 查是否被调用 |
| **某变量被覆盖** | 查写入点 |
| **想删除旧块** | 查是否仍被引用 |
| **项目结构太乱** | 看调用层级 |
| **DB 改了后报错** | 查依赖对象 |
| **CPU 资源不足** | 查资源占用 |

## ③ 计算/选型示例

- **已知：** 项目中有很多旧块，不知道能不能删除  
- **公式：** 删除前 = 先查是否被引用  
- **计算：**
  - 选中旧 FB / FC
  - 查看从属关系或交叉引用
  - 如果没有任何调用，再考虑删除
- **结论：** 不要凭名字判断旧块是否无用，先查引用关系。

> [!tip] 记忆口诀
> **删块先查引用，别删完才报错。**

# 07｜常见错误与排查

> [!info] 核心结论
> **程序信息常见误区是只看块是否存在，不看是否被调用；只看变量名，不查变量写入来源。**

<div class="card red" style="background:#FFF4F3;border-left:6px solid #E74C3C;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>避坑提醒</strong><br>
PLC 程序里“存在一个块”和“这个块正在执行”不是一回事。
</div>

## ① 分类/公式/规则

| 问题 | 常见原因 |
|---|---|
| **块存在但不执行** | 没有被 OB / FB / FC 调用 |
| **变量值被覆盖** | 多处写入同一变量 |
| **删除块后编译报错** | 其它块还在调用 |
| **修改接口后报错** | 调用点未同步更新 |
| **HMI 变量异常** | DB 结构或变量路径变化 |
| **资源不足** | DB、数组、历史记录占用过大 |

## ② 参考表

| 现象 | 排查方向 | 解决建议 |
|---|---|---|
| **FB 不动作** | 调用结构 | 在 OB1 或上级块中调用 |
| **变量自动变回去** | 交叉引用 | 查所有写入点 |
| **编译失败** | 从属关系 | 更新调用接口 |
| **删除后报错** | 块引用 | 恢复或修改调用 |
| **资源超限** | 资源信息 | 优化 DB 和数据类型 |
| **程序交接困难** | 调用结构 | 整理主调用链 |

## ③ 计算/选型示例

- **已知：** 删除 `FC_OldScale` 后项目编译报错  
- **公式：** 编译报错 = 仍有地方引用该 FC  
- **计算：**
  - 查 `FC_OldScale` 的交叉引用
  - 找到某个 FB 仍在调用它
  - 改成新 FC 或删除调用
- **结论：** 删除块前一定要先查引用关系。

> [!tip] 记忆口诀
> **块能删不能删，引用关系说了算。**

# 08｜一页速记总卡

> [!info] 核心结论
> **程序信息的核心价值是让工程师快速看清 PLC 项目的结构、调用、引用和资源状态。**

<div class="card blue" style="background:#F7F9FC;border-left:6px solid #173B63;padding:12px 14px;border-radius:12px;margin:10px 0;">
<strong>总口诀</strong><br>
调用结构看执行，从属关系看影响，交叉引用看读写，资源信息看容量。
</div>

## ① 分类/公式/规则

| 主轴 | 关键词 |
|---|---|
| **对象** | 程序信息 |
| **用途** | 程序结构分析 |
| **调用结构** | 谁调用谁 |
| **从属关系** | 谁依赖谁 |
| **交叉引用** | 谁读写谁 |
| **资源信息** | 占用多少 |
| **核心价值** | 排故、维护、交接、优化 |

## ② 参考表

| 需求 | 推荐查看 |
|---|---|
| **查 FB 是否执行** | 调用结构 |
| **查变量在哪里被写** | 交叉引用 |
| **查块能不能删** | 从属关系 |
| **查接口修改影响** | 从属关系 |
| **查 CPU 内存余量** | 资源信息 |
| **做项目交接** | 调用结构 + 注释 + 资源 |

## ③ 计算/选型示例

- **已知：** 接手一个陌生 PLC 项目  
- **公式：** 项目理解 = `入口 OB + 调用结构 + 关键 DB + 交叉引用`
- **计算：**
  - 先看 OB1 主调用链
  - 再看主要 FB / FC
  - 查关键变量交叉引用
  - 最后看资源和编译状态
- **结论：** 程序信息是接手陌生项目时最快建立全局视角的工具。

> [!tip] 记忆口诀
> **看程序别从细节钻，先用程序信息看全局。**
