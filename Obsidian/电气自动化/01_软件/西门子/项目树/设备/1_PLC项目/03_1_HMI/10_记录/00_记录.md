# 01 记录总览

> [!info] 核心结论
> **记录** 用于保存HMI/SCADA运行过程中的报警、趋势、变量、操作、参数修改、配方切换、生产数据和系统事件，是设备追溯、故障分析、质量分析和报表统计的基础。

## ① 分类/公式/规则

> [!warning] 记录系统公式
> **记录 = 数据源 + 采样/触发条件 + 时间戳 + 存储位置 + 查询/导出方式**

## ② 参考表

| 记录类型 | 作用 |
|---|---|
| **报警记录** | 保存报警发生、确认、恢复过程 |
| **趋势记录** | 保存温度、压力、电流、速度等连续变量 |
| **操作记录** | 保存用户启动、停止、复位等操作 |
| **参数记录** | 保存参数修改前后值 |
| **配方记录** | 保存配方加载、下载、修改、删除 |
| **生产记录** | 保存产量、节拍、批次、良率 |
| **系统记录** | 保存通讯、数据库、Runtime异常 |

## ③ 计算/选型示例

- 已知：  
  设备需要追溯每天的报警、产量、参数修改和配方切换。
- 公式：  
  **追溯需求 → 必须配置记录**
- 计算：  
  没有记录时，出现质量问题后很难判断原因。
- 结论：  
  HMI记录不是附加功能，而是设备交付和维护的重要部分。

> [!tip] 记忆口诀
> 画面看现在，记录看过去；没有记录，就没有追溯。

# 02 记录的作用

> [!info] 核心结论
> 记录的核心作用是把设备运行过程留下证据，便于分析故障、追溯质量、统计产量、优化工艺和判断责任。

## ① 分类/公式/规则

> [!warning] 记录价值公式
> **可追溯 = 时间 + 数据 + 用户 + 事件 + 设备来源**

## ② 参考表

| 作用 | 说明 |
|---|---|
| **故障追溯** | 查报警何时发生、何时恢复 |
| **质量追溯** | 查生产时使用的参数和配方 |
| **操作追溯** | 查谁启动、停止、复位、修改参数 |
| **工艺分析** | 查温度、压力、电流等过程变化 |
| **停机分析** | 统计报警频率和停机时间 |
| **产量统计** | 按班次、批次、日期统计 |
| **维护分析** | 判断设备是否频繁异常 |

## ③ 计算/选型示例

- 已知：  
  某批产品出现不良，需要判断当时温度是否异常。
- 公式：  
  **质量分析 = 批次时间 + 温度记录**
- 计算：  
  根据批次时间查询趋势历史温度。
- 结论：  
  趋势记录可以帮助判断质量异常是否与工艺参数有关。

> [!tip] 记忆口诀
> 记录不是为了存数据，而是为了事后能查清楚。

# 03 报警记录

> [!info] 核心结论
> 报警记录用于保存报警发生、确认、恢复、复位的全过程，是设备故障分析最常用的数据。

## ① 分类/公式/规则

> [!warning] 报警记录公式
> **报警记录 = 报警文本 + 发生时间 + 确认时间 + 恢复时间 + 确认用户**

## ② 参考表

| 字段 | 说明 |
|---|---|
| **报警名称** | 具体故障内容 |
| **报警等级** | 提示、警告、停机、急停 |
| **发生时间** | 报警开始时间 |
| **确认时间** | 操作员确认时间 |
| **恢复时间** | 故障消除时间 |
| **确认用户** | 谁确认了报警 |
| **设备来源** | 哪个设备、工位或区域 |

## ③ 计算/选型示例

- 已知：  
  报警8:10发生，8:12确认，8:25恢复。
- 公式：  
  **报警持续时间 = 恢复时间 - 发生时间**
- 计算：  
  8:25 - 8:10 = **15分钟**
- 结论：  
  报警记录可用于统计故障持续时间。

> [!tip] 记忆口诀
> 当前报警看处理，报警记录看追溯。

# 04 趋势记录

> [!info] 核心结论
> 趋势记录用于保存连续变量随时间变化的数据，适合温度、压力、液位、流量、电流、速度、真空等过程值。

## ① 分类/公式/规则

> [!warning] 趋势记录公式
> **趋势记录 = 变量 + 采样周期 + 时间戳 + 历史存储**

## ② 参考表

| 趋势变量 | 用途 |
|---|---|
| **温度** | 升温、保温、超温分析 |
| **压力** | 气压、液压稳定性 |
| **液位** | 水箱、料仓、油箱高度 |
| **流量** | 工艺流量变化 |
| **电流** | 电机负载分析 |
| **速度** | 输送或伺服运行状态 |
| **真空值** | 吸附稳定性分析 |

## ③ 计算/选型示例

- 已知：  
  20个Real变量，每秒记录1次，每个变量4字节，保存24小时。
- 公式：  
  **数据量 = 20 × 4 × 86400**
- 计算：  
  数据量 = **6,912,000字节 ≈ 6.9MB**
- 结论：  
  单日数据量不大，但长期保存仍要规划存储空间。

> [!tip] 记忆口诀
> 报警告诉结果，趋势记录告诉过程。

# 05 操作记录

> [!info] 核心结论
> 操作记录用于保存用户在HMI上的关键操作，例如启动、停止、复位、手动动作、参数保存、配方下载等。

## ① 分类/公式/规则

> [!warning] 操作记录公式
> **操作记录 = 时间 + 用户 + 操作对象 + 操作动作 + 操作结果**

## ② 参考表

| 操作类型 | 示例 |
|---|---|
| **设备操作** | 启动、停止、暂停、复位 |
| **手动操作** | 气缸伸出、阀门打开、电机点动 |
| **报警操作** | 确认报警、复位报警 |
| **参数操作** | 保存参数、修改设定 |
| **配方操作** | 加载、下载、删除、保存配方 |
| **权限操作** | 登录、注销、切换用户 |
| **系统操作** | 导出数据、清空记录、备份 |

## ③ 计算/选型示例

- 已知：  
  设备突然停机，需要判断是谁按下停止按钮。
- 公式：  
  **操作追溯 = 停机时间 + 操作记录**
- 计算：  
  查询停机前后的操作记录，查看用户和动作。
- 结论：  
  操作记录可以帮助区分设备故障和人为操作。

> [!tip] 记忆口诀
> 谁在什么时候做了什么，操作记录要能回答。

# 06 参数修改记录

> [!info] 核心结论
> 参数修改记录用于保存关键参数的旧值、新值、修改人和修改时间，防止参数变化后无法追溯。

## ① 分类/公式/规则

> [!warning] 参数记录公式
> **参数修改记录 = 参数名称 + 旧值 + 新值 + 修改用户 + 修改时间**

## ② 参考表

| 字段 | 说明 |
|---|---|
| **参数名称** | 速度、温度、压力、位置等 |
| **旧值** | 修改前参数 |
| **新值** | 修改后参数 |
| **单位** | mm/s、℃、MPa、s等 |
| **修改用户** | 谁修改了参数 |
| **修改时间** | 什么时候修改 |
| **修改原因** | 可选，用于质量追溯 |

## ③ 计算/选型示例

- 已知：  
  输送速度从300mm/s改为500mm/s后，设备开始频繁报警。
- 公式：  
  **异常时间点 + 参数修改记录 = 关联分析**
- 计算：  
  如果报警发生在参数修改之后，速度变化可能是原因之一。
- 结论：  
  关键参数修改必须记录旧值和新值。

> [!tip] 记忆口诀
> 参数改了不可怕，怕的是不知道谁改了、改成多少。

# 07 配方记录

> [!info] 核心结论
> 配方记录用于保存配方加载、下载、保存、覆盖、删除和版本变更信息，保证产品切换可追溯。

## ① 分类/公式/规则

> [!warning] 配方记录公式
> **配方记录 = 配方名称 + 操作类型 + 操作用户 + 操作时间 + 结果**

## ② 参考表

| 配方操作 | 记录内容 |
|---|---|
| **加载配方** | 用户选择了哪条配方 |
| **下载配方** | 配方是否写入PLC |
| **上传配方** | 从PLC读取当前参数 |
| **保存配方** | 保存了哪些参数 |
| **覆盖配方** | 哪条配方被覆盖 |
| **删除配方** | 删除人、删除时间 |
| **版本变更** | 配方版本和修改说明 |

## ③ 计算/选型示例

- 已知：  
  生产时HMI显示产品B，但PLC仍使用产品A参数。
- 公式：  
  **配方异常 = 配方记录 + PLC当前配方号**
- 计算：  
  查询是否执行过下载，PLC是否确认当前配方。
- 结论：  
  配方记录有助于判断换型是否真正完成。

> [!tip] 记忆口诀
> 选中配方不算生效，下载并记录才可靠。

# 08 生产记录

> [!info] 核心结论
> 生产记录用于保存批次、工单、产量、良品、不良品、节拍、运行时间和设备状态，是生产统计和质量追溯的重要数据。

## ① 分类/公式/规则

> [!warning] 生产记录公式
> **生产记录 = 批次/工单 + 时间 + 产品 + 产量 + 质量 + 设备状态**

## ② 参考表

| 字段 | 说明 |
|---|---|
| **工单号** | 当前生产任务 |
| **批次号** | 产品追溯编号 |
| **产品型号** | 当前生产产品 |
| **开始时间** | 批次开始时间 |
| **结束时间** | 批次结束时间 |
| **良品数量** | OK数量 |
| **不良品数量** | NG数量 |
| **节拍** | 单件或循环时间 |

## ③ 计算/选型示例

- 已知：  
  一个班次生产1000件，其中不良品25件。
- 公式：  
  **不良率 = 不良品数量 ÷ 总产量 × 100%**
- 计算：  
  25 ÷ 1000 × 100% = **2.5%**
- 结论：  
  生产记录可用于计算良率、不良率和班次绩效。

> [!tip] 记忆口诀
> 产量看数量，质量看良率，效率看节拍。

# 09 系统记录

> [!info] 核心结论
> 系统记录用于保存HMI运行系统、通讯、数据库、存储、网络、用户登录等系统级事件。

## ① 分类/公式/规则

> [!warning] 系统记录公式
> **系统记录 = 系统事件 + 时间戳 + 状态 + 错误信息**

## ② 参考表

| 系统事件 | 示例 |
|---|---|
| **Runtime启动** | HMI运行系统启动 |
| **Runtime停止** | HMI运行系统关闭 |
| **PLC通讯中断** | 主PLC离线 |
| **数据库异常** | 写入失败、连接失败 |
| **磁盘空间不足** | 记录无法继续保存 |
| **用户登录失败** | 密码错误、账号禁用 |
| **数据导出失败** | 路径错误、权限不足 |

## ③ 计算/选型示例

- 已知：  
  某天历史数据缺失2小时。
- 公式：  
  **数据缺失 → 查询系统记录**
- 计算：  
  检查该时间段是否有Runtime停止、数据库断开或磁盘满。
- 结论：  
  系统记录可以帮助解释数据为什么没有保存。

> [!tip] 记忆口诀
> 数据没了先别猜，先查系统记录有没有异常。

# 10 记录触发方式

> [!info] 核心结论
> 记录可以按周期触发、事件触发、变量变化触发、报警触发或用户操作触发。

## ① 分类/公式/规则

> [!warning] 触发方式公式
> **记录触发 = 周期条件 / 事件条件 / 变化条件满足**

## ② 参考表

| 触发方式 | 适合场景 |
|---|---|
| **周期记录** | 温度、压力、液位等趋势数据 |
| **变化记录** | 参数变化、状态变化 |
| **事件记录** | 启动、停止、换型、复位 |
| **报警记录** | 报警发生、确认、恢复 |
| **批次记录** | 批次开始、结束 |
| **条件记录** | 只在自动运行时记录 |
| **手动记录** | 用户点击保存或导出 |

## ③ 计算/选型示例

- 已知：  
  温度变化慢，每秒记录一次没有必要。
- 公式：  
  **慢变量 → 较长周期记录**
- 计算：  
  可设置10秒或30秒记录一次。
- 结论：  
  记录周期应根据变量变化速度和分析需求设置。

> [!tip] 记忆口诀
> 快变量快记录，慢变量慢记录，事件数据按事件记。

# 11 采样周期

> [!info] 核心结论
> 采样周期决定数据保存频率，周期越短数据越细，但存储和系统负载越高。

## ① 分类/公式/规则

> [!warning] 采样数据量公式
> **每日记录次数 = 86400 ÷ 采样周期秒数**

## ② 参考表

| 采样周期 | 适合数据 |
|---|---|
| **100ms~500ms** | 高速过程、特殊调试 |
| **1s** | 压力、电流、速度等较快变量 |
| **5s~10s** | 温度、液位、普通工艺变量 |
| **30s~1min** | 能耗、慢变化工艺数据 |
| **班次记录** | 产量、良率统计 |
| **事件记录** | 报警、操作、参数修改 |
| **按需记录** | 报表或人工导出 |

## ③ 计算/选型示例

- 已知：  
  采样周期为10秒。
- 公式：  
  **每日记录次数 = 86400 ÷ 10**
- 计算：  
  每天记录 **8640次**
- 结论：  
  采样周期越短，每天记录条数越多。

> [!tip] 记忆口诀
> 记录越频繁，数据越细，负载越高。

# 12 存储位置

> [!info] 核心结论
> 记录可以保存在HMI本地、工控机磁盘、数据库、服务器或云平台，不同存储方式适合不同项目规模。

## ① 分类/公式/规则

> [!warning] 存储选择规则
> **小型单机 → 本地文件**  
> **中大型系统 → 数据库/服务器**

## ② 参考表

| 存储方式 | 特点 |
|---|---|
| **HMI本地存储** | 简单，适合小型设备 |
| **CSV文件** | 易导出，适合报表和维护 |
| **SQLite** | 单机数据库，结构化存储 |
| **SQL Server/MySQL** | 多设备集中存储 |
| **MES数据库** | 与生产系统联动 |
| **云平台** | 远程查看和集中分析 |
| **边缘网关缓存** | 断网缓存和补传 |

## ③ 计算/选型示例

- 已知：  
  一条产线有10台设备，需要统一查询报警和产量。
- 公式：  
  **多设备统一查询 → 集中数据库**
- 计算：  
  每台设备本地存储会导致数据分散。
- 结论：  
  多设备项目建议使用数据库或服务器集中记录。

> [!tip] 记忆口诀
> 一台设备本地存，多台设备集中存。

# 13 文件记录

> [!info] 核心结论
> 文件记录常见为CSV、TXT或专用归档文件，适合小型项目、数据导出和维护分析。

## ① 分类/公式/规则

> [!warning] 文件记录公式
> **文件记录 = 文件路径 + 文件名规则 + 字段格式 + 保存周期**

## ② 参考表

| 文件设置 | 说明 |
|---|---|
| **文件路径** | 记录保存位置 |
| **文件名规则** | 按日期、批次、设备命名 |
| **字段分隔符** | 逗号、分号、Tab |

## ③ 计算/选型示例

- 已知：  
  每天生成一个报警CSV文件。
- 公式：  
  **文件名 = 类型 + 日期**
- 计算：  
  可命名为Alarm_2026-05-05.csv。
- 结论：  
  文件命名清晰，后期查询和备份更方便。

> [!tip] 记忆口诀
> 文件记录要按日期和类型命名，别都叫data.csv。

# 14 数据库记录

> [!info] 核心结论
> 数据库记录适合数据量较大、需要查询筛选、多设备集中管理和长期追溯的项目。

## ① 分类/公式/规则

> [!warning] 数据库记录公式
> **数据库记录 = 表结构 + 字段类型 + 写入周期 + 索引 + 备份清理**

## ② 参考表

| 数据库项 | 说明 |
|---|---|
| **表名** | AlarmLog、TrendLog、OperationLog |
| **字段** | 时间、变量、值、用户、事件 |
| **数据类型** | Int、Real、String、DateTime |
| **索引** | 加快按时间、设备查询 |
| **写入周期** | 周期或事件触发 |
| **备份策略** | 定期备份数据库 |
| **清理策略** | 删除或归档旧数据 |

## ③ 计算/选型示例

- 已知：  
  每天产生200MB记录，需要保存180天。
- 公式：  
  **存储容量 = 每日数据量 × 保存天数**
- 计算：  
  200MB × 180 = **36GB**
- 结论：  
  数据库磁盘至少要大于36GB，并预留索引和备份空间。

> [!tip] 记忆口诀
> 数据库不怕写，怕没人管容量、索引和备份。

# 15 时间戳

> [!info] 核心结论
> 时间戳是记录数据的核心字段，用于确定事件发生顺序、计算持续时间和进行质量追溯。

## ① 分类/公式/规则

> [!warning] 时间戳公式
> **有效记录 = 数据值 + 准确时间戳 + 数据来源**

## ② 参考表

| 时间类型 | 说明 |
|---|---|
| **采集时间** | 数据实际采集时刻 |
| **写入时间** | 数据写入文件或数据库时刻 |
| **报警发生时间** | 报警触发时刻 |
| **报警恢复时间** | 报警消除时刻 |
| **操作时间** | 用户执行操作时刻 |
| **批次开始时间** | 生产批次开始 |
| **批次结束时间** | 生产批次结束 |

## ③ 计算/选型示例

- 已知：  
  数据补传后全部显示为联网恢复时间。
- 公式：  
  **补传数据应保留采集时间**
- 计算：  
  如果只保存上传时间，无法判断真实事件顺序。
- 结论：  
  断网缓存和补传时必须保存原始采集时间戳。

> [!tip] 记忆口诀
> 没有准确时间，记录就没有追溯价值。

# 16 保存周期与清理

> [!info] 核心结论
> 保存周期决定记录保留多久，清理策略用于防止磁盘或数据库空间被历史数据占满。

## ① 分类/公式/规则

> [!warning] 保存容量公式
> **所需容量 = 每日数据量 × 保存天数 × 安全系数**

## ② 参考表

| 记录类型 | 常见保存周期 |
|---|---|
| **报警记录** | 3个月~1年 |
| **操作记录** | 3个月~1年 |
| **趋势记录** | 1个月~1年 |
| **生产记录** | 1年或更长 |
| **质量记录** | 按客户或法规要求 |
| **系统记录** | 1个月~6个月 |
| **调试记录** | 项目验收后可清理 |

## ③ 计算/选型示例

- 已知：  
  每天记录量为100MB，保存90天，安全系数1.5。
- 公式：  
  **容量 = 100 × 90 × 1.5**
- 计算：  
  容量 = **13,500MB ≈ 13.5GB**
- 结论：  
  记录存储至少预留13.5GB以上空间。

> [!tip] 记忆口诀
> 记录要能查，也要会清理。

# 18 记录权限

> [!info] 核心结论
> 记录查看、导出、删除和清空应设置权限，防止普通用户误删追溯数据。

## ① 分类/公式/规则

> [!warning] 权限公式
> **记录操作权限 = 用户角色 + 操作类型 + 数据重要性**

## ② 参考表

| 操作 | 推荐权限 |
|---|---|
| **查看记录** | 操作员及以上 |
| **筛选记录** | 操作员及以上 |
| **导出记录** | 班组长或工程师 |
| **删除记录** | 管理员 |
| **清空记录** | 管理员 + 二次确认 |
| **修改记录路径** | 工程师或管理员 |
| **修改保存周期** | 管理员 |

## ③ 计算/选型示例

- 已知：  
  普通操作员可以清空报警记录。
- 公式：  
  **清空记录 = 高风险操作**
- 计算：  
  清空后故障和质量追溯证据会丢失。
- 结论：  
  清空记录必须限制管理员权限，并加二次确认。

> [!tip] 记忆口诀
> 能看记录不代表能删记录。

# 19 记录备份

> [!info] 核心结论
> 记录备份用于防止硬盘损坏、HMI更换、数据库异常或误删导致历史数据丢失。

## ① 分类/公式/规则

> [!warning] 备份公式
> **可靠记录 = 本地保存 + 定期备份 + 可恢复验证**

## ② 参考表

| 备份方式 | 说明 |
|---|---|
| **手动备份** | 维护人员定期导出 |
| **自动备份** | 系统按时间自动备份 |
| **本地备份** | 保存到本机其他目录 |
| **外部U盘备份** | 现场离线导出 |
| **服务器备份** | 上传到服务器 |
| **数据库备份** | 定期备份数据库 |
| **恢复测试** | 验证备份文件可用 |

## ③ 计算/选型示例

- 已知：  
  HMI硬盘损坏，历史记录没有备份。
- 公式：  
  **无备份 = 数据不可恢复**
- 计算：  
  报警、操作、生产数据可能全部丢失。
- 结论：  
  关键设备必须设置记录备份策略。

> [!tip] 记忆口诀
> 没备份的记录，不算真正安全。

# 20 常见故障排查

> [!info] 核心结论
> 记录异常排查按 **数据源、触发条件、采样周期、存储路径、数据库连接、权限、磁盘空间、时间同步** 的顺序进行。

## ① 分类/公式/规则

> [!warning] 排错公式
> **记录异常 = 数据源无效 + 触发未满足 + 路径错误 + 存储失败 + 权限不足 + 磁盘满**

## ② 参考表

| 现象 | 可能原因 |
|---|---|
| **没有记录生成** | 记录功能未启用、触发条件不满足 |
| **记录数值不变** | 变量未刷新、连接异常 |
| **记录时间错误** | 系统时间不准、未同步NTP |
| **历史记录为空** | 存储路径错、数据库断开 |
| **记录写入失败** | 权限不足、磁盘满、文件被占用 |
| **导出失败** | 路径无权限、文件名非法 |
| **记录丢失** | 清理策略错误、未备份 |
| **查询很慢** | 数据量过大、数据库无索引 |

## ③ 计算/选型示例

- 已知：  
  趋势画面有实时曲线，但历史查询为空。
- 公式：  
  **实时显示正常 + 历史为空 = 记录/存储问题**
- 计算：  
  变量能读到，但历史归档可能未启用或数据库写入失败。
- 结论：  
  应检查记录配置、存储路径和数据库状态。

> [!tip] 记忆口诀
> 实时有数据不代表历史有记录。

# 21 记录快速检查表

> [!info] 核心结论
> 记录检查的目标是确保数据能正确采集、准确保存、方便查询、安全导出、长期可追溯。

## ① 分类/公式/规则

> [!note] 十步检查法
> ① 确认需要记录哪些数据  
> ② 检查变量来源和数据类型  
> ③ 检查记录触发方式和采样周期  
> ④ 检查时间戳是否准确  
> ⑤ 检查存储路径或数据库连接  
> ⑥ 检查保存周期和自动清理策略  
> ⑦ 检查报警、趋势、操作、参数、配方记录是否完整  
> ⑧ 检查查询、筛选和导出功能  
> ⑨ 检查权限、删除和清空限制  
> ⑩ 上线前测试断电、断网、重启和磁盘空间不足场景

> [!warning] 最终检查公式
> **合格记录系统 = 数据准确 + 时间准确 + 存储可靠 + 查询方便 + 权限安全 + 备份完整**

## ② 参考表

| 检查项 | 目标 |
|---|---|
| **数据源** | 记录值正确 |
| **采样周期** | 不漏关键变化 |
| **时间戳** | 可追溯 |
| **存储** | 不丢失 |
| **查询** | 能快速定位 |
| **导出** | 便于分析 |
| **权限** | 防止误删 |
| **备份** | 故障后可恢复 |

## ③ 计算/选型示例

- 已知：  
  一台设备需要记录报警、操作、温度趋势、产量和配方切换，保存180天。
- 公式：  
  **完整追溯 = 报警记录 + 操作记录 + 趋势记录 + 生产记录 + 配方记录**
- 计算：  
  需要同时规划数据源、采样周期、存储容量、查询导出和备份。
- 结论：  
  推荐使用结构化记录方案，并在交付前验证查询、导出、备份和恢复。

> [!tip] 选型口诀
> 报警记录查故障，趋势记录看过程；操作记录查人员，参数记录查变化；生产记录看产量，配方记录看换型。
