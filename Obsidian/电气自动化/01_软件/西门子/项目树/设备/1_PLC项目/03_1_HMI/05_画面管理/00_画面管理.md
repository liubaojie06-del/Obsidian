# 01 画面管理总览

> [!info] 核心结论
> 画面管理用于组织、维护和配置HMI、SCADA、上位机中的各类页面，包括 **画面创建、分组、导航、权限、模板、弹窗、参数、报警、趋势、脚本和版本管理**。

## ① 分类/公式/规则

> [!warning] 管理公式
> **画面管理 = 页面结构 + 命名规则 + 导航关系 + 权限控制 + 模板复用 + 版本维护**

## ② 参考表

| 管理项 | 作用 |
|---|---|
| **画面分组** | 首页、自动、手动、参数、报警、维护等分类 |
| **画面命名** | 便于查找、维护、版本管理 |
| **画面导航** | 页面之间跳转关系 |
| **权限管理** | 限制不同用户访问不同页面 |
| **模板管理** | 统一顶部栏、底部栏、按钮样式 |
| **弹窗管理** | 管理确认、输入、报警、登录弹窗 |
| **变量绑定** | 页面控件与PLC/HMI变量关联 |
| **版本维护** | 防止工程混乱和误覆盖 |

## ③ 计算/选型示例

- 已知：  
  一套HMI工程有首页、自动、手动、参数、报警、趋势、配方、维护等页面。
- 公式：  
  **页面多 → 必须做画面管理**
- 计算：  
  若没有命名和分组规则，后期修改很容易找错页面。
- 结论：  
  画面管理应在工程初期建立规范，而不是后期页面混乱后再整理。

> [!tip] 记忆口诀
> 画面少靠记忆，画面多靠管理；没有管理，后期维护必乱。

# 02 底层原理

> [!info] 核心结论
> 画面管理的底层原理，是用工程树、页面属性、控件绑定、权限规则和导航逻辑，把多个画面组织成一个可运行的人机界面系统。

## ① 分类/公式/规则

> [!warning] 画面运行链路
> **画面文件 → 控件对象 → 变量绑定 → 权限判断 → 页面跳转 → 运行显示**

## ② 参考表

| 层级 | 说明 |
|---|---|
| **工程树** | 按文件夹或节点管理页面 |
| **页面对象** | 每个画面是一个独立页面对象 |
| **控件对象** | 按钮、指示灯、输入框、曲线等 |
| **变量对象** | 与PLC、数据库或内部变量关联 |
| **脚本逻辑** | 页面打开、关闭、按钮点击时执行 |
| **权限规则** | 决定用户能否访问或操作 |
| **运行系统** | 按配置加载和显示画面 |

## ③ 计算/选型示例

- 已知：  
  操作员点击“参数设置”，系统要求登录工程师账号。
- 公式：  
  **页面访问 = 导航请求 + 权限判断**
- 计算：  
  如果当前用户权限不足，系统应拒绝进入或弹出登录窗口。
- 结论：  
  画面管理不仅是页面整理，也包含访问规则和运行逻辑。

> [!tip] 记忆口诀
> 画面管理不是文件夹管理，而是页面、权限、跳转和变量的整体管理。

# 03 画面分组

> [!info] 核心结论
> 画面分组用于把不同功能页面分类管理，常见分组有生产、手动、参数、报警、趋势、配方、维护和系统。

## ① 分类/公式/规则

> [!warning] 分组规则
> **高频操作页面放前面，维护诊断页面放后面，危险页面加权限**

## ② 参考表

| 分组 | 常见页面 |
|---|---|
| **首页组** | 总览、状态、产量 |
| **运行组** | 自动运行、流程监控、工位状态 |
| **手动组** | 气缸、电机、伺服、阀门手动 |
| **参数组** | 时间、速度、位置、工艺参数 |
| **报警组** | 当前报警、历史报警、报警帮助 |
| **数据组** | 趋势、报表、记录、统计 |
| **配方组** | 配方选择、配方编辑、下载确认 |
| **维护组** | I/O监控、通讯诊断、版本信息 |

## ③ 计算/选型示例

- 已知：  
  工程中有60个页面。
- 公式：  
  **页面数量多 → 按功能分组**
- 计算：  
  若每组约5~10个页面，60个页面可分为6~10组。
- 结论：  
  页面分组能显著提升查找和维护效率。

> [!tip] 记忆口诀
> 生产页面给操作员，维护页面给工程师，系统页面给管理员。

# 04 画面命名规则

> [!info] 核心结论
> 统一命名规则可以减少维护错误，便于搜索、复制、调试和版本管理。

## ① 分类/公式/规则

> [!warning] 命名公式
> **画面名称 = 功能类别 + 序号 + 页面用途**

## ② 参考表

| 命名方式 | 示例 |
|---|---|
| **首页** | Main_Overview |
| **自动页面** | Auto_Process_01 |
| **手动页面** | Manual_Cylinder |
| **参数页面** | Param_Servo |
| **报警页面** | Alarm_Current |
| **趋势页面** | Trend_Pressure |
| **配方页面** | Recipe_Edit |
| **维护页面** | Maint_IO_Monitor |

## ③ 计算/选型示例

- 已知：  
  有多个手动画面，分别控制气缸、电机和伺服。
- 公式：  
  **同类页面 = 统一前缀 + 功能后缀**
- 计算：  
  可命名为Manual_Cylinder、Manual_Motor、Manual_Servo。
- 结论：  
  不建议使用“画面1、画面2、最终版、最终版2”这类名称。

> [!tip] 记忆口诀
> 名字起得清楚，后期少走弯路。

# 05 画面导航管理

> [!info] 核心结论
> 导航管理用于配置页面之间的跳转关系，保证操作人员能快速进入目标页面并返回首页。

## ① 分类/公式/规则

> [!warning] 导航设计公式
> **导航清晰 = 首页入口 + 一级菜单 + 二级菜单 + 返回按钮 + 当前页面提示**

## ② 参考表

| 导航元素 | 作用 |
|---|---|
| **首页按钮** | 快速返回总览 |
| **顶部菜单** | 显示主功能分类 |
| **侧边菜单** | 显示当前分组页面 |
| **底部导航** | 常用页面快速切换 |
| **返回按钮** | 回到上一级 |
| **面包屑路径** | 显示当前位置 |
| **报警入口** | 任意页面快速进入报警 |

## ③ 计算/选型示例

- 已知：  
  操作员需要从任意页面快速进入报警页面。
- 公式：  
  **高频页面入口 = 全局固定显示**
- 计算：  
  报警入口不应只放在首页，建议放在顶部或底部公共区域。
- 结论：  
  画面管理中应设置全局导航栏。

> [!tip] 记忆口诀
> 页面跳转要少点几次，报警入口要随处可达。

# 06 画面权限管理

> [!info] 核心结论
> 画面权限管理用于限制不同角色能访问和操作的页面，防止误操作和参数误改。

## ① 分类/公式/规则

> [!warning] 权限公式
> **页面可访问 = 当前用户角色 ≥ 页面要求权限**

## ② 参考表

| 角色 | 可访问页面 |
|---|---|
| **操作员** | 首页、自动、报警查看 |
| **班组长** | 首页、自动、报警、部分参数 |
| **工程师** | 手动、参数、配方、维护诊断 |
| **管理员** | 用户管理、系统设置、备份恢复 |
| **维护人员** | I/O监控、通讯诊断、手动调试 |
| **只读用户** | 数据查看、趋势、报表 |
| **审计用户** | 日志、报警历史、参数修改记录 |

## ③ 计算/选型示例

- 已知：  
  普通操作员不应进入伺服调试画面。
- 公式：  
  **危险调试页面 → 工程师权限**
- 计算：  
  进入页面前应验证用户权限。
- 结论：  
  手动调试、伺服参数、系统设置等页面必须加权限。

> [!tip] 记忆口诀
> 不是所有页面都该让所有人进。

# 07 画面模板管理

> [!info] 核心结论
> 画面模板用于统一页面风格、导航栏、状态栏、按钮样式和报警提示，减少重复设计。

## ① 分类/公式/规则

> [!warning] 模板复用公式
> **统一模板 = 顶部状态栏 + 底部导航栏 + 公共按钮 + 统一颜色 + 统一字体**

## ② 参考表

| 模板元素 | 作用 |
|---|---|
| **顶部状态栏** | 显示设备状态、当前用户、时间 |
| **底部导航栏** | 页面切换 |
| **公共报警栏** | 显示当前报警 |
| **统一按钮样式** | 降低误操作 |
| **统一颜色规范** | 运行、报警、停止、等待统一 |
| **统一字体大小** | 现场阅读一致 |
| **统一弹窗样式** | 提升操作体验 |

## ③ 计算/选型示例

- 已知：  
  20个页面都需要显示设备运行状态。
- 公式：  
  **公共元素 → 模板化**
- 计算：  
  如果每个页面单独复制，后期修改状态栏要改20次。
- 结论：  
  应使用画面模板或公共画面对象统一管理。

> [!tip] 记忆口诀
> 能做成模板的，就不要每个页面重复画。

# 08 公共对象管理

> [!info] 核心结论
> 公共对象是多个页面重复使用的控件或组件，例如启动停止按钮、报警条、用户信息、设备状态灯和导航菜单。

## ① 分类/公式/规则

> [!warning] 公共对象规则
> **多页面重复出现的控件，应尽量做成公共对象或全局组件**

## ② 参考表

| 公共对象 | 应用 |
|---|---|
| **启动/停止按钮组** | 多页面通用操作 |
| **报警状态条** | 任意页面提示故障 |
| **当前用户显示** | 权限状态提示 |
| **系统时间显示** | 日志和操作参考 |
| **设备状态灯** | 运行、停止、报警、急停 |
| **导航菜单** | 页面跳转 |
| **语言切换按钮** | 多语言工程 |

## ③ 计算/选型示例

- 已知：  
  所有页面底部都有相同导航按钮。
- 公式：  
  **重复导航 → 公共对象管理**
- 计算：  
  修改一个公共导航对象即可影响所有页面。
- 结论：  
  公共对象可以显著减少维护工作量。

> [!tip] 记忆口诀
> 重复出现三次以上，就要考虑做成公共对象。

# 09 弹窗画面管理

> [!info] 核心结论
> 弹窗画面用于确认、输入、报警、登录、帮助和提示，应集中管理，避免弹窗样式混乱和逻辑重复。

## ① 分类/公式/规则

> [!warning] 弹窗管理公式
> **弹窗 = 触发条件 + 显示内容 + 权限判断 + 确认动作 + 关闭逻辑**

## ② 参考表

| 弹窗类型 | 用途 |
|---|---|
| **确认弹窗** | 清零、删除、配方覆盖 |
| **输入弹窗** | 数值输入、密码输入 |
| **报警弹窗** | 重大故障强提示 |
| **登录弹窗** | 权限验证 |
| **帮助弹窗** | 故障处理说明 |
| **提示弹窗** | 保存成功、操作完成 |
| **维护弹窗** | 版本、通讯、诊断信息 |

## ③ 计算/选型示例

- 已知：  
  清空产量按钮被误点后直接清零。
- 公式：  
  **不可逆操作 → 确认弹窗 + 权限**
- 计算：  
  清零前应弹出“是否确认清空产量？”并要求权限。
- 结论：  
  弹窗管理能降低误操作风险。

> [!tip] 记忆口诀
> 弹窗不是越多越好，关键操作才需要弹窗。

# 10 画面变量管理

> [!info] 核心结论
> 画面变量管理用于维护页面中使用的PLC变量、内部变量、脚本变量和数据库变量，避免地址错乱。

## ① 分类/公式/规则

> [!warning] 变量管理公式
> **变量可靠 = 命名清楚 + 地址正确 + 类型正确 + 读写权限明确**

## ② 参考表

| 变量类型 | 说明 |
|---|---|
| **PLC变量** | 启动、停止、状态、报警、参数 |
| **内部变量** | 页面切换、临时状态、显示控制 |
| **数据库变量** | 产量、报表、历史数据 |
| **脚本变量** | 页面逻辑、计算结果 |
| **系统变量** | 当前用户、时间、语言 |
| **报警变量** | 报警位、报警字、报警文本 |
| **配方变量** | 产品参数组 |

## ③ 计算/选型示例

- 已知：  
  一个按钮应该写入启动变量，但实际写到了复位变量。
- 公式：  
  **变量绑定错误 → 操作异常**
- 计算：  
  用户按启动，PLC收到复位，设备行为异常。
- 结论：  
  画面管理必须包含变量绑定检查。

> [!tip] 记忆口诀
> 页面问题很多时候不是画错了，而是变量绑错了。

# 11 画面脚本管理

> [!info] 核心结论
> 画面脚本用于实现页面打开、按钮点击、数据计算、条件判断和动态显示，但脚本过多会增加维护难度。

## ① 分类/公式/规则

> [!warning] 脚本规则
> **简单逻辑尽量放PLC，显示辅助逻辑可放HMI脚本**

## ② 参考表

| 脚本位置 | 应用 |
|---|---|
| **页面打开脚本** | 初始化数据、刷新状态 |
| **页面关闭脚本** | 清除临时变量 |
| **按钮点击脚本** | 确认、保存、切换 |
| **定时脚本** | 周期刷新、时间判断 |
| **变量变化脚本** | 状态变化触发动作 |
| **数据处理脚本** | 换算、格式化、统计 |
| **权限脚本** | 判断是否允许操作 |

## ③ 计算/选型示例

- 已知：  
  画面打开速度很慢。
- 公式：  
  **页面慢 = 控件多 + 变量多 + 脚本重**
- 计算：  
  页面打开时执行大量数据库查询或循环计算，会导致卡顿。
- 结论：  
  画面脚本应简洁，复杂运算应放到PLC、网关或后台程序中。

> [!tip] 记忆口诀
> 脚本能少则少，能清楚则清楚，别把画面写成黑盒程序。

# 12 画面层级管理

> [!info] 核心结论
> 画面层级管理决定页面、弹窗、遮罩、导航栏和报警栏的显示优先级。

## ① 分类/公式/规则

> [!warning] 层级规则
> **普通页面 < 弹窗 < 重要报警 < 急停提示**

## ② 参考表

| 层级 | 示例 |
|---|---|
| **底层页面** | 首页、自动、参数 |
| **公共栏** | 顶部状态栏、底部导航栏 |
| **普通弹窗** | 参数输入、保存提示 |
| **确认弹窗** | 删除、清零、配方覆盖 |
| **报警弹窗** | 严重故障提示 |
| **安全提示** | 急停、安全门、光栅 |
| **锁屏层** | 权限锁定、登录界面 |

## ③ 计算/选型示例

- 已知：  
  报警弹窗被普通参数输入弹窗挡住。
- 公式：  
  **重要报警层级 > 普通弹窗**
- 计算：  
  严重报警应优先显示，不能被普通窗口遮挡。
- 结论：  
  画面管理中应定义弹窗优先级。

> [!tip] 记忆口诀
> 重要信息必须在最上层，不能被普通页面盖住。

# 13 画面刷新管理

> [!info] 核心结论
> 画面刷新频率影响系统流畅度和通讯负载，变量刷新太快、控件太多会导致卡顿。

## ① 分类/公式/规则

> [!warning] 刷新负载公式
> **刷新负载 = 变量数量 × 刷新频率 × 页面控件复杂度**

## ② 参考表

| 内容 | 建议 |
|---|---|
| **关键状态** | 可较快刷新 |
| **温度压力** | 中等刷新即可 |
| **产量统计** | 低频刷新即可 |
| **趋势曲线** | 按采样周期刷新 |
| **隐藏页面变量** | 尽量不刷新或降低刷新 |
| **大量表格** | 分页显示 |
| **复杂动画** | 避免过度使用 |

## ③ 计算/选型示例

- 已知：  
  一个页面有300个变量，每个100ms刷新一次。
- 公式：  
  **每秒刷新量 = 300 × 10**
- 计算：  
  每秒刷新3000次变量数据，负载较高。
- 结论：  
  应降低刷新频率、减少同屏变量或分页显示。

> [!tip] 记忆口诀
> 画面卡顿不一定是电脑差，也可能是变量刷得太猛。

# 14 多语言画面管理

> [!info] 核心结论
> 多语言画面管理用于统一维护按钮文字、报警文本、参数名称、单位和提示信息。

## ① 分类/公式/规则

> [!warning] 多语言公式
> **多语言工程 = 文本库 + 语言变量 + 控件引用 + 报警翻译 + 单位统一**

## ② 参考表

| 内容 | 说明 |
|---|---|
| **按钮文本** | 启动、停止、复位 |
| **菜单文本** | 首页、参数、报警、维护 |
| **报警文本** | 故障原因和处理建议 |
| **参数名称** | 速度、压力、温度 |
| **单位文本** | ℃、MPa、mm、s |
| **帮助文本** | 操作说明 |
| **日志文本** | 操作记录和事件说明 |

## ③ 计算/选型示例

- 已知：  
  设备需要中文和英文切换。
- 公式：  
  **文本统一管理 → 语言切换方便**
- 计算：  
  如果文字直接写死在图片里，切换语言会非常困难。
- 结论：  
  多语言工程应使用文本库管理，不建议文字图片化。

> [!tip] 记忆口诀
> 多语言不是复制两套画面，而是统一文本库切换。

# 15 画面版本管理

> [!info] 核心结论
> 画面版本管理用于记录页面修改历史，防止误删、误覆盖和现场版本不一致。

## ① 分类/公式/规则

> [!warning] 版本规则
> **每次交付 = 工程版本号 + 修改说明 + 备份文件 + 发布日期**

## ② 参考表

| 版本内容 | 说明 |
|---|---|
| **版本号** | V1.0、V1.1、V2.0 |
| **修改人** | 谁修改了工程 |
| **修改日期** | 修改发生时间 |
| **修改说明** | 改了哪些画面和功能 |
| **备份文件** | 可恢复旧版本 |
| **现场版本** | 当前设备实际运行版本 |
| **测试记录** | 修改后验证结果 |

## ③ 计算/选型示例

- 已知：  
  现场设备运行的是旧画面，但办公室工程是新画面。
- 公式：  
  **版本不一致 → 维护风险**
- 计算：  
  工程师按新画面变量修改，可能与现场实际工程不匹配。
- 结论：  
  每次下载画面前后都要确认版本号和备份。

> [!tip] 记忆口诀
> 没版本号的工程，迟早分不清哪个是最新版。

# 16 画面备份与恢复

> [!info] 核心结论
> 画面备份用于保存工程文件、运行参数、配方、报警文本、变量表和脚本，保证故障后可恢复。

## ① 分类/公式/规则

> [!warning] 备份公式
> **可恢复画面 = 工程文件 + 变量表 + 配方 + 脚本 + 图片资源 + 版本说明**

## ② 参考表

| 备份内容 | 作用 |
|---|---|
| **画面工程** | 页面和控件源文件 |
| **变量表** | 通讯地址和标签 |
| **图片资源** | 图标、背景、设备图 |
| **脚本文件** | 页面逻辑 |
| **报警文本** | 报警配置和说明 |
| **配方文件** | 工艺参数组 |
| **版本说明** | 便于恢复到正确版本 |

## ③ 计算/选型示例

- 已知：  
  HMI屏损坏，需要更换新屏。
- 公式：  
  **快速恢复 = 新屏 + 工程备份 + 参数备份**
- 计算：  
  如果只有空白屏，没有工程备份，恢复时间不可控。
- 结论：  
  画面工程必须随设备一起备份交付。

> [!tip] 记忆口诀

# 17 画面资源管理

> [!info] 核心结论
> 画面资源包括图片、图标、字体、颜色、控件库和动画素材，资源管理不好会导致工程臃肿、加载慢和风格混乱。

## ① 分类/公式/规则

> [!warning] 资源管理规则
> **资源统一命名、统一尺寸、统一风格、定期清理无用资源**

## ② 参考表

| 资源 | 注意 |
|---|---|
| **图标** | 风格统一，含义清楚 |
| **设备图片** | 不宜过大，避免加载慢 |
| **背景图** | 简洁，不影响文字阅读 |
| **字体** | 保证系统支持 |
| **颜色表** | 运行、报警、等待统一 |
| **控件库** | 标准按钮、灯、输入框 |
| **动画素材** | 简洁实用，避免过度炫酷 |

## ③ 计算/选型示例

- 已知：  
  HMI工程文件体积很大，打开页面很慢。
- 公式：  
  **工程臃肿 = 图片过大 + 无用资源多 + 动画复杂**
- 计算：  
  高分辨率图片直接导入小屏工程，会浪费存储和加载时间。
- 结论：  
  应压缩图片、清理无用资源，并统一控件库。

> [!tip] 记忆口诀
> 资源越乱，工程越重；图标统一，画面更专业。

# 18 画面测试管理

> [!info] 核心结论
> 画面测试用于验证页面跳转、按钮操作、变量显示、权限限制、报警弹窗、参数保存和异常恢复是否正常。

## ① 分类/公式/规则

> [!warning] 测试公式
> **画面验收 = 正常操作测试 + 异常报警测试 + 权限测试 + 断电恢复测试**

## ② 参考表

| 测试项 | 目标 |
|---|---|
| **页面跳转** | 所有菜单能正确跳转 |
| **按钮操作** | 启动、停止、复位有效 |
| **变量显示** | 数值、状态、单位正确 |
| **参数保存** | 修改后重启不丢失 |
| **报警显示** | 当前和历史报警正常 |
| **权限限制** | 无权限不能操作危险功能 |
| **弹窗逻辑** | 确认、取消、关闭正常 |
| **通讯中断** | 画面能显示离线状态 |

## ③ 计算/选型示例

- 已知：  
  画面上线前只测试了正常启动。
- 公式：  
  **只测正常不够，必须测试异常**
- 计算：  
  断网、报警、权限不足、参数越界等情况都应测试。
- 结论：  
  画面管理应包含测试清单和验收记录。

> [!tip] 记忆口诀
> 好画面不是能显示就行，还要异常时也不乱。

# 19 常见故障排查

> [!info] 核心结论
> 画面管理故障排查按 **页面结构、导航、权限、变量、脚本、模板、资源、版本** 的顺序进行。

## ① 分类/公式/规则

> [!warning] 排错公式
> **画面管理异常 = 页面丢失 + 跳转错误 + 权限错误 + 变量错绑 + 脚本错误 + 版本不一致**

## ② 参考表

| 现象 | 可能原因 |
|---|---|
| **页面打不开** | 页面被删除、路径错误、权限不足 |
| **跳转到错误页面** | 按钮导航地址配置错误 |
| **页面显示空白** | 资源丢失、控件异常、脚本错误 |
| **按钮灰色不可用** | 权限不足、条件未满足 |
| **页面卡顿** | 控件太多、变量刷新太快、图片太大 |
| **报警弹窗不显示** | 层级错误、触发条件错误 |
| **语言切换不完整** | 文本库缺项 |
| **现场画面与工程不一致** | 版本不一致或未下载最新工程 |

## ③ 计算/选型示例

- 已知：  
  参数页面按钮点击后跳到报警页面。
- 公式：  
  **跳转错误 = 按钮目标页面配置错误**
- 计算：  
  可能复制按钮后未修改跳转目标。
- 结论：  
  页面复制后必须逐项检查导航目标和变量绑定。

> [!tip] 记忆口诀
> 页面复制最容易忘改跳转和变量。

# 20 画面管理快速检查表

> [!info] 核心结论
> 画面管理的目标是让工程结构清楚、页面风格统一、变量绑定准确、权限安全可靠、后期维护方便。

## ① 分类/公式/规则

> [!note] 十步检查法
> ① 页面是否按功能分组  
> ② 页面名称是否统一规范  
> ③ 首页、自动、手动、参数、报警、维护是否完整  
> ④ 导航按钮是否正确跳转  
> ⑤ 页面权限是否符合角色要求  
> ⑥ 公共模板是否统一  
> ⑦ 弹窗层级和确认逻辑是否正确  
> ⑧ 变量绑定、数据类型和单位是否正确  
> ⑨ 版本号、修改记录和备份是否完整  
> ⑩ 上线前是否做正常、异常、权限和重启测试

> [!warning] 最终管理公式
> **合格画面管理 = 结构清楚 + 命名统一 + 导航正确 + 权限安全 + 变量准确 + 版本可追溯**

## ② 参考表

| 检查项 | 目标 |
|---|---|
| **分组** | 页面容易查找 |
| **命名** | 后期容易维护 |
| **导航** | 操作路径清晰 |
| **权限** | 防止误操作 |
| **模板** | 风格统一 |
| **变量** | 数据正确 |
| **弹窗** | 关键操作可确认 |
| **版本** | 修改可追溯 |
| **备份** | 故障可恢复 |

## ③ 计算/选型示例

- 已知：  
  一个HMI工程从10个页面扩展到80个页面。
- 公式：  
  **页面规模扩大 → 画面管理必须规范化**
- 计算：  
  页面越多，命名、分组、权限、模板和版本越重要。
- 结论：  
  大型HMI/SCADA工程必须建立画面管理规范，否则后期维护成本会快速上升。

> [!tip] 选型口诀
> 页面先分组，名称要统一；导航要清楚，权限要管住；模板要复用，变量要查准；版本要备份，交付要测试。
