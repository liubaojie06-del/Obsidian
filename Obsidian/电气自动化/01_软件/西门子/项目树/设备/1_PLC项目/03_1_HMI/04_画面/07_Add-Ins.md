# 01 Add-Ins 总览

> [!info] 核心结论
> **Add-Ins** 用于管理HMI/SCADA工程软件中的插件扩展功能，可用于增加工具、模板、自动化脚本、对象库、导入导出功能、诊断工具和企业标准化组件。

## ① 分类/公式/规则

> [!warning] 插件使用公式
> **Add-In可用 = 插件文件完整 + 版本兼容 + 证书可信 + 权限允许 + 状态启用**

## ② 参考表

| 项目 | 说明 |
|---|---|
| **Corporate Add-Ins** | 企业级插件，通常由公司或组织统一管理 |
| **Add-Ins** | 普通插件或用户级插件 |
| **名称** | 插件显示名称 |
| **状态** | 启用、禁用、异常、未加载等 |
| **详细信息** | 插件路径、作者、版本、产品、描述 |
| **可信度证明** | 证书、签名、可信发布者 |
| **所需权限** | 插件运行需要访问的工程、文件或系统权限 |

## ③ 计算/选型示例

- 已知：  
  一个HMI工程需要批量检查变量命名是否符合规范。
- 公式：  
  **重复检查任务 → 可用Add-In扩展工具**
- 计算：  
  插件可自动扫描变量、对象和画面，提高工程检查效率。
- 结论：  
  Add-Ins适合把常用工程工具插件化、标准化。

> [!tip] 记忆口诀
> 软件原生功能不够用，插件来扩展；插件能不能用，先看版本、证书和权限。

# 02 Corporate Add-Ins

> [!info] 核心结论
> **Corporate Add-Ins** 通常是企业统一发布和维护的插件，用于保证多个项目、多个工程师使用一致的工具和标准。

## ① 分类/公式/规则

> [!warning] 企业插件规则
> **公司标准工具、统一模板、统一检查规则 → 放Corporate Add-Ins**

## ② 参考表

| 内容 | 示例 |
|---|---|
| **标准检查插件** | 检查变量命名、画面命名、报警文本 |
| **企业模板插件** | 生成标准画面、导航栏、报警条 |
| **批量工具插件** | 批量导入变量、批量生成画面对象 |
| **导出工具插件** | 导出报警、变量、画面清单 |
| **项目审查插件** | 工程交付前自动检查 |
| **权限管理插件** | 检查页面权限设置 |
| **文档生成插件** | 自动生成工程说明或测试表 |

## ③ 计算/选型示例

- 已知：  
  公司要求所有项目都使用统一的报警文本格式。
- 公式：  
  **企业统一规范 → Corporate Add-In**
- 计算：  
  企业插件可扫描报警文本是否包含故障对象、原因和处理建议。
- 结论：  
  Corporate Add-Ins适合沉淀企业级工程规范。

> [!tip] 记忆口诀
> Corporate Add-Ins管标准，普通Add-Ins管扩展。

# 03 普通 Add-Ins

> [!info] 核心结论
> 普通 Add-Ins 多用于个人、项目或特定功能扩展，适合快速增强工程软件能力。

## ① 分类/公式/规则

> [!warning] 使用规则
> **项目临时工具 / 工程师个人工具 / 第三方扩展 → 普通Add-Ins**

## ② 参考表

| 插件类型 | 用途 |
|---|---|
| **对象生成插件** | 根据变量自动生成按钮、指示灯、I/O监控 |
| **变量导入插件** | 从Excel、CSV导入变量 |
| **报警导入插件** | 批量导入报警文本 |
| **画面检查插件** | 查找超出边界对象、未绑定变量对象 |
| **脚本辅助插件** | 生成常用C/VB脚本片段 |
| **资源管理插件** | 检查图片、字体、语言资源 |
| **通讯辅助插件** | 检查PLC连接配置或变量地址 |

## ③ 计算/选型示例

- 已知：  
  项目有500个I/O点，需要生成I/O监控页面。
- 公式：  
  **大量重复对象 → Add-In自动生成**
- 计算：  
  插件可根据变量表自动生成I/O名称、地址、状态灯和分组。
- 结论：  
  普通Add-In可以大幅减少重复画面开发工作。

> [!tip] 记忆口诀
> 重复劳动越多，插件价值越大。

# 04 详细信息

> [!info] 核心结论
> 详细信息用于查看插件的名称、路径、作者、修改日期、产品、版本、状态、描述和证书信息，是判断插件是否可信和可用的重要依据。

## ① 分类/公式/规则

> [!warning] 详细信息检查公式
> **插件可信 = 来源明确 + 作者明确 + 版本明确 + 路径明确 + 证书有效**

## ② 参考表

| 字段 | 说明 |
|---|---|
| **名称** | 插件名称 |
| **路径** | 插件文件所在位置 |
| **作者** | 插件开发者或发布者 |
| **修改日期** | 最近更新时间 |
| **产品** | 插件适用的软件或产品 |
| **版本** | 插件版本号 |
| **状态** | 当前是否启用或可用 |
| **描述** | 插件功能说明 |

## ③ 计算/选型示例

- 已知：  
  插件名称不清楚，路径来自未知文件夹，作者为空。
- 公式：  
  **来源不明 → 不建议启用**
- 计算：  
  插件可能存在兼容性、安全或稳定性风险。
- 结论：  
  启用前应确认插件来源、作者和功能说明。

> [!tip] 记忆口诀
> 插件能装不代表该装，先看详细信息。

# 05 插件状态

> [!info] 核心结论
> 插件状态表示插件当前是否可用，例如启用、禁用、加载失败、版本不兼容、证书不可信或权限不足。

## ① 分类/公式/规则

> [!warning] 状态判断公式
> **插件运行状态 = 文件完整性 + 版本兼容性 + 权限 + 证书 + 软件加载结果**

## ② 参考表

| 状态 | 含义 |
|---|---|
| **启用** | 插件可正常使用 |
| **禁用** | 插件存在但未启用 |
| **加载失败** | 插件文件异常或依赖缺失 |
| **版本不兼容** | 插件与当前软件版本不匹配 |
| **证书异常** | 签名无效或发布者不可信 |
| **权限不足** | 插件需要更高权限 |
| **未知状态** | 软件无法识别插件状态 |

## ③ 计算/选型示例

- 已知：  
  插件安装后没有出现在工具栏。
- 公式：  
  **未显示 = 未启用 / 加载失败 / 版本不兼容 / 权限不足**
- 计算：  
  先在Add-Ins面板查看状态，再看详细信息和证书。
- 结论：  
  插件问题优先查状态，不要直接重装软件。

> [!tip] 记忆口诀
> 插件不能用，先看状态栏。

# 06 插件版本

> [!info] 核心结论
> 插件版本决定其功能、兼容性和稳定性；工程软件升级后，旧插件可能无法正常运行。

## ① 分类/公式/规则

> [!warning] 版本兼容公式
> **插件兼容 = 插件版本支持当前工程软件版本**

## ② 参考表

| 版本信息 | 作用 |
|---|---|
| **主版本号** | 功能大改或架构变化 |
| **次版本号** | 新增功能或改进 |
| **修订号** | Bug修复 |
| **发布日期** | 判断是否过旧 |
| **兼容范围** | 支持哪些软件版本 |
| **更新说明** | 了解修改内容 |
| **回退版本** | 新版本异常时恢复 |

## ③ 计算/选型示例

- 已知：  
  工程软件从旧版本升级到新版本后，插件加载失败。
- 公式：  
  **软件升级 → 插件兼容性重新确认**
- 计算：  
  旧插件可能调用了新版本不再支持的接口。
- 结论：  
  软件升级前应检查所有Add-Ins是否有兼容版本。

> [!tip] 记忆口诀
> 软件升级别只看工程文件，插件也要一起检查。

# 07 插件路径

> [!info] 核心结论
> 插件路径用于定位插件文件来源，路径异常可能导致插件无法加载、权限不足或安全风险。

## ① 分类/公式/规则

> [!warning] 路径规则
> **插件路径应来自可信目录，不建议从临时文件夹或未知路径加载**

## ② 参考表

| 路径类型 | 风险/特点 |
|---|---|
| **企业统一目录** | 易管理，适合Corporate Add-Ins |
| **软件安装目录** | 稳定但修改需权限 |
| **用户目录** | 适合个人插件 |
| **项目目录** | 适合项目专用插件 |
| **网络共享目录** | 方便共享，但依赖网络 |
| **临时目录** | 不稳定，不推荐 |
| **未知路径** | 安全风险较高 |

## ③ 计算/选型示例

- 已知：  
  插件路径指向桌面临时文件夹。
- 公式：  
  **临时路径 → 易丢失 / 易误删 / 不稳定**
- 计算：  
  文件移动或删除后插件可能加载失败。
- 结论：  
  应把插件放到规范目录，并统一管理。

> [!tip] 记忆口诀
> 插件路径要固定，别放临时目录。

# 08 作者与发布者

> [!info] 核心结论
> 作者和发布者用于判断插件来源是否可信，企业项目中应优先使用可信发布者或内部验证过的插件。

## ① 分类/公式/规则

> [!warning] 来源可信公式
> **可信插件 = 已知作者 + 已知发布者 + 可验证签名 + 内部测试通过**

## ② 参考表

| 来源 | 建议 |
|---|---|
| **软件厂商官方** | 优先可信 |
| **企业内部开发** | 需内部测试和版本管理 |
| **系统集成商提供** | 需确认适用范围和责任 |
| **第三方工具** | 需验证证书和安全 |
| **网络下载插件** | 高风险，谨慎使用 |
| **无作者信息** | 不建议直接启用 |
| **来源未知** | 不建议用于正式工程 |

## ③ 计算/选型示例

- 已知：  
  插件作者为空，发布者未知。
- 公式：  
  **来源未知 → 安全风险高**
- 计算：  
  插件可能读取工程数据、修改脚本或生成对象。
- 结论：  
  正式工程中不应随意启用来源未知插件。

> [!tip] 记忆口诀
> 插件有权限改工程，来源必须查清楚。

# 09 可信度证明

> [!info] 核心结论
> 可信度证明通常指插件证书、数字签名或可信发布者信息，用于判断插件是否被篡改、是否来自可信来源。

## ① 分类/公式/规则

> [!warning] 证书判断公式
> **可信度 = 证书有效 + 发布者可信 + 文件未被篡改**

## ② 参考表

| 项目 | 说明 |
|---|---|
| **Add-In证书** | 插件签名证书 |
| **证书颁发者** | 谁颁发了证书 |
| **证书有效期** | 是否过期 |
| **发布者名称** | 插件发布方 |
| **数字签名** | 判断文件是否被修改 |
| **查看证书** | 查看证书详情 |
| **可信发布者** | 已被系统或企业信任的发布者 |

## ③ 计算/选型示例

- 已知：  
  插件证书过期或无签名。
- 公式：  
  **证书异常 → 不建议直接启用**
- 计算：  
  即使插件功能正常，也可能存在安全和合规风险。
- 结论：  
  企业项目应优先启用有有效证书的Add-In。

> [!tip] 记忆口诀
> 证书不是摆设，是插件可信的身份证。

# 10 所需权限

> [!info] 核心结论
> 所需权限表示插件运行时需要访问哪些资源，例如工程文件、变量表、画面对象、脚本、库文件、网络或系统目录。

## ① 分类/公式/规则

> [!warning] 权限最小化原则
> **插件权限 = 完成功能所需的最小权限**

## ② 参考表

| 权限类型 | 示例 |
|---|---|
| **读取工程** | 扫描画面、变量、报警 |
| **修改工程** | 自动生成对象、批量替换内容 |
| **访问文件** | 导入导出Excel、CSV、图片 |
| **执行脚本** | 调用自动化脚本 |
| **访问网络** | 连接服务器或许可证服务 |
| **访问库** | 读取项目库、全局库 |
| **管理员权限** | 修改系统级设置或安装目录 |

## ③ 计算/选型示例

- 已知：  
  一个只用于导出变量清单的插件要求管理员权限。
- 公式：  
  **权限需求 > 功能需求 → 需要谨慎**
- 计算：  
  只读导出通常不应需要过高权限。
- 结论：  
  权限异常过高时，应确认插件安全性和必要性。

> [!tip] 记忆口诀
> 插件权限越大，审核越要严格。

# 11 插件启用与禁用

> [!info] 核心结论
> 启用插件会让其功能加载到工程软件中；禁用插件可以减少风险、冲突和软件负载。

## ① 分类/公式/规则

> [!warning] 启用规则
> **只启用当前项目需要的插件，不需要的插件保持禁用**

## ② 参考表

| 操作 | 作用 |
|---|---|
| **启用插件** | 加载插件功能 |
| **禁用插件** | 停止插件运行 |
| **刷新插件列表** | 重新扫描插件状态 |
| **查看详细信息** | 检查来源和版本 |
| **查看证书** | 判断可信度 |
| **删除插件** | 移除无用或风险插件 |
| **更新插件** | 修复问题或增加功能 |

## ③ 计算/选型示例

- 已知：  
  工程软件启动变慢，加载了很多不用的插件。
- 公式：  
  **插件越多 → 启动负载越大**
- 计算：  
  禁用不需要的插件可减少加载时间和潜在冲突。
- 结论：  
  插件不是越多越好，应按项目需要启用。

> [!tip] 记忆口诀
> 用的插件才启用，不用的插件先关掉。

# 12 插件冲突

> [!info] 核心结论
> 多个插件可能修改相同对象、调用相同函数、占用相同资源或使用不同版本依赖，导致冲突。

## ① 分类/公式/规则

> [!warning] 冲突公式
> **插件冲突 = 功能重复 + 资源重复 + 版本依赖不一致**

## ② 参考表

| 冲突类型 | 说明 |
|---|---|
| **功能重复** | 两个插件都做变量导入 |
| **对象冲突** | 都修改同一画面对象 |
| **脚本冲突** | 同名函数被覆盖 |
| **资源冲突** | 使用同名图片或文本资源 |
| **版本冲突** | 依赖库版本不同 |
| **权限冲突** | 一个插件限制另一个插件操作 |
| **加载顺序冲突** | 插件加载先后影响结果 |

## ③ 计算/选型示例

- 已知：  
  启用两个变量导入插件后，变量表格式异常。
- 公式：  
  **同类插件同时修改变量表 → 冲突风险**
- 计算：  
  两个插件可能使用不同字段格式或覆盖规则。
- 结论：  
  同类插件尽量只启用一个，并先在测试工程中验证。

> [!tip] 记忆口诀
> 插件功能重叠时，冲突概率就上升。

# 13 插件与工程安全

> [!info] 核心结论
> 插件可能读取、修改或批量生成工程内容，因此必须做好备份、权限、测试和来源审查。

## ① 分类/公式/规则

> [!warning] 安全使用公式
> **安全使用插件 = 工程备份 + 插件可信 + 权限合理 + 测试工程验证 + 正式工程应用**

## ② 参考表

| 安全项 | 作用 |
|---|---|
| **工程备份** | 防止插件误改工程 |
| **来源审查** | 防止恶意或不可靠插件 |
| **权限检查** | 避免过度访问 |
| **测试工程验证** | 先小范围试用 |
| **操作日志** | 记录插件修改内容 |
| **版本记录** | 追溯插件版本 |
| **恢复方案** | 出错后能回退 |

## ③ 计算/选型示例

- 已知：  
  插件会批量替换变量地址。
- 公式：  
  **批量修改工程 → 必须先备份**
- 计算：  
  一旦替换错误，可能导致大量控件变量失效。
- 结论：  
  批量类插件必须先在备份工程或测试工程中运行。

> [!tip] 记忆口诀
> 插件越强大，越要先备份。

# 14 插件更新

> [!info] 核心结论
> 插件更新可修复Bug、提升兼容性或增加功能，但也可能改变行为、接口或依赖。

## ① 分类/公式/规则

> [!warning] 更新规则
> **插件更新前 = 查看更新说明 + 备份工程 + 测试验证 + 再正式使用**

## ② 参考表

| 更新内容 | 影响 |
|---|---|
| **Bug修复** | 解决已知问题 |
| **功能新增** | 增加新工具 |
| **接口变化** | 旧工程可能不兼容 |
| **权限变化** | 可能需要更多权限 |
| **证书更新** | 提高可信度 |
| **依赖更新** | 可能影响加载 |
| **界面变化** | 操作方式可能改变 |

## ③ 计算/选型示例

- 已知：  
  插件从V1.0升级到V2.0后，导入模板格式变化。
- 公式：  
  **版本大改 → 功能行为可能变化**
- 计算：  
  原来的Excel模板可能无法直接使用。
- 结论：  
  插件升级前应确认旧模板和旧工程是否兼容。

> [!tip] 记忆口诀
> 插件升级不是只点更新，还要看兼容性。

# 15 插件卸载

> [!info] 核心结论
> 卸载插件前必须确认工程是否仍依赖该插件生成的对象、脚本、资源或函数。

## ① 分类/公式/规则

> [!warning] 卸载检查公式
> **可卸载 = 工程不再依赖插件功能 + 资源已保留 + 脚本无缺失**

## ② 参考表

| 检查项 | 说明 |
|---|---|
| **插件生成对象** | 是否还在工程中使用 |
| **脚本函数** | 是否依赖插件函数 |
| **图形资源** | 是否由插件提供 |
| **导入导出模板** | 是否仍需使用 |
| **库对象** | 是否与插件绑定 |
| **运行功能** | 是否影响运行系统 |
| **工程备份** | 卸载前保留备份 |

## ③ 计算/选型示例

- 已知：  
  某插件用于生成标准电机组件，工程中已有这些组件。
- 公式：  
  **生成结果是否依赖插件运行？**
- 计算：  
  如果组件已独立存在，卸载可能不影响显示；如果运行时需要插件函数，则会报错。
- 结论：  
  卸载前必须确认插件是否仅用于开发，还是运行/编辑时仍被依赖。

> [!tip] 记忆口诀
> 插件卸载前，先查工程有没有依赖它。

# 17 Add-Ins 故障排查

> [!info] 核心结论
> Add-Ins故障排查按 **状态、路径、版本、证书、权限、依赖、冲突、日志** 的顺序进行。

## ① 分类/公式/规则

> [!warning] 排错公式
> **插件异常 = 状态未启用 + 路径错误 + 版本不兼容 + 证书异常 + 权限不足 + 插件冲突**

## ② 参考表

| 现象 | 可能原因 |
|---|---|
| **插件不显示** | 未安装、路径错误、未刷新 |
| **插件无法启用** | 版本不兼容、证书异常 |
| **插件加载失败** | 依赖缺失、文件损坏 |
| **插件运行报错** | 参数错误、权限不足 |
| **软件启动变慢** | 插件过多或插件加载慢 |
| **功能异常** | 插件冲突或工程版本不匹配 |
| **证书警告** | 发布者不可信或证书过期 |
| **无法访问工程** | 权限不足或工程锁定 |

## ③ 计算/选型示例

- 已知：  
  插件以前能用，今天突然加载失败。
- 公式：  
  **突然失败 = 路径变化 / 软件更新 / 权限变化 / 证书过期**
- 计算：  
  先检查插件文件是否还在原路径，再检查版本和证书。
- 结论：  
  插件故障不要只重启软件，要按状态和详细信息排查。

> [!tip] 记忆口诀
> 插件出问题，先看状态，再看版本和证书。

# 18 Add-Ins 快速检查表

> [!info] 核心结论
> Add-Ins使用前要确认来源可信、版本兼容、权限合理、证书有效、路径规范，并在测试工程中验证。

## ① 分类/公式/规则

> [!note] 十步检查法
> ① 确认插件属于Corporate Add-Ins还是普通Add-Ins  
> ② 查看插件名称、路径、作者和描述  
> ③ 检查插件版本和适用产品  
> ④ 检查插件状态是否正常  
> ⑤ 查看证书和可信度证明  
> ⑥ 检查所需权限是否合理  
> ⑦ 确认是否与其他插件冲突  
> ⑧ 正式使用前备份工程  
> ⑨ 先在测试工程中运行插件  
> ⑩ 使用后检查工程对象、变量、脚本和日志

> [!warning] 最终检查公式
> **合格Add-In = 来源可信 + 版本兼容 + 权限合理 + 证书有效 + 测试通过**

## ② 参考表

| 检查项 | 目标 |
|---|---|
| **名称** | 确认插件用途 |
| **路径** | 确认文件来源 |
| **作者** | 判断来源可信 |
| **版本** | 判断兼容性 |
| **状态** | 判断是否可用 |
| **证书** | 判断是否可信 |
| **权限** | 判断是否过度访问 |
| **描述** | 明确插件功能 |
| **测试** | 防止误改正式工程 |

## ③ 计算/选型示例

- 已知：  
  准备启用一个可批量修改画面对象的Add-In。
- 公式：  
  **批量修改类插件 = 高影响插件**
- 计算：  
  必须先备份工程，在测试工程中验证修改结果，再用于正式工程。
- 结论：  
  对工程有写入能力的Add-In，要比只读查看类Add-In更严格审查。

> [!tip] 选型口诀
> 插件先看来源，再看证书；先看版本，再看权限；先测后用，先备份后批量。
