# 01 HMI报警总览

> [!info] 核心结论
> **HMI报警** 用于把PLC、控制器、设备、通讯或系统中的异常状态显示给操作员，帮助现场快速发现故障、确认故障、处理故障并追溯历史记录。

## ① 分类/公式/规则

> [!warning] 报警系统公式
> **HMI报警 = 报警变量 + 报警文本 + 报警等级 + 时间戳 + 确认/复位 + 历史记录**

## ② 参考表

| 项目 | 说明 |
|---|---|
| **报警变量** | PLC报警位、报警字、故障码、系统报警 |
| **报警文本** | 说明具体故障原因 |
| **报警等级** | 提示、警告、停机、急停等 |
| **当前报警** | 当前仍然存在的报警 |
| **历史报警** | 曾经发生过的报警记录 |
| **报警确认** | 操作员确认已知晓报警 |
| **报警复位** | 故障消除后复位报警状态 |
| **报警记录** | 保存发生、确认、恢复时间 |

## ③ 计算/选型示例

- 已知：  
  PLC中有一个气压不足报警位。
- 公式：  
  **PLC报警位 → HMI报警变量 → 报警文本显示**
- 计算：  
  当报警位ON时，HMI显示“气压不足，请检查气源”。
- 结论：  
  HMI报警本质上是把PLC故障状态转换成现场人员能看懂的信息。

> [!tip] 记忆口诀
> PLC知道哪里故障，HMI负责告诉人怎么处理。

# 02 报警变量

> [!info] 核心结论
> 报警变量是触发HMI报警的信号来源，常见形式有Bool报警位、报警字、故障码、模拟量超限和系统通讯报警。

## ① 分类/公式/规则

> [!warning] 报警变量公式
> **报警触发 = 报警变量满足触发条件**

## ② 参考表

| 报警变量类型 | 说明 |
|---|---|
| **Bool报警位** | 一个报警对应一个Bool变量 |
| **报警字Word** | 一个Word包含多个报警位 |
| **故障码Int** | 数值对应不同故障文本 |
| **状态字** | 多个位表示不同设备状态 |
| **模拟量超限** | 温度、压力、液位超过阈值 |
| **通讯报警** | PLC、网关、服务器连接中断 |
| **系统报警** | Runtime、数据库、存储异常 |

## ③ 计算/选型示例

- 已知：  
  一个Word报警字有16个位，每个位对应一个故障。
- 公式：  
  **1个Word = 16个报警位**
- 计算：  
  AlarmWord.0~AlarmWord.15 可对应16条报警。
- 结论：  
  报警字适合集中管理报警，但必须建立清楚的位定义表。

> [!tip] 记忆口诀
> 报警变量不清楚，报警文本就容易对不上。

# 03 当前报警

> [!info] 核心结论
> 当前报警显示的是当前仍存在或尚未处理的故障，适合操作员实时排查设备异常。

## ① 分类/公式/规则

> [!warning] 当前报警规则
> **当前报警 = 报警条件仍然存在 / 尚未恢复的报警**

## ② 参考表

| 内容 | 作用 |
|---|---|
| **报警名称** | 显示故障内容 |
| **报警等级** | 判断严重程度 |
| **发生时间** | 记录故障开始时间 |
| **确认状态** | 判断是否已被操作员确认 |
| **设备来源** | 判断哪个设备或工位故障 |
| **处理建议** | 指导现场排查 |
| **恢复状态** | 判断故障是否消除 |

## ③ 计算/选型示例

- 已知：  
  安全门打开报警仍然存在。
- 公式：  
  **安全门未关闭 → 当前报警保持显示**
- 计算：  
  即使操作员点击确认，只要安全门还未关闭，报警仍然应显示。
- 结论：  
  当前报警不能因为确认就消失，必须等故障条件恢复。

> [!tip] 记忆口诀
> 当前报警看现在还坏不坏。

# 04 历史报警

> [!info] 核心结论
> 历史报警用于追溯故障发生、确认、恢复的全过程，是设备维护、停机分析和质量追溯的重要依据。

## ① 分类/公式/规则

> [!warning] 历史报警公式
> **历史报警 = 发生时间 + 报警内容 + 确认时间 + 恢复时间 + 操作用户**

## ② 参考表

| 字段 | 说明 |
|---|---|
| **发生时间** | 报警开始时间 |
| **确认时间** | 操作员确认时间 |
| **恢复时间** | 故障消除时间 |
| **报警文本** | 故障说明 |
| **报警等级** | 严重程度 |
| **确认用户** | 谁确认了报警 |
| **设备来源** | 哪个设备或工位报警 |

## ③ 计算/选型示例

- 已知：  
  某报警8:00发生，8:05确认，8:20恢复。
- 公式：  
  **故障持续时间 = 恢复时间 - 发生时间**
- 计算：  
  故障持续时间 = 8:20 - 8:00 = **20分钟**
- 结论：  
  历史报警可用于统计停机时间和故障频率。

> [!tip] 记忆口诀
> 当前报警用于处理，历史报警用于追溯。

# 05 报警等级

> [!info] 核心结论
> 报警等级用于区分故障严重程度，帮助操作员判断是否需要停机、立即处理或仅提示关注。

## ① 分类/公式/规则

> [!warning] 报警等级规则
> **越影响安全和设备保护，报警等级越高**

## ② 参考表

| 等级 | 典型含义 |
|---|---|
| **提示** | 普通提醒，不影响运行 |
| **警告** | 需要关注，可能影响稳定性 |
| **一般报警** | 需要处理，可能影响生产 |
| **停机报警** | 设备停止，需排故 |
| **急停报警** | 安全相关，必须立即处理 |
| **通讯报警** | 设备或系统连接异常 |
| **系统报警** | HMI、数据库、Runtime异常 |

## ③ 计算/选型示例

- 已知：  
  气压低于最低工作压力，设备气缸可能无法动作。
- 公式：  
  **影响设备动作 → 停机报警**
- 计算：  
  继续运行可能造成动作不到位和产品异常。
- 结论：  
  气压不足通常应设置为停机或高等级报警。

> [!tip] 记忆口诀
> 只是提醒叫提示，影响运行叫报警，影响安全叫急停。

# 06 报警文本

> [!info] 核心结论
> 报警文本要具体、清楚、可执行，最好包含故障对象、故障原因和处理建议。

## ① 分类/公式/规则

> [!warning] 报警文本公式
> **报警文本 = 故障对象 + 故障原因 + 处理建议**

## ② 参考表

| 文本质量 | 示例 |
|---|---|
| **差** | 设备故障 |
| **一般** | 气缸故障 |
| **较好** | 气缸1伸出不到位 |
| **优秀** | 气缸1伸出不到位，请检查伸出传感器和气源 |
| **对象** | 气缸1、伺服A、安全门、真空1 |
| **原因** | 未到位、超时、通讯中断、压力不足 |
| **建议** | 检查传感器、线路、气源、驱动器 |

## ③ 计算/选型示例

- 已知：  
  报警文本只写“故障”。
- 公式：  
  **报警不具体 → 排故时间增加**
- 计算：  
  操作员不知道该查气缸、伺服、安全门还是通讯。
- 结论：  
  报警文本必须具体到故障点。

> [!tip] 记忆口诀
> 好报警不是告诉你“坏了”，而是告诉你“哪里坏、怎么查”。

# 07 报警确认

> [!info] 核心结论
> 报警确认表示操作员已经看到并确认该报警，但确认不等于故障已经消除。

## ① 分类/公式/规则

> [!warning] 确认规则
> **报警确认 = 人已知晓报警，不代表设备已恢复**

## ② 参考表

| 状态 | 含义 |
|---|---|
| **未确认** | 报警发生后无人确认 |
| **已确认** | 操作员已确认报警 |
| **仍存在** | 故障条件仍然存在 |
| **已恢复** | 故障条件已消除 |
| **确认时间** | 操作员确认的时间 |
| **确认用户** | 谁确认了报警 |
| **确认记录** | 用于操作追溯 |

## ③ 计算/选型示例

- 已知：  
  安全门打开报警已确认，但安全门仍未关闭。
- 公式：  
  **已确认 + 故障仍存在 → 当前报警仍显示**
- 计算：  
  报警状态从“未确认”变为“已确认”，但不会恢复。
- 结论：  
  报警确认只是确认已知晓，故障仍需处理。

> [!tip] 记忆口诀
> 确认是“我知道了”，恢复是“故障没了”。

# 08 报警复位

> [!info] 核心结论
> 报警复位用于在故障消除后清除报警锁存或让PLC重新进入可运行状态，通常由PLC逻辑最终决定是否复位成功。

## ① 分类/公式/规则

> [!warning] 报警复位公式
> **复位成功 = 故障条件消除 + 复位命令有效 + PLC允许复位**

## ② 参考表

| 复位相关项 | 说明 |
|---|---|
| **复位按钮** | HMI写入复位命令 |
| **复位变量** | PLC接收的复位位 |
| **故障条件** | 必须先消除 |
| **PLC逻辑** | 判断是否允许复位 |
| **报警锁存** | 故障发生后保持报警 |
| **复位权限** | 部分报警需工程师权限 |
| **复位记录** | 记录用户和时间 |

## ③ 计算/选型示例

- 已知：  
  气压不足报警未消除，操作员点击复位。
- 公式：  
  **气压仍不足 → 复位无效**
- 计算：  
  PLC会继续保持或重新触发报警。
- 结论：  
  报警复位前必须先解决现场故障原因。

> [!tip] 记忆口诀
> 不处理原因，只点复位，报警还会回来。

# 09 报警锁存

> [!info] 核心结论
> 报警锁存用于保证报警发生后不会因为瞬间恢复而立即消失，便于操作员发现和追溯。

## ① 分类/公式/规则

> [!warning] 锁存公式
> **锁存报警 = 报警发生后保持，直到故障消除并执行复位**

## ② 参考表

| 报警方式 | 特点 |
|---|---|
| **非锁存报警** | 条件消失后报警自动消失 |
| **锁存报警** | 条件消失后仍需复位 |
| **确认锁存** | 需要人工确认 |
| **复位锁存** | 需要复位命令清除 |
| **安全报警锁存** | 常用于急停、安全门、光栅 |
| **过程报警锁存** | 常用于重要工艺异常 |
| **瞬时报警记录** | 即使很短也记录历史 |

## ③ 计算/选型示例

- 已知：  
  光栅被遮挡0.2秒后恢复。
- 公式：  
  **安全类报警 → 建议锁存**
- 计算：  
  如果不锁存，操作员可能完全看不到发生过异常。
- 结论：  
  重要安全类报警通常应锁存并记录历史。

> [!tip] 记忆口诀
> 重要报警要留痕，不能一闪就没。

# 10 报警延时

> [!info] 核心结论
> 报警延时用于过滤瞬时波动或信号抖动，防止误报警，但延时过长会导致故障发现变慢。

## ① 分类/公式/规则

> [!warning] 报警延时公式
> **正式报警 = 报警条件持续时间 ≥ 设定延时**

## ② 参考表

| 应用 | 说明 |
|---|---|
| **气压波动** | 可设置短延时 |
| **温度超限** | 可根据工艺设置延时 |
| **液位波动** | 可过滤波浪和抖动 |
| **传感器抖动** | 可避免误报警 |
| **通讯瞬断** | 可设置离线延时 |
| **安全报警** | 不宜设置过长延时 |
| **急停报警** | 应立即触发 |

## ③ 计算/选型示例

- 已知：  
  温度超过80℃持续2秒，但报警延时为5秒。
- 公式：  
  **2秒 < 5秒 → 不触发正式报警**
- 计算：  
  该次温度波动不会进入正式报警。
- 结论：  
  报警延时能减少瞬时波动误报警。

> [!tip] 记忆口诀
> 延时能防误报，但安全报警不能慢。

# 11 报警恢复

> [!info] 核心结论
> 报警恢复表示故障条件已经消失，系统可以记录恢复时间，并根据报警类型决定是否还需要人工复位。

## ① 分类/公式/规则

> [!warning] 恢复规则
> **报警恢复 = 报警条件不再满足**

## ② 参考表

| 状态 | 说明 |
|---|---|
| **发生** | 报警条件成立 |
| **确认** | 操作员确认已知晓 |
| **恢复** | 报警条件消失 |
| **复位** | 清除锁存或恢复运行 |
| **恢复时间** | 故障消失时间 |
| **持续时间** | 发生到恢复的时间 |
| **恢复记录** | 进入历史报警 |

## ③ 计算/选型示例

- 已知：  
  气压不足报警10:00发生，10:03气压恢复。
- 公式：  
  **报警持续时间 = 10:03 - 10:00**
- 计算：  
  持续时间 = **3分钟**
- 结论：  
  恢复时间可用于统计故障处理效率。

> [!tip] 记忆口诀
> 发生看故障开始，恢复看故障结束。

# 12 报警显示控件

> [!info] 核心结论
> 报警显示控件用于集中显示当前报警、历史报警、报警确认、报警过滤和报警详细信息。

## ① 分类/公式/规则

> [!warning] 报警控件公式
> **报警控件 = 报警源 + 显示列 + 过滤条件 + 操作按钮**

## ② 参考表

| 控件内容 | 说明 |
|---|---|
| **报警列表** | 显示多条报警 |
| **报警等级列** | 显示严重程度 |
| **时间列** | 发生、确认、恢复时间 |
| **确认列** | 显示确认状态 |
| **报警文本列** | 显示故障说明 |
| **设备来源列** | 显示工位或设备 |
| **操作按钮** | 确认、复位、过滤、导出 |

## ③ 计算/选型示例

- 已知：  
  现场需要快速查看当前还未处理的报警。
- 公式：  
  **报警控件过滤 = 当前报警 + 未恢复**
- 计算：  
  只显示仍存在的报警，隐藏已恢复历史报警。
- 结论：  
  当前报警和历史报警建议分开显示。

> [!tip] 记忆口诀
> 当前报警看处理，历史报警看追溯。

# 13 报警过滤

> [!info] 核心结论
> 报警过滤用于按设备、工位、等级、时间、确认状态或报警类型筛选报警，适合报警数量较多的系统。

## ① 分类/公式/规则

> [!warning] 过滤公式
> **报警显示结果 = 全部报警 - 不符合过滤条件的报警**

## ② 参考表

| 过滤方式 | 用途 |
|---|---|
| **按设备过滤** | 只看某台设备报警 |
| **按工位过滤** | 只看上料、检测、下料区域 |
| **按等级过滤** | 只看停机或急停报警 |
| **按时间过滤** | 查询某班次报警 |
| **按确认状态过滤** | 查未确认报警 |
| **按恢复状态过滤** | 查仍存在报警 |
| **按关键词过滤** | 查气压、伺服、通讯等报警 |

## ③ 计算/选型示例

- 已知：  
  一条产线有200条历史报警，需要查询昨天伺服相关报警。
- 公式：  
  **过滤条件 = 时间为昨天 + 文本包含伺服**
- 计算：  
  报警控件只显示符合条件的记录。
- 结论：  
  大型项目必须配置报警过滤功能。

> [!tip] 记忆口诀
> 报警少靠列表，报警多靠过滤。

# 14 报警声音与弹窗

> [!info] 核心结论
> 报警声音和弹窗用于提醒操作员注意重要故障，但应避免过度弹窗和过多声音造成干扰。

## ① 分类/公式/规则

> [!warning] 提示规则
> **严重报警强提醒，普通提示弱提醒**

## ② 参考表

| 提示方式 | 适合场景 |
|---|---|
| **声音报警** | 停机、急停、关键工艺异常 |
| **闪烁提示** | 当前重要报警 |
| **弹窗报警** | 需要立即处理的报警 |
| **状态栏提示** | 普通报警或提示 |
| **报警灯颜色** | 按等级显示 |
| **蜂鸣器确认** | 确认后停止声音 |
| **静默报警** | 只记录历史，不打扰操作 |

## ③ 计算/选型示例

- 已知：  
  每个小提示都弹窗并蜂鸣。
- 公式：  
  **提示过多 → 操作员疲劳**
- 计算：  
  操作员可能习惯性忽略所有弹窗。
- 结论：  
  只有高等级报警才建议弹窗和声音提示。

> [!tip] 记忆口诀
> 报警提醒要分级，不要所有报警都大喊大叫。

---

# 15 报警与权限

> [!info] 核心结论
> 不同报警操作应有不同权限，例如普通操作员可确认报警，工程师或管理员才能复位关键报警、清空历史或修改报警参数。

## ① 分类/公式/规则

> [!warning] 权限公式
> **报警操作允许 = 用户权限 + 报警等级 + 操作类型**

## ② 参考表

| 操作 | 推荐权限 |
|---|---|
| **查看当前报警** | 所有用户 |
| **查看历史报警** | 操作员及以上 |
| **确认报警** | 操作员及以上 |
| **普通复位** | 操作员或班组长 |
| **安全报警复位** | 工程师或管理员 |
| **清空报警历史** | 管理员 |
| **修改报警阈值** | 工程师或管理员 |

## ③ 计算/选型示例

- 已知：  
  普通操作员可以清空报警历史。
- 公式：  
  **清空历史 = 高风险操作**
- 计算：  
  清空后会影响故障追溯和责任分析。
- 结论：  
  清空历史报警应限制管理员权限，并加二次确认。

> [!tip] 记忆口诀
> 能看报警不代表能清报警，能确认不代表能删记录。

# 16 报警与PLC逻辑

> [!info] 核心结论
> HMI负责显示、确认和记录报警，报警判断和设备保护逻辑通常应由PLC完成。

## ① 分类/公式/规则

> [!warning] 分工规则
> **PLC负责判断与保护，HMI负责显示与操作入口**

## ② 参考表

| 功能 | 推荐实现位置 |
|---|---|
| **安全联锁** | PLC/安全PLC |
| **急停报警** | PLC/安全系统 |
| **动作超时判断** | PLC |
| **设备故障判断** | PLC |
| **报警显示** | HMI |
| **报警确认** | HMI + PLC配合 |
| **报警历史记录** | HMI/SCADA/数据库 |

## ③ 计算/选型示例

- 已知：  
  气缸伸出超时只在HMI脚本里判断。
- 公式：  
  **关键设备保护 → PLC判断更可靠**
- 计算：  
  如果HMI通讯中断或画面未运行，脚本可能不执行。
- 结论：  
  动作超时这类设备保护报警应放在PLC中判断。

> [!tip] 记忆口诀
> 保护逻辑放PLC，显示记录放HMI。

# 17 报警与通讯中断

> [!info] 核心结论
> 通讯中断时，HMI无法继续读取PLC报警变量，因此必须单独配置通讯报警或连接状态报警。

## ① 分类/公式/规则

> [!warning] 通讯报警公式
> **连接状态异常 → 通讯报警触发**

## ② 参考表

| 通讯报警对象 | 示例 |
|---|---|
| **PLC连接中断** | 主PLC离线 |
| **远程I/O掉站** | 分布式I/O离线 |
| **网关断线** | 工业网关无响应 |
| **数据库断开** | 历史报警无法写入 |
| **MES断开** | 数据上传失败 |
| **OPC UA断开** | 服务器连接失败 |
| **网络异常** | IP不可达、端口不通 |

## ③ 计算/选型示例

- 已知：  
  PLC断线后，设备报警位无法刷新。
- 公式：  
  **PLC离线 → 触发HMI通讯报警**
- 计算：  
  HMI应显示“主PLC通讯中断”，而不是只显示空白数据。
- 结论：  
  通讯报警是报警系统的重要组成部分。

> [!tip] 记忆口诀
> 设备报警靠PLC变量，通讯报警靠连接状态。

# 18 报警记录与存储

> [!info] 核心结论
> 报警记录需要保存到HMI本地、数据库或服务器，必须规划保存周期、容量、导出和备份。

## ① 分类/公式/规则

> [!warning] 报警容量公式
> **报警记录容量 = 每条记录大小 × 每日报警条数 × 保存天数**

## ② 参考表

| 存储项 | 说明 |
|---|---|
| **本地存储** | HMI或工控机本地保存 |
| **数据库存储** | SQL Server、MySQL等 |
| **CSV文件** | 便于导出分析 |
| **保存天数** | 30天、90天、1年等 |
| **自动清理** | 防止磁盘满 |
| **备份策略** | 定期备份报警记录 |
| **导出功能** | 给维护和质量分析使用 |

## ③ 计算/选型示例

- 已知：  
  每天约1000条报警记录，每条约500字节，保存90天。
- 公式：  
  **容量 = 1000 × 500 × 90**
- 计算：  
  容量 = **45,000,000字节 ≈ 45MB**
- 结论：  
  报警记录容量不一定很大，但仍需设置清理和备份策略。

> [!tip] 记忆口诀
> 报警记录要能查，也要防止存储爆满。

# 19 报警多语言

> [!info] 核心结论
> 多语言项目中，报警文本必须准确翻译，尤其是故障对象、故障原因和处理建议不能丢失。

## ① 分类/公式/规则

> [!warning] 多语言报警公式
> **报警文本多语言 = 文本ID + 中文 + 英文 + 其他语言 + 处理建议**

## ② 参考表

| 内容 | 注意 |
|---|---|
| **故障对象** | 翻译必须具体 |
| **故障原因** | 不能简化成Alarm |
| **处理建议** | 应保留操作指导 |
| **单位** | 温度、压力、速度单位不能漏 |
| **等级名称** | Warning、Fault、Emergency等统一 |
| **按钮文本** | Confirm、Reset、History统一 |
| **文本长度** | 英文可能更长，注意控件宽度 |

## ③ 计算/选型示例

- 已知：  
  中文报警为“气缸1伸出不到位，请检查伸出传感器”。
- 公式：  
  **翻译要保留对象 + 原因 + 建议**
- 计算：  
  英文不能只写“Cylinder alarm”。
- 结论：  
  报警翻译必须服务于现场排故。

> [!tip] 记忆口诀
> 报警翻译不是装饰，是维修说明。

# 20 常见故障排查

> [!info] 核心结论
> HMI报警异常排查按 **报警变量、连接状态、报警配置、文本映射、确认复位、历史记录、权限、数据库** 的顺序进行。

## ① 分类/公式/规则

> [!warning] 排错公式
> **报警异常 = 变量错误 + 连接异常 + 配置错误 + 文本错误 + 权限不足 + 存储失败**

## ② 参考表

| 现象 | 可能原因 |
|---|---|
| **报警不显示** | 变量未绑定、报警位未触发、报警配置未启用 |
| **报警文本不对** | 位映射错误、文本ID错误 |
| **报警无法确认** | 权限不足、控件配置错误 |
| **报警无法复位** | 故障未消除、复位变量错误、PLC不允许 |
| **历史报警为空** | 历史记录未启用、数据库异常 |
| **报警时间错误** | HMI时间不准、未同步NTP |
| **报警频繁闪现** | 信号抖动、未设置延时 |
| **报警太多难查** | 未分级、未分组、未过滤 |

## ③ 计算/选型示例

- 已知：  
  PLC报警位已经ON，但HMI没有报警。
- 公式：  
  **报警位ON + HMI无显示 = 变量/报警配置问题**
- 计算：  
  检查HMI变量是否在线、地址是否正确、报警是否启用。
- 结论：  
  先在线监控报警变量，再查报警文本和控件配置。

> [!tip] 记忆口诀
> 报警不显示先查变量，报警不对先查映射。

# 21 HMI报警快速检查表

> [!info] 核心结论
> HMI报警检查的目标是确保报警能准确触发、清楚显示、正确确认、可靠复位、完整记录、便于追溯。

## ① 分类/公式/规则

> [!note] 十步检查法
> ① 检查报警变量来源是否正确  
> ② 检查报警位、报警字或故障码映射是否正确  
> ③ 检查报警文本是否具体清楚  
> ④ 检查报警等级是否合理  
> ⑤ 检查当前报警和历史报警是否分开  
> ⑥ 检查报警确认和复位逻辑是否正确  
> ⑦ 检查通讯中断报警是否配置  
> ⑧ 检查报警记录、保存周期和导出功能  
> ⑨ 检查权限、清空历史和高等级报警操作限制  
> ⑩ 上线前测试发生、确认、恢复、复位和断电重启

> [!warning] 最终检查公式
> **合格HMI报警 = 触发准确 + 文本清楚 + 等级合理 + 操作安全 + 记录完整 + 可追溯**

## ② 参考表

| 检查项 | 目标 |
|---|---|
| **变量** | 报警源正确 |
| **文本** | 现场能看懂 |
| **等级** | 严重程度清楚 |
| **确认** | 操作员已知晓 |
| **复位** | 故障消除后恢复 |
| **历史** | 故障可追溯 |
| **权限** | 防止误操作 |
| **通讯报警** | 设备离线能提示 |
| **存储** | 记录不丢失 |

## ③ 计算/选型示例

- 已知：  
  设备有气压不足、安全门打开、伺服报警、PLC通讯中断四类报警。
- 公式：  
  **报警分类 = 工艺报警 + 安全报警 + 驱动报警 + 通讯报警**
- 计算：  
  每类报警应有不同等级、文本、处理建议和记录方式。
- 结论：  
  推荐按报警来源和严重程度建立分组、等级和过滤规则。

> [!tip] 选型口诀
> 报警变量要准，报警文本要清；当前报警看处理，历史报警看追溯；确认不等于恢复，复位必须看PLC。
