# 01 公共数据总览

> [!info] 核心结论
> **公共数据** 是项目级共享配置区，用来统一管理协作设备、报警类别、系统诊断、监控、日志和指令配置文件。

## ① 分类/公式/规则

> [!note] 🟦 功能主线
> 公共数据不是某一台PLC或HMI的专属数据，而是面向整个项目的 **通用配置、统一标准和共享规则**。

> [!warning] 🟧 公共数据配置公式
> **公共数据 = 协作设备 + 报警类别 + 系统诊断 + 监控设置 + 日志 + 指令配置文件**

## ② 参考表

| 项目 | 主要作用 |
|---|---|
| **协作设备** | 多设备之间的协同关系 |
| **报警类别** | 统一报警等级和分类 |
| **系统诊断设置** | 配置系统诊断信息 |
| **监控设置** | 监控变量、状态或设备信息 |
| **日志** | 记录事件、报警、运行数据 |
| **指令配置文件** | 管理指令或功能配置 |

## ③ 计算/选型示例

- 已知：  
  一个项目包含 PLC、HMI、驱动、远程I/O，需要统一报警和诊断。
- 公式：  
  **项目公共配置 = 设备协作 + 报警规则 + 诊断规则 + 日志记录**
- 计算：  
  如果每个设备单独配置，后期维护容易混乱；统一放到公共数据更清晰。
- 结论：  
  应在 **公共数据** 中统一规划报警类别、诊断设置、日志和协作设备，具体功能需按TIA版本和设备支持情况校核。

> [!tip] 记忆口诀
> 公共数据管全局，报警诊断统一配。

# 02 协作设备

> [!info] 核心结论
> **协作设备** 用于管理多个设备之间的协同关系，适合多PLC、多HMI、多站点联动项目。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> 协作设备强调“设备之间一起工作”，不是孤立配置某一台设备。

> [!warning] 🟧 协作关系公式
> **设备协作 = 设备对象 + 通讯关系 + 数据交互 + 功能联动**

## ② 参考表

| 项目 | 关注点 |
|---|---|
| **设备对象** | PLC、HMI、驱动、I/O站 |
| **协作方式** | 数据交换、状态共享、报警联动 |
| **通讯基础** | 网络连接、地址、设备名 |
| **应用场景** | 产线联动、多站控制 |
| **校核重点** | 通讯协议、权限、版本兼容 |

## ③ 计算/选型示例

- 已知：  
  一条产线有主PLC、从PLC和HMI，需要共享运行状态。
- 公式：  
  **协作成功 = 网络连通 + 数据一致 + 设备角色明确**
- 计算：  
  主PLC负责整线协调，从PLC负责局部控制，HMI负责显示和操作。
- 结论：  
  应通过 **协作设备** 明确设备之间的协同关系，并配合网络和变量配置完成联动。

> [!tip] 记忆口诀
> 多设备不怕多，关系理清就好做。

# 03 报警类别

> [!info] 核心结论
> **报警类别** 用于统一管理报警等级、显示方式和处理优先级，是HMI和诊断系统的重要基础。

## ① 分类/公式/规则

> [!note] 🟦 报警分类思路
> 报警不要全部混在一起，应按严重程度和处理方式分类。

> [!warning] 🟧 报警分级公式
> **报警类别 = 严重程度 + 处理优先级 + 显示方式 + 是否需要确认**

## ② 参考表

| 报警类别 | 建议含义 |
|---|---|
| **故障报警** | 设备停止或严重异常 |
| **警告报警** | 不立即停机，但需关注 |
| **提示信息** | 操作提示或状态提醒 |
| **安全报警** | 安全回路或人员保护相关 |
| **系统诊断** | 模块、通讯、系统状态异常 |

## ③ 计算/选型示例

- 已知：  
  电机过载、气压低、门未关、生产完成都需要提示。
- 公式：  
  **报警等级 = 风险程度 + 是否停机 + 是否人工确认**
- 计算：  
  电机过载属于故障报警；气压低属于警告；门未关可能属于安全或联锁；生产完成属于提示信息。
- 结论：  
  应将报警按类别分类，避免所有信息都显示成同一等级。

> [!tip] 记忆口诀
> 故障要停机，警告要注意，提示只提醒，安全最优先。

# 04 系统诊断设置

> [!info] 核心结论
> **系统诊断设置** 用于配置设备、模块、通讯和系统状态的诊断信息，帮助快速定位硬件和网络问题。

## ① 分类/公式/规则

> [!note] 🟦 诊断对象
> 系统诊断通常关注 CPU、I/O模块、通讯模块、远程站点、驱动设备和网络状态。

> [!warning] 🟧 诊断有效公式
> **有效诊断 = 诊断启用 + 对象正确 + 信息显示 + 报警关联**

## ② 参考表

| 诊断对象 | 常见信息 |
|---|---|
| **CPU** | 运行状态、故障、模式 |
| **I/O模块** | 模块掉线、通道故障 |
| **通讯网络** | 设备不可达、通讯中断 |
| **远程I/O** | 站点故障、模块缺失 |
| **驱动设备** | 驱动故障、状态异常 |

## ③ 计算/选型示例

- 已知：  
  现场远程I/O站偶尔掉线，HMI需要显示诊断信息。
- 公式：  
  **诊断显示 = 设备诊断 + 通讯状态 + HMI报警**
- 计算：  
  只在PLC中有诊断不够，还需要把诊断信息显示到HMI或报警系统中。
- 结论：  
  应配置 **系统诊断设置**，并确认HMI能正确显示对应诊断信息。

> [!tip] 记忆口诀
> 程序故障看逻辑，硬件问题看诊断。

# 05 监控设置

> [!info] 核心结论
> **监控设置** 用于观察设备状态、变量变化和运行过程，是调试和维护的重要工具。

## ① 分类/公式/规则

> [!note] 🟦 监控重点
> 监控不是盲目看所有变量，而是选择关键变量、关键状态和关键条件。

> [!warning] 🟧 监控价值公式
> **监控价值 = 关键变量 + 触发条件 + 状态变化 + 故障关联**

## ② 参考表

| 监控对象 | 重点看什么 |
|---|---|
| **输入信号** | 传感器是否到位 |
| **输出信号** | 执行器是否动作 |
| **中间变量** | 程序条件是否成立 |
| **报警位** | 故障是否触发 |
| **通讯状态** | 设备是否在线 |

## ③ 计算/选型示例

- 已知：  
  气缸不动作，需要判断是传感器、程序条件还是电磁阀问题。
- 公式：  
  **动作分析 = 输入条件 + 程序逻辑 + 输出状态 + 反馈信号**
- 计算：  
  先看启动条件是否成立，再看输出是否置位，最后看反馈是否返回。
- 结论：  
  通过 **监控设置** 选择关键变量，可快速定位故障位置。

> [!tip] 记忆口诀
> 监控别贪多，抓住输入、输出和条件。

# 06 日志

> [!info] 核心结论
> **日志** 用于记录设备运行、报警、事件和数据变化，方便故障追溯和生产分析。

## ① 分类/公式/规则

> [!note] 🟦 日志作用
> 日志的核心价值是“事后可追溯”，特别适合分析偶发故障、报警历史和生产过程。

> [!warning] 🟧 日志配置公式
> **有效日志 = 记录对象 + 触发条件 + 时间戳 + 存储位置 + 查询方式**

## ② 参考表

| 日志类型 | 用途 |
|---|---|
| **报警日志** | 查看历史报警 |
| **事件日志** | 记录操作和状态变化 |
| **数据日志** | 记录温度、压力、速度等 |
| **诊断日志** | 记录系统和设备诊断 |
| **操作日志** | 追踪人员操作记录 |

## ③ 计算/选型示例

- 已知：  
  设备每天随机停机一次，现场无法复现。
- 公式：  
  **故障追溯 = 报警日志 + 事件日志 + 关键数据记录**
- 计算：  
  通过查看停机前后的报警、变量和操作记录，分析故障原因。
- 结论：  
  应配置 **日志**，记录关键报警和运行数据，存储周期和容量需按实际规范校核。

> [!tip] 记忆口诀
> 当场查不到，日志帮你找。

# 07 指令配置文件

> [!info] 核心结论
> **指令配置文件** 用于管理项目中某些指令、功能或配置参数，便于标准化和复用。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> 指令配置文件可以理解为与指令或功能块相关的配置资料，用于统一参数、规范调用和减少重复配置。

> [!warning] 🟧 配置文件价值公式
> **配置文件价值 = 标准化参数 + 重复使用 + 易维护 + 版本一致**

## ② 参考表

| 项目 | 关注点 |
|---|---|
| **配置对象** | 指令、功能块、工艺对象 |
| **作用** | 参数统一、调用规范 |
| **适合场景** | 多设备、多工位、重复逻辑 |
| **风险点** | 版本不一致、参数误改 |
| **校核重点** | TIA版本、库版本、设备支持 |

## ③ 计算/选型示例

- 已知：  
  项目中多个工位使用相同的控制逻辑和参数结构。
- 公式：  
  **标准化收益 = 重复次数 × 配置一致性 × 维护便利性**
- 计算：  
  如果每个工位单独配置，容易出现参数不一致；使用配置文件可统一管理。
- 结论：  
  可通过 **指令配置文件** 管理相关配置，但具体可配置内容需按TIA版本和功能支持情况校核。

> [!tip] 记忆口诀
> 重复功能要标准，配置文件来帮忙。

# 08 公共数据快速检查表

> [!info] 核心结论
> 公共数据配置前，先确认项目是否需要统一报警、诊断、日志、监控和多设备协作。

## ① 分类/公式/规则

> [!note] 🟦 五步检查法
> ① 是否有多设备协作  
> ② 是否需要统一报警类别  
> ③ 是否启用系统诊断  
> ④ 是否需要监控关键变量  
> ⑤ 是否需要日志追溯

> [!warning] 🟧 最终检查公式
> **公共数据完整 = 协作清楚 + 报警统一 + 诊断可见 + 监控有效 + 日志可查**

## ② 参考表

| 检查项 | 建议 |
|---|---|
| **协作设备** | 多设备项目建议配置 |
| **报警类别** | 按严重程度统一 |
| **系统诊断** | 调试和维护项目建议启用 |
| **监控设置** | 只选关键变量 |
| **日志** | 偶发故障和追溯场景必备 |
| **指令配置文件** | 重复功能建议标准化 |

## ③ 计算/选型示例

- 已知：  
  一个项目有多台设备，客户要求报警可追溯、故障可诊断、数据可查询。
- 公式：  
  **项目可维护性 = 报警分类 + 系统诊断 + 日志记录 + 监控手段**
- 计算：  
  只写PLC程序无法满足完整维护需求，还要配置公共数据。
- 结论：  
  应统一配置 **报警类别、系统诊断设置、监控设置和日志**，并按项目规范校核记录周期、存储容量和权限。

> [!tip] 选型口诀
> 报警先分类，诊断要打开；关键量要监控，重要事进日志。
