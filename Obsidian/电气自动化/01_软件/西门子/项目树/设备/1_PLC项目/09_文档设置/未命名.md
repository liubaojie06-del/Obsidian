# 01 文档设置总览

> [!info] 核心结论
> **文档设置** 用于管理项目文档的基础信息、打印框架和封面，是生成规范化工程资料的重要入口。

## ① 分类/公式/规则

> [!note] 🟦 功能主线
> 文档设置主要包括三部分：  
> **文档信息**：填写项目、客户、作者等基础资料；  
> **框架**：设置图框、页眉、页脚、版式；  
> **封面**：设置项目文档首页信息。

> [!warning] 🟧 文档规范公式
> **规范文档 = 文档信息完整 + 框架统一 + 封面清晰 + 版本明确**

## ② 参考表

| 项目 | 主要作用 |
|---|---|
| **文档设置** | 管理项目文档输出规则 |
| **文档信息** | 填写项目基础资料 |
| **框架** | 设置页面框架、页眉页脚 |
| **封面** | 生成项目文档首页 |
| **输出文档** | 用于打印、归档、交付 |

## ③ 计算/选型示例

- 已知：  
  项目需要交付PLC程序说明、硬件清单和网络配置文档。
- 公式：  
  **交付文档 = 项目信息 + 图框模板 + 封面 + 正文内容**
- 计算：  
  如果没有统一文档设置，打印出来的资料可能缺少项目名、版本号或页码。
- 结论：  
  应先配置 **文档设置**，再生成或打印项目文档。

> [!tip] 记忆口诀
> 文档先设好，交付才专业。

---

# 02 文档信息

> [!info] 核心结论
> **文档信息** 用于填写项目名称、客户、作者、版本、日期等基础资料，是文档追溯的关键。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> 文档信息相当于工程资料的“身份证”，用于说明这份文档属于哪个项目、哪个版本、由谁编制。

> [!warning] 🟧 信息完整公式
> **完整文档信息 = 项目名称 + 客户名称 + 作者 + 日期 + 版本号 + 说明**

## ② 参考表

| 信息项 | 建议填写 |
|---|---|
| **项目名称** | 项目或设备名称 |
| **客户名称** | 使用方或业主单位 |
| **作者/工程师** | 编制人或负责人 |
| **日期** | 创建或更新日期 |
| **版本号** | V1.0、V1.1、Release等 |
| **备注说明** | 修改说明、交付说明 |

## ③ 计算/选型示例

- 已知：  
  项目经历了调试版、客户确认版、最终交付版三个阶段。
- 公式：  
  **文档可追溯 = 版本号 + 日期 + 修改说明**
- 计算：  
  每次交付前都更新版本号和日期，避免客户拿到旧资料。
- 结论：  
  文档信息必须与当前项目版本保持一致，具体字段按公司文档规范校核。

> [!tip] 记忆口诀
> 文档有没有用，先看信息清不清。

---

# 03 框架

> [!info] 核心结论
> **框架** 用于设置文档页面的固定版式，例如页眉、页脚、图框、页码和公司信息。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> 框架决定文档打印出来的外观结构，类似电气图纸中的图框模板。

> [!warning] 🟧 框架配置公式
> **标准框架 = 页眉 + 页脚 + 页码 + 项目信息 + 公司标识**

## ② 参考表

| 框架元素 | 作用 |
|---|---|
| **页眉** | 显示项目名、文档名 |
| **页脚** | 显示页码、日期、版本 |
| **图框** | 统一页面边界和版式 |
| **公司信息** | 公司名称、部门、标识 |
| **项目字段** | 客户、设备、工程编号 |

## ③ 计算/选型示例

- 已知：  
  公司要求所有工程文档都带有项目名、页码、版本号和公司名称。
- 公式：  
  **统一输出 = 框架模板 + 文档字段 + 自动页码**
- 计算：  
  将这些信息写入框架后，所有打印页面都能自动保持一致。
- 结论：  
  应配置统一 **框架**，避免每页手动修改造成格式不一致。

> [!tip] 记忆口诀
> 框架定版式，页页都统一。

---

# 04 封面

> [!info] 核心结论
> **封面** 是项目文档的首页，用于展示项目名称、客户、版本、日期和编制信息。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> 封面不是装饰，而是交付资料的首页索引，方便客户和维护人员快速确认文档归属。

> [!warning] 🟧 封面信息公式
> **标准封面 = 项目名称 + 文档名称 + 客户 + 版本 + 日期 + 编制人**

## ② 参考表

| 封面内容 | 作用 |
|---|---|
| **项目名称** | 明确所属项目 |
| **文档名称** | 说明文档类型 |
| **客户名称** | 明确交付对象 |
| **版本号** | 区分不同版本 |
| **日期** | 标识交付时间 |
| **编制/审核** | 明确责任人 |

## ③ 计算/选型示例

- 已知：  
  需要输出一份“PLC硬件组态说明”给客户。
- 公式：  
  **封面识别度 = 项目名 + 文档名 + 版本号**
- 计算：  
  封面写清楚项目、文档类型和版本，客户归档时不容易混淆。
- 结论：  
  应配置 **封面**，并确保封面信息与文档信息一致。

> [!tip] 记忆口诀
> 封面写清楚，资料不迷路。

---

# 05 文档交付规范

> [!info] 核心结论
> 文档交付前，要确认文档信息、框架、封面、版本和打印效果全部正确。

## ① 分类/公式/规则

> [!note] 🟦 交付前五查
> ① 项目名称是否正确  
> ② 版本号是否正确  
> ③ 封面是否完整  
> ④ 页眉页脚是否统一  
> ⑤ 打印预览是否正常

> [!warning] 🟧 交付检查公式
> **合格文档 = 信息正确 + 格式统一 + 版本清晰 + 可打印 + 可追溯**

## ② 参考表

| 检查项 | 建议 |
|---|---|
| **文档信息** | 与项目版本一致 |
| **框架** | 页码、页眉、页脚完整 |
| **封面** | 项目名、客户、版本齐全 |
| **打印预览** | 检查换页和排版 |
| **归档命名** | 文件名包含日期和版本 |

## ③ 计算/选型示例

- 已知：  
  项目已经完成现场调试，准备输出最终交付资料。
- 公式：  
  **最终交付资料 = Release版本 + 正确封面 + 统一框架 + 完整内容**
- 计算：  
  如果文档版本仍是调试版，客户后期维护可能拿错资料。
- 结论：  
  交付前必须更新文档信息和封面版本，并检查打印效果。

> [!tip] 选型口诀
> 信息要准，框架要齐，封面要清，版本要明。
