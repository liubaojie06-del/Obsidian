# 01 版本控制接口总览

> [!info] 核心结论
> **版本控制接口** 用于把 TIA Portal 项目与版本管理流程连接起来，方便进行项目备份、版本追踪、多人协作和变更回溯。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> 版本控制接口不是某个PLC或HMI设备，而是项目管理工具，用来帮助工程师管理 **谁改了、改了什么、什么时候改的、能不能回退**。

> [!warning] 🟧 版本管理公式
> **可靠版本管理 = 项目备份 + 变更记录 + 版本命名 + 权限控制 + 可回退**

## ② 参考表

| 项目 | 作用 |
|---|---|
| **版本控制接口** | 连接项目与版本管理流程 |
| **项目版本** | 记录不同阶段的工程状态 |
| **变更记录** | 追踪程序、画面、硬件改动 |
| **版本回退** | 出问题时恢复旧版本 |
| **多人协作** | 降低覆盖、误改、冲突风险 |

## ③ 计算/选型示例

- 已知：  
  一个项目有多名工程师同时参与，包含PLC程序、HMI画面和硬件组态。
- 公式：  
  **项目风险 = 多人修改 + 无版本记录 + 无备份 + 无回退方案**
- 计算：  
  如果没有版本控制，现场调试出现问题时，很难判断是哪次修改导致故障。
- 结论：  
  应使用 **版本控制接口** 或规范化版本管理流程，保证项目可追溯、可比较、可恢复。

> [!tip] 记忆口诀
> 项目能改不可怕，怕的是改完找不回。

# 02 版本控制接口的作用

> [!info] 核心结论
> 版本控制接口的核心作用，是让工程项目从“靠文件名管理”升级为“靠版本记录管理”。

## ① 分类/公式/规则

> [!note] 🟦 主要作用
> 版本控制接口通常用于：  
> ① 记录工程变更  
> ② 管理历史版本  
> ③ 支持多人协作  
> ④ 辅助项目回退  
> ⑤ 保留交付版本

> [!warning] 🟧 版本追踪公式
> **一次有效提交 = 修改内容 + 修改原因 + 修改人 + 修改时间 + 可恢复版本**

## ② 参考表

| 功能 | 价值 |
|---|---|
| **变更追踪** | 知道哪里被改过 |
| **历史记录** | 查看项目演变过程 |
| **版本对比** | 分析修改差异 |
| **版本回退** | 恢复到稳定状态 |
| **协作管理** | 降低多人误覆盖风险 |

## ③ 计算/选型示例

- 已知：  
  调试人员修改了PLC程序后，设备出现异常，但不清楚改动位置。
- 公式：  
  **故障追溯 = 最近修改记录 + 修改对象 + 修改原因**
- 计算：  
  若有版本记录，可查看最近一次提交内容，快速定位可疑改动。
- 结论：  
  版本控制接口能帮助工程师快速回溯问题，减少现场排查时间。

> [!tip] 记忆口诀
> 有记录好追踪，没记录靠回忆。

# 03 适用场景

> [!info] 核心结论
> 项目越复杂、参与人员越多、交付周期越长，越需要版本控制接口。

## ① 分类/公式/规则

> [!note] 🟦 典型适用项目
> 版本控制特别适合大型产线、长期维护项目、多工程师协作项目、客户定制项目和多版本交付项目。

> [!warning] 🟧 是否需要版本控制
> **版本控制需求 = 项目规模 + 修改频率 + 参与人数 + 交付版本数量**

## ② 参考表

| 场景 | 是否建议使用 |
|---|---|
| **单人小项目** | 建议做基础备份 |
| **多人协作项目** | 强烈建议 |
| **长期维护项目** | 强烈建议 |
| **客户定制项目** | 强烈建议 |
| **频繁调试修改** | 强烈建议 |
| **一次性临时测试** | 可简化管理 |

## ③ 计算/选型示例

- 已知：  
  一个项目需要交付国内版、海外版、测试版、现场正式版。
- 公式：  
  **版本复杂度 = 项目分支数量 + 修改频率 + 交付对象**
- 计算：  
  多个版本并存，如果只靠文件名区分，容易混淆。
- 结论：  
  应使用版本控制或严格版本命名规则，区分测试版、正式版和客户交付版。

> [!tip] 记忆口诀
> 项目越多人越多，版本越要管清楚。

# 04 版本命名规则

> [!info] 核心结论
> 好的版本命名能让工程师一眼看出项目阶段、修改日期和版本状态。

## ① 分类/公式/规则

> [!note] 🟦 命名建议
> 版本名称应包含：  
> **项目名 + 日期 + 版本号 + 状态说明**

> [!warning] 🟧 推荐命名公式
> **版本名 = 项目名称_日期_版本号_状态**

## ② 参考表

| 字段 | 示例 |
|---|---|
| **项目名称** | Line01、PackingMachine |
| **日期** | 2026-05-02 |
| **版本号** | V1.0、V1.1、V2.0 |
| **状态** | Test、Debug、Release |
| **说明** | HMI修改、PLC优化、现场交付 |

## ③ 计算/选型示例

- 已知：  
  包装机项目在2026年5月2日完成现场正式交付。
- 公式：  
  **版本名 = 项目名 + 日期 + 版本号 + 状态**
- 计算：  
  PackingMachine_2026-05-02_V1.0_Release
- 结论：  
  该命名能清楚表达项目、时间、版本和状态，便于后期查找。

> [!tip] 记忆口诀
> 名字写清楚，版本不迷路。

# 05 提交流程

> [!info] 核心结论
> 每次提交版本前，应先编译、测试、备注修改内容，再保存版本。

## ① 分类/公式/规则

> [!note] 🟦 推荐流程
> ① 修改项目  
> ② 编译检查  
> ③ 在线测试  
> ④ 填写修改说明  
> ⑤ 提交或备份版本

> [!warning] 🟧 提交前检查公式
> **可提交版本 = 编译无错误 + 功能已验证 + 修改说明清楚 + 版本号正确**

## ② 参考表

| 步骤 | 检查内容 |
|---|---|
| **修改前** | 先备份当前稳定版 |
| **修改中** | 记录修改点 |
| **编译后** | 确认无错误 |
| **测试后** | 确认功能正常 |
| **提交时** | 写清楚修改说明 |

## ③ 计算/选型示例

- 已知：  
  工程师修改了HMI报警画面和PLC报警逻辑。
- 公式：  
  **提交说明 = 修改对象 + 修改原因 + 测试结果**
- 计算：  
  提交说明可写：优化报警逻辑，新增电机过载报警显示，已完成编译和HMI测试。
- 结论：  
  不建议只写“修改完成”或“最新版”，应写清楚具体修改内容。

> [!tip] 记忆口诀
> 提交不写说明，后期全靠猜。

# 06 版本回退

> [!info] 核心结论
> 版本回退用于在新版本异常时恢复到旧的稳定状态，是现场调试的重要保险。

## ① 分类/公式/规则

> [!note] 🟦 回退场景
> 当新程序导致设备异常、通讯异常、画面异常或客户不接受修改时，需要考虑回退。

> [!warning] 🟧 回退判断公式
> **是否回退 = 新版本异常 + 原因不明 + 影响生产 + 有稳定旧版本**

## ② 参考表

| 情况 | 建议 |
|---|---|
| **新版本编译失败** | 不提交正式版 |
| **下载后设备异常** | 评估回退 |
| **客户要求恢复** | 回退到交付版 |
| **误删程序块** | 从旧版本恢复 |
| **通讯配置混乱** | 回退硬件组态 |

## ③ 计算/选型示例

- 已知：  
  修改程序后，设备运行节拍异常，短时间无法定位原因。
- 公式：  
  **现场恢复优先级 = 生产恢复 > 问题分析**
- 计算：  
  如果旧版本稳定，应先恢复生产，再离线分析新版本问题。
- 结论：  
  可回退到最近稳定版本，避免长时间停机；回退前应保存当前异常版本用于分析。

> [!tip] 记忆口诀
> 新版出问题，旧版来救场。

# 07 常见风险

> [!info] 核心结论
> 版本管理混乱的根本问题，通常是文件名不规范、多人覆盖、无备份、无修改说明。

## ① 分类/公式/规则

> [!note] 🟦 常见风险
> 最常见的问题是：  
> “最终版”“最终最终版”“现场最新版”“客户最新版”同时存在，没人知道哪个才是真的正式版。

> [!warning] 🟧 版本混乱公式
> **版本混乱 = 命名随意 + 无提交说明 + 多人覆盖 + 没有交付基线**

## ② 参考表

| 问题 | 后果 |
|---|---|
| **文件名随意** | 找不到正确版本 |
| **无修改说明** | 不知道改过什么 |
| **多人覆盖** | 丢失他人修改 |
| **无稳定基线** | 无法安全回退 |
| **只存本地** | 电脑损坏导致丢失 |

## ③ 计算/选型示例

- 已知：  
  文件夹中有“最新版”“最新版2”“最终版”“客户版”等多个项目。
- 公式：  
  **查找风险 = 文件数量 × 命名混乱程度**
- 计算：  
  文件越多、名称越乱，越容易下载错版本。
- 结论：  
  应统一版本命名，保留正式交付版，并清理或归档临时版本。

> [!tip] 记忆口诀
> 最怕不是没版本，而是版本太多还看不懂。

# 08 快速检查表

> [!info] 核心结论
> 使用版本控制接口前，先确认版本命名、提交说明、备份位置、权限和回退方案。

## ① 分类/公式/规则

> [!note] 🟦 五步检查法
> ① 当前版本是否稳定  
> ② 版本名是否规范  
> ③ 修改说明是否清楚  
> ④ 是否完成编译测试  
> ⑤ 是否可回退

> [!warning] 🟧 最终检查公式
> **可靠版本 = 名称规范 + 说明清楚 + 编译通过 + 测试完成 + 可恢复**

## ② 参考表

| 检查项 | 建议 |
|---|---|
| **版本名称** | 包含日期、版本号、状态 |
| **修改说明** | 写清楚修改对象和原因 |
| **编译状态** | 无错误再提交 |
| **测试状态** | 测试通过再标正式版 |
| **备份位置** | 不只保存在本机 |
| **回退方案** | 保留最近稳定版 |

## ③ 计算/选型示例

- 已知：  
  项目准备交付客户，需要保存正式版本。
- 公式：  
  **交付版本 = 编译通过 + 功能验证 + 客户确认 + 独立备份**
- 计算：  
  交付前应保存正式Release版本，并记录交付内容。
- 结论：  
  正式交付版应单独标记，避免后续调试版本覆盖交付基线。

> [!tip] 选型口诀
> 版本要有名，提交要有因；交付留基线，出错能回退。
