# 01 安全设置总览

> [!info] 核心结论
> **安全设置** 用于管理项目权限、用户账号、角色分配和访问保护，核心目的是防止误操作和未授权修改。

## ① 分类/公式/规则

> [!note] 🟦 分类主线
> 安全设置主要分为两部分：  
> **设置**：配置项目安全策略；  
> **用户与角色**：管理谁能访问、谁能修改、谁能下载。

> [!warning] 🟧 权限管理公式
> **安全权限 = 用户账号 + 角色权限 + 访问范围 + 密码策略 + 操作记录**

## ② 参考表

| 项目 | 作用 |
|---|---|
| **安全设置** | 管理项目安全入口 |
| **设置** | 配置安全策略和保护规则 |
| **用户与角色** | 管理用户、角色、权限 |
| **权限分配** | 限制查看、修改、下载等操作 |
| **访问保护** | 防止未授权人员操作项目 |

## ③ 计算/选型示例

- 已知：  
  一个自动化项目由电气工程师、调试人员、操作员共同维护。
- 公式：  
  **项目安全 = 用户分类 + 权限分级 + 密码保护 + 操作边界**
- 计算：  
  电气工程师可修改程序，调试人员可下载和监控，操作员只允许查看或运行。
- 结论：  
  应在 **安全设置** 中建立用户和角色，并按职责分配权限。

> [!tip] 记忆口诀
> 谁能看、谁能改、谁能下，安全设置说了算。

# 02 设置

> [!info] 核心结论
> **设置** 用于配置项目级安全策略，是安全管理的基础入口。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> 设置主要用于定义项目是否启用安全功能、密码策略、访问保护、权限规则等内容。

> [!warning] 🟧 安全策略公式
> **安全策略 = 启用保护 + 密码规则 + 权限限制 + 项目访问控制**

## ② 参考表

| 设置项 | 重点关注 |
|---|---|
| **项目保护** | 是否限制项目访问 |
| **密码策略** | 密码复杂度、有效期 |
| **访问权限** | 查看、编辑、下载限制 |
| **安全等级** | 按项目风险设置 |
| **兼容性** | 与TIA版本、设备版本匹配 |

## ③ 计算/选型示例

- 已知：  
  项目需要防止非授权人员修改PLC程序。
- 公式：  
  **防误改能力 = 项目保护 + 编辑权限 + 下载权限**
- 计算：  
  只设置密码不够，还需要限制哪些用户可以编辑和下载。
- 结论：  
  应在 **设置** 中启用项目安全策略，并结合用户角色进行权限分配；具体功能需按TIA版本和项目规范校核。

> [!tip] 记忆口诀
> 先开安全策略，再谈用户权限。

# 03 用户与角色

> [!info] 核心结论
> **用户与角色** 用来管理不同人员的访问权限，让不同岗位只拥有自己需要的操作权限。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> 用户代表具体人员，角色代表权限集合；  
> 一个用户可以分配一个或多个角色。

> [!warning] 🟧 角色权限公式
> **用户权限 = 用户账号 × 角色权限 × 项目访问范围**

## ② 参考表

| 角色类型 | 建议权限 |
|---|---|
| **管理员** | 用户管理、权限管理、项目配置 |
| **工程师** | 程序编辑、硬件组态、下载调试 |
| **调试员** | 在线监控、下载、故障诊断 |
| **维护员** | 查看、诊断、有限修改 |
| **操作员** | 查看状态、基本操作 |
| **访客** | 只读或无权限 |

## ③ 计算/选型示例

- 已知：  
  现场有工程师、维护人员、操作员三类人员。
- 公式：  
  **权限分配 = 岗位职责 + 操作风险 + 最小权限原则**
- 计算：  
  工程师需要编辑和下载；维护人员需要诊断和有限修改；操作员只需要查看和基本操作。
- 结论：  
  应分别建立对应角色，并把用户加入对应角色，避免所有人都使用管理员权限。

> [!tip] 记忆口诀
> 用户是“人”，角色是“权限包”；人进角色，权限就到。

# 04 权限分级原则

> [!info] 核心结论
> 权限分配要遵循 **最小权限原则**，只给完成工作所需的权限，不多给。

## ① 分类/公式/规则

> [!note] 🟦 最小权限原则
> 能查看就不给修改；  
> 能调试就不给管理；  
> 能维护就不给管理员权限。

> [!warning] 🟧 权限风险公式
> **权限风险 = 权限过大 + 账号共用 + 密码泄露 + 无操作边界**

## ② 参考表

| 操作 | 风险等级 |
|---|---|
| **查看项目** | 低 |
| **在线监控** | 中 |
| **修改程序** | 高 |
| **下载到PLC** | 高 |
| **修改安全设置** | 很高 |
| **管理用户角色** | 很高 |

## ③ 计算/选型示例

- 已知：  
  操作员只需要查看设备状态，不需要修改程序。
- 公式：  
  **合理权限 = 工作需要 - 多余权限**
- 计算：  
  操作员不需要编辑、下载和安全管理权限。
- 结论：  
  操作员角色应限制为查看和基础操作，不应赋予工程师或管理员权限。

> [!tip] 记忆口诀
> 权限宁可少一点，也别随手给管理员。

# 05 密码与账号管理

> [!info] 核心结论
> 密码和账号管理是项目安全的第一道门，重点是避免共用账号和弱密码。

## ① 分类/公式/规则

> [!note] 🟦 账号管理原则
> 一人一账号，岗位配角色；  
> 离岗删权限，密码定期换。

> [!warning] 🟧 账号安全公式
> **账号安全 = 独立账号 + 强密码 + 定期更新 + 权限回收**

## ② 参考表

| 项目 | 建议 |
|---|---|
| **账号使用** | 一人一号 |
| **密码强度** | 避免简单密码 |
| **账号共享** | 不建议 |
| **离职人员** | 及时禁用或删除 |
| **管理员账号** | 严格控制 |
| **密码记录** | 按企业规范保存 |

## ③ 计算/选型示例

- 已知：  
  现场多人共用一个管理员账号。
- 公式：  
  **安全隐患 = 共用账号 + 高权限 + 无责任追踪**
- 计算：  
  如果出现误修改，难以判断是谁操作的。
- 结论：  
  应改为独立账号，并按岗位分配角色，管理员账号只给少数负责人。

> [!tip] 记忆口诀
> 账号不共用，权限不乱给，密码不简单。

# 06 常见风险与排查

> [!info] 核心结论
> 安全设置出问题，常见原因是权限不足、账号错误、密码遗忘或角色分配不合理。

## ① 分类/公式/规则

> [!note] 🟦 排查顺序
> 先确认登录用户，再确认角色权限，最后检查项目安全策略。

> [!warning] 🟧 排错公式
> **权限异常 = 用户错误 + 角色错误 + 权限不足 + 安全策略限制**

## ② 参考表

| 问题 | 可能原因 |
|---|---|
| **无法修改项目** | 没有编辑权限 |
| **无法下载PLC** | 没有下载权限 |
| **无法管理用户** | 不是管理员角色 |
| **无法访问项目** | 账号或密码错误 |
| **权限混乱** | 多人共用账号 |
| **交付后无法维护** | 权限资料未交接 |

## ③ 计算/选型示例

- 已知：  
  调试人员可以打开项目，但无法下载PLC。
- 公式：  
  **下载权限检查 = 登录账号 + 所属角色 + 下载权限**
- 计算：  
  账号能打开项目，不代表拥有下载权限。
- 结论：  
  应检查该用户所属角色是否包含下载权限，必要时由管理员调整权限。

> [!tip] 记忆口诀
> 能打开不等于能下载，能查看不等于能修改。

# 07 安全设置快速检查表

> [!info] 核心结论
> 项目交付前，应确认用户、角色、密码、权限和交接资料完整，避免后期无法维护。

## ① 分类/公式/规则

> [!note] 🟦 交付前五查
> ① 管理员账号是否明确  
> ② 用户角色是否合理  
> ③ 下载权限是否受控  
> ④ 密码资料是否交接  
> ⑤ 权限是否符合项目规范

> [!warning] 🟧 交付检查公式
> **安全交付 = 账号清晰 + 权限清晰 + 密码交接 + 责任明确**

## ② 参考表

| 检查项 | 建议 |
|---|---|
| **管理员账号** | 明确负责人 |
| **工程师账号** | 保留必要编辑权限 |
| **操作员账号** | 限制高风险权限 |
| **临时账号** | 交付前删除或禁用 |
| **密码资料** | 按规范交接 |
| **权限记录** | 建议形成文档 |

## ③ 计算/选型示例

- 已知：  
  项目准备交付客户，但现场还有临时调试账号。
- 公式：  
  **交付风险 = 临时账号 + 高权限 + 无交接记录**
- 计算：  
  临时账号若保留，后期可能造成未授权访问或责任不清。
- 结论：  
  交付前应删除临时账号，确认正式账号和角色权限，并完成密码交接。

> [!tip] 选型口诀
> 安全设置要闭环：账号有人管，权限有边界，密码有交接。
