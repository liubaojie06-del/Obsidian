# 01 跨设备功能总览

> [!info] 核心结论
> **跨设备功能** 用于对多个设备或整个项目进行统一测试、跟踪、测量和长期数据记录。

## ① 分类/公式/规则

> [!note] 🟦 功能主线
> 跨设备功能不是针对单个PLC或HMI，而是面向 **整个项目、多设备联动、调试验证和长期分析**。

> [!warning] 🟧 使用判断公式
> **跨设备功能需求 = 多设备对象 + 联动测试 + 数据跟踪 + 长期诊断**

## ② 参考表

| 功能 | 主要作用 |
|---|---|
| **Safety Activation Test** | 安全激活测试 |
| **添加新的 Safety Activation Test** | 新建安全激活测试 |
| **完成** | 结束或确认测试流程 |
| **项目跟踪** | 记录项目运行或调试数据 |
| **项目跟踪 - 测量** | 查看或执行测量 |
| **长期项目轨迹** | 长时间记录项目数据 |
| **新增长期项目轨迹** | 新建长期跟踪任务 |
| **长期项目轨迹 - 测量** | 查看长期测量数据 |

## ③ 计算/选型示例

- 已知：  
  项目包含PLC、HMI、驱动和安全相关功能，需要联动调试。
- 公式：  
  **调试验证 = 安全测试 + 项目跟踪 + 测量记录 + 长期分析**
- 计算：  
  如果只看单个设备，难以判断整体联动是否正常；跨设备功能可帮助统一观察。
- 结论：  
  可通过 **跨设备功能** 进行安全测试、项目跟踪和长期数据分析，具体功能需按TIA版本和设备支持情况校核。

> [!tip] 记忆口诀
> 单设备看局部，跨设备看全局；调试要联动，跟踪要记录。

# 02 Safety Activation Test

> [!info] 核心结论
> **Safety Activation Test** 用于验证安全相关功能是否被正确激活和确认。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> Safety Activation Test 主要用于安全功能相关测试，帮助确认安全功能是否处于预期状态。

> [!warning] 🟧 安全测试判断公式
> **安全测试有效 = 安全功能存在 + 测试步骤明确 + 激活状态正确 + 结果可确认**

## ② 参考表

| 项目 | 关注点 |
|---|---|
| **测试对象** | 安全相关设备或安全功能 |
| **测试目的** | 验证安全功能激活状态 |
| **测试结果** | 通过、未通过、待确认 |
| **适用阶段** | 调试、验收、维护 |
| **注意事项** | 需按安全规范和厂家手册校核 |

## ③ 计算/选型示例

- 已知：  
  项目中存在安全功能，需要确认其是否正确激活。
- 公式：  
  **安全确认 = 功能存在 + 激活测试 + 结果确认**
- 计算：  
  先创建测试，再按流程执行测试，最后确认测试结果。
- 结论：  
  应使用 **Safety Activation Test** 进行验证，测试过程必须符合项目安全规范。

> [!tip] 记忆口诀
> 安全功能不能猜，激活测试要做完。

# 03 添加新的 Safety Activation Test

> [!info] 核心结论
> **添加新的 Safety Activation Test** 用于创建一条新的安全激活测试任务。

## ① 分类/公式/规则

> [!note] 🟦 创建逻辑
> 新建测试前，要先明确测试对象、测试目的、测试步骤和验收标准。

> [!warning] 🟧 新建测试公式
> **新建安全测试 = 选择对象 + 定义测试项 + 执行测试 + 保存结果**

## ② 参考表

| 步骤 | 内容 |
|---|---|
| **1 选择对象** | 选择安全相关设备或功能 |
| **2 定义测试** | 明确测试内容 |
| **3 执行测试** | 按流程操作 |
| **4 检查结果** | 判断是否通过 |
| **5 保存记录** | 便于追溯和验收 |

## ③ 计算/选型示例

- 已知：  
  新增了一个安全相关功能，需要进行首次测试。
- 公式：  
  **首次测试流程 = 新建测试 + 执行测试 + 结果确认**
- 计算：  
  新功能没有历史测试记录，因此需要新建 Safety Activation Test。
- 结论：  
  点击 **添加新的 Safety Activation Test**，建立新的测试任务，并按规范完成验证。

> [!tip] 记忆口诀
> 新增安全功能，先建测试再验证。

# 04 完成

> [!info] 核心结论
> **完成** 表示安全激活测试流程已经结束或结果已经确认。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> 完成不是简单关闭窗口，而是对测试状态和结果进行确认。

> [!warning] 🟧 完成判断公式
> **测试完成 = 测试已执行 + 结果已确认 + 问题已处理 + 记录已保存**

## ② 参考表

| 检查项 | 是否重要 |
|---|---|
| **测试是否执行** | 必查 |
| **结果是否通过** | 必查 |
| **异常是否处理** | 必查 |
| **记录是否保存** | 建议保存 |
| **是否可追溯** | 验收关键 |

## ③ 计算/选型示例

- 已知：  
  Safety Activation Test 已执行，但出现一个异常提示。
- 公式：  
  **能否完成 = 测试结果通过 + 无未处理异常**
- 计算：  
  如果异常未处理，不建议直接标记完成。
- 结论：  
  应先处理异常并确认测试结果，再执行 **完成** 操作。

> [!tip] 记忆口诀
> 测试不闭环，不能随手点完成。

# 05 项目跟踪

> [!info] 核心结论
> **项目跟踪** 用于在项目运行或调试过程中记录关键变量、状态和事件变化。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> 项目跟踪适合分析设备动作顺序、变量变化、通讯状态和调试过程。

> [!warning] 🟧 跟踪判断公式
> **项目跟踪价值 = 关键变量 + 时间变化 + 故障现象 + 调试需求**

## ② 参考表

| 项目 | 关注点 |
|---|---|
| **跟踪对象** | PLC变量、状态量、过程数据 |
| **使用场景** | 调试、故障分析、动作验证 |
| **重点参数** | 采样周期、触发条件、记录长度 |
| **结果用途** | 分析趋势、定位问题 |
| **注意事项** | 采样能力需按设备和软件版本校核 |

## ③ 计算/选型示例

- 已知：  
  某个气缸偶发不到位，现场很难直接观察原因。
- 公式：  
  **故障分析 = 输入信号 + 输出信号 + 中间条件 + 时间顺序**
- 计算：  
  跟踪气缸到位信号、电磁阀输出、启动条件和报警位。
- 结论：  
  可使用 **项目跟踪** 记录变量变化，判断是传感器、程序逻辑还是执行机构问题。

> [!tip] 记忆口诀
> 偶发故障别硬猜，项目跟踪查变化。

# 06 项目跟踪 - 测量

> [!info] 核心结论
> **测量** 是项目跟踪中的数据采集结果，用于查看变量在时间轴上的变化过程。

## ① 分类/公式/规则

> [!note] 🟦 测量内容
> 测量通常关注变量值、状态变化、触发时刻和持续时间。

> [!warning] 🟧 测量分析公式
> **测量结论 = 时间轴 + 变量变化 + 触发条件 + 异常点**

## ② 参考表

| 测量项目 | 作用 |
|---|---|
| **变量曲线** | 查看数值变化 |
| **开关量状态** | 分析ON/OFF时序 |
| **触发点** | 定位故障发生时刻 |
| **持续时间** | 判断动作是否超时 |
| **异常点** | 找到不符合预期的位置 |

## ③ 计算/选型示例

- 已知：  
  电机启动后2秒内必须收到运行反馈。
- 公式：  
  **反馈延迟 = 运行反馈时间 - 启动输出时间**
- 计算：  
  若启动输出时间为 10.000s，反馈到达时间为 12.500s，反馈延迟 = **2.5s**。
- 结论：  
  若工艺要求小于2秒，则反馈超时，需要检查电机、变频器、通讯或程序逻辑。

> [!tip] 记忆口诀
> 测量看时间，故障看差值。

# 07 长期项目轨迹

> [!info] 核心结论
> **长期项目轨迹** 用于长时间记录项目数据，适合分析偶发故障、慢变化趋势和长期运行状态。

## ① 分类/公式/规则

> [!note] 🟦 核心理解
> 普通项目跟踪适合短时间调试，长期项目轨迹适合长周期观察。

> [!warning] 🟧 长期跟踪判断公式
> **长期跟踪需求 = 偶发故障 + 长周期运行 + 趋势变化 + 需要历史记录**

## ② 参考表

| 项目 | 关注点 |
|---|---|
| **记录周期** | 长时间运行数据 |
| **适用问题** | 偶发报警、间歇故障 |
| **分析对象** | 温度、压力、速度、状态位 |
| **重点设置** | 采样周期、存储容量、触发条件 |
| **注意事项** | 数据量和存储位置需校核 |

## ③ 计算/选型示例

- 已知：  
  设备每天随机报警1次，现场无法复现。
- 公式：  
  **偶发故障分析 = 长期记录 + 报警触发 + 前后数据对比**
- 计算：  
  长时间记录报警位、关键传感器、执行器状态和工艺参数。
- 结论：  
  应使用 **长期项目轨迹** 记录数据，便于在故障发生后回看前后状态。

> [!tip] 记忆口诀
> 短故障用跟踪，偶发故障用长期轨迹。

# 08 新增长期项目轨迹

> [!info] 核心结论
> **新增长期项目轨迹** 用于创建新的长周期数据记录任务。

## ① 分类/公式/规则

> [!note] 🟦 创建逻辑
> 新增长期轨迹时，要先确定记录变量、采样周期、触发条件和保存方式。

> [!warning] 🟧 新增长期轨迹公式
> **长期轨迹任务 = 记录变量 + 采样周期 + 触发条件 + 存储策略**

## ② 参考表

| 设置项 | 说明 |
|---|---|
| **记录变量** | 要观察的关键数据 |
| **采样周期** | 多久记录一次 |
| **触发条件** | 何时开始或保存 |
| **记录时长** | 记录多久 |
| **存储方式** | 本地或项目指定位置 |

## ③ 计算/选型示例

- 已知：  
  需要连续观察设备温度和报警状态一周。
- 公式：  
  **数据量 ≈ 变量数量 × 采样次数 × 单次数据大小**
- 计算：  
  变量越多、采样越快、记录越久，数据量越大。
- 结论：  
  新增长期项目轨迹时，应合理设置采样周期和变量数量，避免数据量过大。

> [!tip] 记忆口诀
> 长期记录别贪多，变量少一点，周期稳一点。

# 09 长期项目轨迹 - 测量

> [!info] 核心结论
> 长期项目轨迹中的 **测量** 用于查看长时间记录的数据结果，重点分析趋势和异常点。

## ① 分类/公式/规则

> [!note] 🟦 分析重点
> 长期测量不是只看某一瞬间，而是看一段时间内的变化规律。

> [!warning] 🟧 趋势分析公式
> **长期结论 = 趋势变化 + 异常时刻 + 报警关系 + 工艺状态**

## ② 参考表

| 分析对象 | 重点看什么 |
|---|---|
| **温度** | 是否逐渐升高 |
| **压力** | 是否波动异常 |
| **速度** | 是否周期性变化 |
| **报警位** | 是否与某变量相关 |
| **通讯状态** | 是否存在间歇掉线 |

## ③ 计算/选型示例

- 已知：  
  设备每天下午偶发报警，怀疑与温度升高有关。
- 公式：  
  **关联判断 = 报警时间点 + 温度曲线 + 设备负载**
- 计算：  
  若报警前温度持续升高，并在超过某范围后触发报警，则可能与温度有关。
- 结论：  
  可通过长期项目轨迹的测量结果分析报警与温度、负载或运行时间的关系。

> [!tip] 记忆口诀
> 长期测量看趋势，异常前后找关系。

# 10 跨设备功能快速检查表

> [!info] 核心结论
> 使用跨设备功能前，要先确认对象、变量、触发条件、记录周期和权限是否正确。

## ① 分类/公式/规则

> [!note] 🟦 使用前五查
> ① 测试对象是否正确  
> ② 变量是否选对  
> ③ 触发条件是否合理  
> ④ 采样周期是否合适  
> ⑤ 数据是否能保存和追溯

> [!warning] 🟧 最终检查公式
> **有效诊断 = 正确对象 + 正确变量 + 合理采样 + 可用记录 + 清晰结论**

## ② 参考表

| 检查项 | 建议 |
|---|---|
| **安全测试对象** | 与安全功能一致 |
| **跟踪变量** | 选关键变量，不要过多 |
| **采样周期** | 兼顾精度和数据量 |
| **触发条件** | 尽量贴近故障发生条件 |
| **记录时长** | 与故障周期匹配 |
| **结果保存** | 便于复盘和交付 |

## ③ 计算/选型示例

- 已知：  
  一个故障每隔几小时才出现一次。
- 公式：  
  **跟踪方式选择 = 故障频率 + 数据量 + 分析目标**
- 计算：  
  短时间项目跟踪可能抓不到故障，长期项目轨迹更合适。
- 结论：  
  对于偶发、长周期问题，优先使用 **长期项目轨迹**；对于现场立即复现的问题，可使用 **项目跟踪**。

> [!tip] 选型口诀
> 立即复现用跟踪，偶发问题用长期；安全相关先测试，完成之前要确认。
