# 01 边缘计算网关总览

> [!info] 核心结论
> 边缘计算网关用于在设备现场完成 **数据采集、协议转换、本地计算、报警判断、数据缓存、模型推理、数据上云和远程运维**。

## ① 分类/公式/规则

> [!warning] 选型公式
> **边缘计算网关选型 = 采集协议 + 点数规模 + 计算能力 + 存储缓存 + 上云协议 + 安全能力 + 工业环境等级**

## ② 参考表

| 项目 | 说明 |
|---|---|
| **数据采集** | 读取PLC、仪表、传感器、CNC、机器人数据 |
| **协议转换** | Modbus、S7、OPC UA、EtherNet/IP、MQTT等 |
| **边缘计算** | 本地计算、过滤、统计、报警、规则引擎 |
| **数据缓存** | 断网保存数据，恢复后补传 |
| **模型推理** | 部分网关支持AI模型、异常检测 |
| **数据上云** | MQTT、HTTP、OPC UA、数据库等 |
| **远程运维** | 远程配置、升级、监控、日志查看 |

## ③ 计算/选型示例

- 已知：  
  设备每秒产生大量温度、压力、电流和振动数据，但平台只需要报警和统计结果。
- 公式：  
  **现场原始数据 → 边缘计算 → 关键结果上传**
- 计算：  
  网关在本地计算平均值、最大值、报警状态，只上传结果。
- 结论：  
  推荐使用 **边缘计算网关**，减少网络和平台压力。

> [!tip] 记忆口诀
> 普通网关负责传数据，边缘网关负责先算一遍再上传。

# 02 底层原理

> [!info] 核心结论
> 边缘计算网关的底层原理，是把现场设备数据采集到本地，在靠近设备的位置完成计算、判断、缓存和转发。

## ① 分类/公式/规则

> [!note] 工作链路
> 边缘计算网关向下连接设备，向上连接平台。  
> 它不是简单“搬运数据”，而是在现场先进行 **清洗、换算、压缩、报警判断和边缘分析**。

> [!warning] 数据链路
> **现场设备 → 数据采集 → 协议解析 → 边缘计算 → 缓存/判断 → 云平台/MES/SCADA**

## ② 参考表

| 环节 | 作用 |
|---|---|
| **设备接入** | PLC、仪表、传感器、机器人、相机 |
| **协议解析** | 把寄存器、变量、报文解析成数据点 |
| **数据清洗** | 去除异常值、无效值、重复值 |
| **工程换算** | 原始值转温度、压力、流量等工程值 |
| **规则计算** | 超限、趋势、状态、能耗、稼动率 |
| **缓存补传** | 断网时本地保存，联网后补传 |
| **安全上传** | 加密传输到平台或服务器 |

## ③ 计算/选型示例

- 已知：  
  PLC寄存器读取到压力原始值6000，对应量程0~10000，实际压力0~1.0MPa。
- 公式：  
  **压力 = 6000 ÷ 10000 × 1.0**
- 计算：  
  压力 = **0.6MPa**
- 结论：  
  边缘网关可以在本地完成工程量换算，再上传真实压力值。

> [!tip] 记忆口诀
> 边缘计算就是把一部分“平台要做的事”提前放到现场做。

# 03 边缘计算网关 vs 普通工业网关

> [!info] 核心结论
> 普通工业网关偏采集和协议转换，边缘计算网关更强调本地计算、规则判断、缓存、算法和自治能力。

## ① 分类/公式/规则

> [!warning] 选择规则
> **只采集上传 → 普通工业网关可用**  
> **需要本地计算、报警、缓存、AI、规则引擎 → 边缘计算网关**

## ② 参考表

| 对比项 | 普通工业网关 | 边缘计算网关 |
|---|---|---|
| 协议采集 | 有 | 有 |
| 数据上传 | 有 | 有 |
| 本地计算 | 较弱 | **较强** |
| 规则引擎 | 视型号 | **常见** |
| 数据缓存 | 基础 | **更强** |
| AI推理 | 少见 | **部分支持** |
| 本地自治 | 较弱 | **更强** |

## ③ 计算/选型示例

- 已知：  
  项目只需要每10秒读取PLC的100个变量并上传MQTT。
- 公式：  
  **简单采集上传 → 普通工业网关即可**
- 计算：  
  不需要复杂规则、AI推理和本地分析。
- 结论：  
  普通工业网关即可；若后期要做预测维护和异常检测，再考虑边缘计算网关。

> [!tip] 记忆口诀
> 网关解决“能不能传”，边缘网关解决“传之前要不要先算”。

# 04 边缘计算网关 vs 工控机

> [!info] 核心结论
> 工控机计算能力更强、软件自由度更高；边缘计算网关更嵌入式、更稳定、更适合固定采集和现场边缘处理。

## ① 分类/公式/规则

> [!warning] 选择规则
> **复杂视觉、数据库、大型软件 → 工控机**  
> **稳定采集、协议转换、规则计算、轻量AI → 边缘计算网关**

## ② 参考表

| 对比项 | 工控机 | 边缘计算网关 |
|---|---|---|
| 算力 | 强 | 中等到较强 |
| 软件自由度 | 高 | 受平台限制 |
| 维护复杂度 | 较高 | **较低** |
| 系统稳定性 | 取决于系统维护 | **嵌入式更稳定** |
| 工业协议 | 需软件支持 | **通常内置较多** |
| 部署方式 | 上位机软件部署 | Web/平台配置 |
| 适合场景 | 视觉、数据库、复杂应用 | 采集、上云、边缘判断 |

## ③ 计算/选型示例

- 已知：  
  现场需要运行复杂机器视觉算法，并保存大量图片。
- 公式：  
  **重算法 + 大图片存储 → 工控机优先**
- 计算：  
  边缘网关通常不适合大规模图像处理。
- 结论：  
  可采用 **工控机做视觉计算，边缘网关做设备数据采集与上传**。

> [!tip] 记忆口诀
> 算得重用工控机，采得稳用边缘网关。

# 05 系统架构

> [!info] 核心结论
> 边缘计算网关通常位于设备层和平台层之间，向下采集设备数据，向上对接MES、SCADA、云平台或数据库。

## ① 分类/公式/规则

> [!warning] 标准架构
> **设备层 → 边缘计算网关 → 平台层 → 应用层**

## ② 参考表

| 层级 | 典型对象 |
|---|---|
| **设备层** | PLC、仪表、机器人、CNC、传感器 |
| **网络层** | 工业交换机、路由器、4G/5G、Wi-Fi |
| **边缘层** | 边缘计算网关、边缘服务器 |
| **平台层** | MES、SCADA、IoT平台、数据库 |
| **应用层** | 报表、看板、报警、预测维护、能耗分析 |

## ③ 计算/选型示例

- 已知：  
  设备要采集PLC数据，上传云平台，同时现场断网不能丢数据。
- 公式：  
  **采集 + 缓存 + 上传 → 边缘计算网关**
- 计算：  
  网关本地保存断网期间数据，网络恢复后补传。
- 结论：  
  边缘网关适合作为设备和平台之间的可靠数据节点。

> [!tip] 记忆口诀
> 边缘网关在设备和平台中间，下面读设备，上面连系统。

# 06 下行采集协议

> [!info] 核心结论
> 下行协议决定边缘计算网关能不能读到现场设备数据，必须与PLC、仪表、机器人或控制器匹配。

## ① 分类/公式/规则

> [!warning] 采集成功公式
> **采集成功 = 协议支持 + 地址正确 + 数据类型正确 + 通讯参数正确**

## ② 参考表

| 下行协议 | 常见对象 |
|---|---|
| **Modbus RTU** | RS485仪表、电表、温控器 |
| **Modbus TCP** | PLC、控制器、仪表 |
| **Siemens S7** | 西门子PLC |
| **EtherNet/IP** | 罗克韦尔PLC、部分设备 |
| **OPC UA Client** | 读取OPC UA服务器数据 |
| **CAN/CANopen** | 部分驱动器和设备 |
| **自定义协议** | 老设备、专用设备 |

## ③ 计算/选型示例

- 已知：  
  现场有西门子PLC和12台Modbus RTU电表。
- 公式：  
  **网关协议 ≥ S7 + Modbus RTU**
- 计算：  
  网关需要同时支持西门子PLC采集和RS485仪表采集。
- 结论：  
  选型时必须确认多协议并发采集能力。

> [!tip] 记忆口诀
> 下面设备说什么协议，网关就要听得懂什么协议。

# 07 上行平台协议

> [!info] 核心结论
> 上行协议决定边缘计算网关如何把数据传给MES、SCADA、数据库、云平台或第三方系统。

## ① 分类/公式/规则

> [!warning] 上传成功公式
> **上传成功 = 协议一致 + 数据格式正确 + 认证通过 + 网络可达**

## ② 参考表

| 上行协议 | 常见用途 |
|---|---|
| **MQTT** | 工业物联网平台、云平台 |
| **HTTP/HTTPS** | Web API、MES接口 |
| **OPC UA Server** | SCADA、MES、上位机读取 |
| **数据库写入** | SQL Server、MySQL、PostgreSQL |
| **TCP/UDP Socket** | 自定义系统 |
| **Kafka/AMQP** | 大数据平台、消息队列 |
| **FTP/SFTP** | 文件、报表、CSV上传 |

## ③ 计算/选型示例

- 已知：  
  云平台要求MQTT上传，且必须使用TLS证书认证。
- 公式：  
  **MQTT + TLS → 网关需支持加密认证**
- 计算：  
  需要配置Broker地址、端口、Topic、证书、Client ID。
- 结论：  
  应选择支持MQTT加密认证的边缘计算网关。

> [!tip] 记忆口诀
> 下行看设备协议，上行看平台接口。

# 08 点表与变量管理

> [!info] 核心结论
> 点表是边缘计算网关的数据基础，定义采集哪些变量、变量名称、地址、类型、单位、倍率和采样周期。

## ① 分类/公式/规则

> [!warning] 点表公式
> **数据点 = 设备 + 地址 + 数据类型 + 单位 + 倍率 + 采样周期 + 上传策略**

## ② 参考表

| 字段 | 说明 |
|---|---|
| **设备名称** | PLC1、电表1、温控器1 |
| **变量名称** | 温度、压力、电流、产量、报警 |
| **地址** | PLC地址、Modbus寄存器、OPC UA节点 |
| **数据类型** | Bool、Int、DInt、Float、String |
| **单位** | ℃、MPa、A、kWh、pcs |
| **倍率** | 原始值换算比例 |
| **采样周期** | 1s、5s、10s、1min等 |
| **上传策略** | 周期上传、变化上传、报警上传 |

## ③ 计算/选型示例

- 已知：  
  电表寄存器读数为12345，倍率0.01，单位kWh。
- 公式：  
  **实际电量 = 原始值 × 倍率**
- 计算：  
  实际电量 = 12345 × 0.01 = **123.45kWh**
- 结论：  
  点表中必须配置倍率，否则平台显示值会错误。

> [!tip] 记忆口诀
> 点表错，数据就错；地址、类型、倍率都要核对。

# 09 数据清洗

> [!info] 核心结论
> 数据清洗用于去除无效值、异常值、重复值和噪声数据，提升上传数据质量。

## ① 分类/公式/规则

> [!warning] 数据清洗公式
> **有效数据 = 原始数据 - 无效数据 - 异常数据 - 重复数据**

## ② 参考表

| 清洗方式 | 作用 |
|---|---|
| **范围过滤** | 超出合理范围的数据丢弃或报警 |
| **空值过滤** | 去除无效空数据 |
| **重复过滤** | 相同数据不重复上传 |
| **变化阈值** | 变化超过阈值才上传 |
| **滑动平均** | 平滑波动数据 |
| **中值滤波** | 抑制尖峰干扰 |
| **异常标记** | 不删除，但标记异常状态 |

## ③ 计算/选型示例

- 已知：  
  压力正常范围0~1.0MPa，某次采集值为8.5MPa。
- 公式：  
  **超范围数据 = 异常值**
- 计算：  
  8.5MPa明显超出合理范围。
- 结论：  
  网关可将该数据标记为异常，或触发传感器故障报警。

> [!tip] 记忆口诀
> 原始数据不等于好数据，先清洗再上传更可靠。

# 10 边缘规则引擎

> [!info] 核心结论
> 规则引擎让边缘计算网关能在本地根据条件执行判断、报警、联动和数据处理。

## ① 分类/公式/规则

> [!warning] 规则公式
> **规则 = 条件判断 + 动作执行**

## ② 参考表

| 规则类型 | 示例 |
|---|---|
| **超限报警** | 温度 > 80℃触发报警 |
| **状态判断** | 电流 > 阈值判断设备运行 |
| **变化上报** | 压力变化超过0.05MPa上传 |
| **联动输出** | 超温时输出报警信号 |
| **数据汇总** | 每分钟计算平均值 |
| **班次统计** | 每班汇总产量和能耗 |
| **异常检测** | 连续波动异常触发维护提醒 |

## ③ 计算/选型示例

- 已知：  
  电机电流大于10A且持续30秒，判定为过载预警。
- 公式：  
  **过载预警 = 电流 > 10A 持续30s**
- 计算：  
  网关本地连续判断，不必等平台返回。
- 结论：  
  边缘规则可实现现场快速报警和减少上传延迟。

> [!tip] 记忆口诀
> 规则引擎就是让网关会“如果……那么……”。

# 11 本地报警判断

> [!info] 核心结论
> 边缘计算网关可在现场本地判断报警，避免全部依赖云端，提高响应速度和可靠性。

## ① 分类/公式/规则

> [!warning] 报警判断公式
> **报警 = 数据值超过阈值 + 持续时间满足 + 复位条件未满足**

## ② 参考表

| 报警类型 | 示例 |
|---|---|
| **上限报警** | 温度过高、压力过高 |
| **下限报警** | 气压不足、液位过低 |
| **变化率报警** | 温度上升过快 |
| **状态报警** | 设备停机、通信中断 |
| **组合报警** | 高温 + 高电流同时发生 |
| **延时报警** | 持续超过设定时间才报警 |
| **恢复报警** | 回到安全范围后复位 |

## ③ 计算/选型示例

- 已知：  
  温度报警阈值80℃，当前温度82℃，要求持续10秒才报警。
- 公式：  
  **报警条件 = 温度 > 80℃ 且持续时间 ≥ 10s**
- 计算：  
  若82℃只持续2秒，不触发正式报警。
- 结论：  
  延时报警可避免瞬时波动造成误报警。

> [!tip] 记忆口诀
> 报警不要只看瞬间值，要看阈值、时间和复位条件。

# 12 数据压缩与降频上传

> [!info] 核心结论
> 边缘计算网关可通过降频、压缩、汇总和事件触发上传减少网络流量和平台存储压力。

## ① 分类/公式/规则

> [!warning] 上传优化公式
> **上传数据量 = 点数 × 单点大小 × 上传频率**

## ② 参考表

| 方法 | 作用 |
|---|---|
| **周期上传** | 按固定周期发送 |
| **变化上传** | 数据变化超过阈值才发送 |
| **事件上传** | 报警、启停、故障时发送 |
| **汇总上传** | 上传平均值、最大值、最小值 |
| **压缩上传** | 减少报文大小 |
| **边缘存储** | 本地保存明细，平台只收摘要 |

## ③ 计算/选型示例

- 已知：  
  1000个变量每秒上传，每个变量4字节。
- 公式：  
  **数据量 = 1000 × 4 × 86400**
- 计算：  
  日数据量 = **345,600,000字节 ≈ 345.6MB**
- 结论：  
  如果不是所有变量都需要秒级上传，可在边缘侧汇总或降频。

> [!tip] 记忆口诀
> 数据不是越多越好，上传有价值的数据才重要。

# 13 本地缓存与断点续传

> [!info] 核心结论
> 边缘计算网关必须具备本地缓存能力，网络断开时保存数据，恢复后按时间戳补传。

## ① 分类/公式/规则

> [!warning] 缓存容量公式
> **缓存容量 ≥ 每分钟数据量 × 断网分钟数 × 安全余量**

## ② 参考表

| 功能 | 作用 |
|---|---|
| **本地缓存** | 断网期间保存数据 |
| **时间戳** | 保留真实采集时间 |
| **断点续传** | 网络恢复后补传历史数据 |
| **去重机制** | 防止重复写入平台 |
| **缓存告警** | 存储快满时提示 |
| **循环覆盖** | 存储满后按策略覆盖旧数据 |
| **边缘数据库** | SQLite、时序数据库等 |

## ③ 计算/选型示例

- 已知：  
  每分钟产生2MB数据，现场可能断网8小时。
- 公式：  
  **缓存容量 ≥ 2MB × 60 × 8**
- 计算：  
  缓存容量 ≥ **960MB**
- 结论：  
  网关本地存储至少要大于960MB，并留足余量。

> [!tip] 记忆口诀

# 14 时间同步

> [!info] 核心结论
> 边缘计算网关的数据必须有准确时间戳，时间不同步会严重影响追溯、报警分析和数据对齐。

## ① 分类/公式/规则

> [!warning] 追溯公式
> **有效追溯数据 = 数值 + 时间戳 + 设备来源 + 数据质量标记**

## ② 参考表

| 时间功能 | 说明 |
|---|---|
| **RTC时钟** | 断电后保持时间 |
| **NTP同步** | 通过网络校时 |
| **平台授时** | 平台统一下发时间 |
| **GPS授时** | 特殊场景高精度时间 |
| **本地时区** | 避免时间偏移 |
| **采集时间** | 数据产生时刻 |
| **上传时间** | 数据到平台时刻 |

## ③ 计算/选型示例

- 已知：  
  断网补传后，平台显示所有数据都发生在恢复联网的同一时刻。
- 公式：  
  **补传应使用采集时间，不应只用上传时间**
- 计算：  
  否则无法判断故障实际发生顺序。
- 结论：  
  网关必须保存原始采集时间戳并随数据上传。

> [!tip] 记忆口诀
> 没有准确时间戳，数据就很难追溯。

# 18 安全与权限

> [!info] 核心结论
> 边缘计算网关连接设备和平台，是工业网络的重要边界点，必须做好加密、认证、权限、日志和白名单。

## ① 分类/公式/规则

> [!warning] 安全公式
> **安全边缘网关 = 强密码 + 证书认证 + 加密传输 + 权限分级 + 日志审计 + 白名单**

## ② 参考表

| 安全项 | 作用 |
|---|---|
| **账号权限** | 区分管理员、工程师、只读用户 |
| **HTTPS/TLS** | 加密Web配置和数据上传 |
| **MQTT证书** | 云平台安全认证 |
| **VPN访问** | 安全远程维护 |
| **访问白名单** | 只允许指定IP或设备访问 |
| **操作日志** | 记录登录、配置、升级、控制操作 |
| **固件升级** | 修复漏洞和提升功能 |

## ③ 计算/选型示例

- 公式：  
  **默认密码 = 高安全风险**
- 计算：  
  任何知道默认密码的人都可能登录网关。
- 结论：  
  上线前必须修改默认密码，关闭不必要服务，并启用日志审计。

> [!tip] 记忆口诀

# 19 供电与安装

> [!info] 核心结论
> 边缘计算网关通常安装在控制柜内，常用DC24V供电，需关注散热、接地、天线、存储寿命和工业环境适应性。

## ① 分类/公式/规则

> [!warning] 稳定运行公式
> **稳定运行 = 稳定电源 + 良好接地 + 合理散热 + 可靠存储 + 规范布线**

## ② 参考表

| 项目 | 注意 |
|---|---|

# 20 常见故障排查

> [!info] 核心结论
> 边缘计算网关故障排查按 **供电、网络、设备协议、点表、计算规则、缓存、平台接口、安全权限** 的顺序进行。

## ① 分类/公式/规则

> [!warning] 排错公式
> **边缘网关异常 = 供电异常 + 网络不通 + 协议不匹配 + 点表错误 + 规则错误 + 上传失败 + 权限阻断**

## ② 参考表

| 现象 | 可能原因 |
|---|---|
| **网关不上电** | 电源无电、接线错误、设备故障 |
| **读不到PLC** | IP、协议、端口、权限、PLC未运行 |
| **读不到仪表** | RS485接反、波特率、站号、校验位错误 |
| **数值异常** | 地址错、数据类型错、字节序错、倍率错 |
| **规则不触发** | 条件写错、变量名错、周期设置不合理 |
| **上传失败** | 网络、平台地址、证书、Topic、账号错误 |
| **断网丢数据** | 缓存未开启或容量不足 |
| **运行变慢** | 点数过多、采样太快、脚本过重、存储满 |

## ③ 计算/选型示例

- 已知：  
  网关能Ping通PLC，但采集不到变量。
- 公式：  
  **Ping通 ≠ 协议采集成功**
- 计算：  
  网络层通了，但PLC协议端口、访问权限或地址配置可能错误。
- 结论：  
  应检查PLC通讯协议、端口、机架槽号、变量地址和网关驱动配置。

> [!tip] 记忆口诀
> 网络通只是第一步，协议通、点表对，数据才是真的通。

# 21 边缘计算网关快速选型表

> [!info] 核心结论
> 边缘计算网关选型要先确认采集对象和平台接口，再确认算力、存储、规则、缓存、安全和工业环境。

## ① 分类/公式/规则

> [!note] 十步选型法
> ① 确认采集对象：PLC、仪表、机器人、CNC、传感器  
> ② 确认下行协议：Modbus、S7、OPC UA、EtherNet/IP等  
> ③ 统计采集点数和采样周期  
> ④ 确认数据清洗、工程换算和规则判断需求  
> ⑤ 确认是否需要AI推理、脚本、Node-RED或Docker  
> ⑥ 确认上行协议：MQTT、HTTP、OPC UA、数据库等  
> ⑦ 确认断网缓存、断点续传和本地存储容量  
> ⑧ 确认网络方式：双网口、Wi-Fi、4G/5G、VPN  
> ⑨ 确认安全：证书、加密、权限、白名单、日志  
> ⑩ 确认供电、安装、温度、散热、接地和存储寿命

> [!warning] 最终选型公式
> **推荐边缘计算网关 = 多协议采集 + 边缘算力 + 存储缓存 + 上云接口 + 安全能力 + 工业环境等级**

## ② 参考表

| 需求 | 推荐关注点 |
|---|---|
| PLC数据采集 | 协议驱动、点表管理 |
| 仪表采集 | RS485、Modbus RTU、站号管理 |
| 数据上云 | MQTT、HTTP、TLS证书 |
| MES/SCADA对接 | OPC UA、数据库、API |
| 断网不丢数据 | 本地缓存、断点续传 |
| 本地报警判断 | 规则引擎、事件触发 |
| 预测维护 | 振动、电流、温度数据和AI推理 |
| 复杂业务逻辑 | 脚本、Node-RED、Docker |
| 高安全要求 | VPN、白名单、日志审计 |
| 长期运行 | 工业级电源、散热、存储寿命 |

## ③ 计算/选型示例

- 已知：  
  项目有5台PLC、20台RS485仪表、共2000个采集点，每5秒采集一次，要求MQTT上传云平台，断网可缓存24小时。
- 公式：  
  **网关需求 = 多协议 + 2000点 + 5秒周期 + MQTT + 24小时缓存**
- 计算：  
  需要较强采集性能、足够存储、断点续传、时间戳和安全认证。
- 结论：  
  推荐选择 **高性能边缘计算网关，支持多PLC协议、Modbus RTU、MQTT TLS、规则引擎、本地缓存、断点续传、双网口隔离和工业导轨安装**。

> [!tip] 选型口诀
> 先看下面读什么，再看上面传给谁；点数决定性能，断网决定缓存；要本地判断看规则，要智能分析看边缘算力。
