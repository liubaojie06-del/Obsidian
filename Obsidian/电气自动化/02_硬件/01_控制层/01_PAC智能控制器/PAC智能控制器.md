# 01 PAC智能控制器总览

> [!info] 核心结论
> PAC智能控制器是介于PLC、工业PC和运动控制器之间的高性能控制平台，适合 **复杂逻辑、多轴运动、过程控制、数据处理、通讯联网和边缘计算** 场景。

## ① 分类/公式/规则

> [!note] 核心理解
> **PAC = Programmable Automation Controller，可编程自动化控制器**。  
> 它比传统PLC更强调开放性、数据处理能力、复杂算法、多任务和网络集成。

> [!warning] 选型公式
> **PAC选型 = I/O规模 + 控制周期 + 轴数 + 通讯协议 + 数据处理能力 + 软件平台 + 扩展能力 + 安全需求**

## ② 参考表

| 项目 | 说明 |
|---|---|
| **逻辑控制** | 顺序、互锁、报警、设备流程 |
| **运动控制** | 伺服、多轴同步、插补、电子凸轮 |
| **过程控制** | 温度、压力、流量、液位、PID |
| **通讯联网** | 工业以太网、现场总线、OPC UA等 |
| **数据处理** | 配方、报警、追溯、统计、边缘计算 |
| **开放性** | 支持高级语言、数据库或上位系统集成 |
| **典型应用** | 智能装备、复杂产线、测试系统、过程装置 |

## ③ 计算/选型示例

- 已知：  
  设备有8轴伺服、120点I/O、视觉检测、MES上传和配方管理。
- 公式：  
  **复杂控制 + 数据联网 + 多轴运动 → PAC优先评估**
- 计算：  
  普通PLC可能在算法、数据处理和多系统通讯方面受限。
- 结论：  
  可选择 **PAC智能控制器**，并校核轴控能力、通讯协议和软件开发环境。

> [!tip] 记忆口诀
> PLC管逻辑，PAC管逻辑、运动、数据和联网。

# 02 底层原理

> [!info] 核心结论
> PAC的底层逻辑是：实时采集现场信号，按多任务程序和控制算法运算，再通过I/O、总线或通讯控制设备。

## ① 分类/公式/规则

> [!note] 工作链路
> PAC读取传感器、编码器、仪表、伺服和远程I/O数据，执行逻辑、运动、PID、数据处理等任务，再输出控制指令。

> [!warning] 控制链路
> **现场输入 → PAC实时运算 → 逻辑/运动/过程/数据处理 → 输出控制 → 反馈修正 → 数据上传**

## ② 参考表

| 环节 | 作用 |
|---|---|
| **输入采集** | 读取传感器、按钮、模拟量、编码器 |
| **实时任务** | 执行高优先级控制逻辑 |
| **运动任务** | 控制伺服轴、插补、同步 |
| **过程任务** | PID、温控、流量压力控制 |
| **数据任务** | 记录、统计、配方、追溯 |
| **通讯任务** | HMI、MES、SCADA、远程IO通讯 |

## ③ 计算/选型示例

- 已知：  
  设备要求20ms内完成检测、判断和剔除输出。
- 公式：  
  **响应时间 = 输入采样 + 控制任务周期 + 输出响应 + 执行机构响应**
- 计算：  
  若输入采样5ms、任务周期10ms、输出响应2ms，控制侧约17ms。
- 结论：  
  理论满足20ms控制侧响应，但还需校核执行机构动作时间。

> [!tip] 记忆口诀
> PAC不是只跑程序，还要同时跑控制、运动、通讯和数据任务。

# 03 PAC vs PLC

> [!info] 核心结论
> PLC适合稳定可靠的逻辑控制，PAC更适合复杂控制、开放通讯、多轴运动和数据密集型应用。

## ① 分类/公式/规则

> [!note] 快速区别
> PLC偏传统工业顺序控制；  
> PAC兼具PLC可靠性和工业计算平台的开放性。

> [!warning] 选择规则
> **简单逻辑控制 → PLC**  
> **复杂算法 / 多轴 / 数据联网 / 高开放性 → PAC**

## ② 参考表

| 对比项 | PLC | PAC |
|---|---|---|
| 逻辑控制 | 强 | 强 |
| 运动控制 | 视型号而定 | **通常更强** |
| 数据处理 | 一般 | **更强** |
| 开放性 | 相对封闭 | **更开放** |
| 高级语言 | 部分支持 | **更常见** |
| 系统集成 | 常规设备级 | **设备+数据+网络集成** |

## ③ 计算/选型示例

- 已知：  
  设备只有30点I/O和几个气缸顺序动作。
- 公式：  
  **简单顺控 + 少量I/O → PLC即可**
- 计算：  
  不需要复杂算法、多轴同步和大量数据接口。
- 结论：  
  不必上PAC，普通小型或中型PLC即可。

> [!tip] 记忆口诀
> 简单可靠用PLC，复杂开放用PAC。

# 04 PAC vs 工业PC

> [!info] 核心结论
> 工业PC计算能力强，PAC更偏实时控制；两者区别在于实时性、稳定性、I/O集成和工业控制软件生态。

## ① 分类/公式/规则

> [!note] 快速区别
> 工业PC适合上位计算、视觉、数据库和算法；  
> PAC适合把实时控制、I/O和计算集成在一个工业控制平台中。

> [!warning] 选择规则
> **强实时控制 → PAC优先**  
> **大规模视觉/AI/数据库 → 工业PC或PAC+IPC组合**

## ② 参考表

| 对比项 | 工业PC | PAC |
|---|---|---|
| 计算能力 | 强 | 中高 |
| 实时控制 | 需实时系统支持 | **更偏实时控制** |
| I/O集成 | 依赖扩展 | **工业I/O集成更直接** |
| 开放性 | 很强 | 较强 |
| 可靠性 | 取决于系统设计 | 工业控制取向 |
| 应用 | 视觉、数据库、上位系统 | 设备控制、运动、过程控制 |

## ③ 计算/选型示例

- 已知：  
  设备需要高速多轴同步，同时还要做少量数据追溯。
- 公式：  
  **实时控制为主 + 数据处理为辅 → PAC适合**
- 计算：  
  PAC可直接处理运动控制和基础数据功能。
- 结论：  
  可优先选择PAC；若视觉AI很重，再增加工业PC。

> [!tip] 记忆口诀
> 工控机更会算，PAC更会控；既要算又要控，就看系统架构。

# 05 系统架构

> [!info] 核心结论
> PAC通常位于设备控制核心，向下连接现场设备，向上连接HMI、SCADA、MES、数据库或云平台。

## ① 分类/公式/规则

> [!warning] 标准架构
> **现场设备 → PAC → HMI/SCADA/MES → 数据平台/云端**

## ② 参考表

| 层级 | 典型对象 |
|---|---|
| **现场层** | 传感器、执行器、阀岛、仪表 |
| **驱动层** | 伺服、变频器、步进驱动 |
| **控制层** | PAC智能控制器 |
| **操作层** | HMI、触摸屏、工控机 |
| **管理层** | SCADA、MES、数据库 |
| **分析层** | 能耗、质量、预测维护 |

## ③ 计算/选型示例

- 已知：  
  设备要本地控制，同时上传产量、报警、质量数据到MES。
- 公式：  
  **设备控制 + 数据上传 → PAC需支持上位通讯**
- 计算：  
  PAC需要同时连接现场总线和上位系统协议。
- 结论：  
  应关注PAC是否支持OPC UA、数据库接口、MQTT或厂家网关方案。

> [!tip] 记忆口诀
> PAC下控设备，上连系统，中间负责控制和数据。

# 06 硬件组成

> [!info] 核心结论
> PAC硬件通常由CPU、实时内核、I/O模块、通讯接口、运动控制接口、存储、电源和扩展模块组成。

## ① 分类/公式/规则

> [!warning] 硬件选型公式
> **硬件能力 = CPU性能 + 实时周期 + I/O容量 + 轴控能力 + 通讯接口 + 存储容量 + 扩展能力**

## ② 参考表

| 硬件模块 | 作用 |
|---|---|
| **CPU处理器** | 逻辑、算法和任务调度 |
| **实时系统** | 保证控制周期稳定 |
| **DI/DO模块** | 开关量输入输出 |
| **AI/AO模块** | 模拟量采集和输出 |
| **运动接口** | 控制伺服、步进、多轴运动 |
| **通讯接口** | 工业以太网、串口、现场总线 |
| **存储单元** | 程序、配方、数据、日志 |

## ③ 计算/选型示例

- 已知：  
  系统需要64DI、48DO、8AI、4AO、6轴伺服。
- 公式：  
  **PAC配置 ≥ 实际点数和轴数 + 预留余量**
- 计算：  
  I/O总点数较多，建议预留10%~20%扩展空间。
- 结论：  
  需选择可扩展I/O和支持6轴以上运动控制的PAC。

> [!tip] 记忆口诀
> CPU看性能，I/O看点数，轴控看能力，通讯看协议。

# 07 软件平台

> [!info] 核心结论
> PAC的软件平台决定开发效率和系统开放性，重点看IEC 61131-3语言、运动库、通讯库、数据接口和调试工具。

## ① 分类/公式/规则

> [!note] 软件理解
> PAC常支持梯形图、功能块、结构化文本，也可能支持C/C++、Python、脚本或边缘应用环境。

> [!warning] 软件选型公式
> **软件能力 = 编程语言 + 实时任务 + 运动库 + 通讯库 + 数据库接口 + 诊断工具**

## ② 参考表

| 软件能力 | 作用 |
|---|---|
| **梯形图LD** | 逻辑控制直观 |
| **功能块FB** | 模块化复用 |
| **结构化文本ST** | 算法和数据处理方便 |
| **运动控制库** | 轴控、插补、电子凸轮 |
| **通讯库** | 连接仪表、机器人、MES |
| **数据功能** | 记录、追溯、配方、报表 |
| **在线调试** | 监控变量、趋势、报警 |

## ③ 计算/选型示例

- 已知：  
  设备需要复杂配方计算和多轴联动。
- 公式：  
  **复杂算法 + 多轴控制 → 软件平台要支持ST和运动库**
- 计算：  
  仅靠简单梯形图维护困难。
- 结论：  
  应选择支持结构化编程、功能块复用和运动控制库的PAC平台。

> [!tip] 记忆口诀
> 硬件决定上限，软件决定好不好开发和维护。

# 08 运动控制能力

> [!info] 核心结论
> PAC常用于多轴运动控制，适合定位、同步、插补、电子齿轮、电子凸轮、飞剪和追剪等应用。

## ① 分类/公式/规则

> [!warning] 运动选型公式
> **运动能力 = 轴数 + 总线周期 + 同步精度 + 插补类型 + 运动功能库**

## ② 参考表

| 功能 | 应用 |
|---|---|
| **点位运动** | 滑台、转盘、送料 |
| **速度控制** | 输送线、卷绕 |
| **直线插补** | XY平台、切割 |
| **圆弧插补** | 涂胶、加工轨迹 |
| **电子齿轮** | 主从轴同步 |
| **电子凸轮** | 包装、飞剪、追剪 |
| **多轴同步** | 印刷、包装、装配线 |

## ③ 计算/选型示例

- 已知：  
  设备有X/Y/Z三轴，需要按轨迹运动。
- 公式：  
  **轨迹运动 → 多轴插补**
- 计算：  
  单轴点位控制无法保证轨迹连续性。
- 结论：  
  需选择支持多轴插补的PAC，并校核总线周期和轴数。

> [!tip] 记忆口诀
> 单轴看定位，多轴看同步，轨迹运动看插补。

# 09 过程控制能力

> [!info] 核心结论
> PAC不仅能做离散逻辑和运动控制，也能做温度、压力、流量、液位等过程控制。

## ① 分类/公式/规则

> [!note] 过程控制理解
> PAC可通过模拟量模块、PID功能块、通讯仪表和配方参数实现过程变量闭环控制。

> [!warning] PID控制公式
> **控制偏差 = 设定值SP - 反馈值PV**

## ② 参考表

| 控制对象 | 常见执行机构 |
|---|---|
| **温度** | 加热器、冷却阀、温控模块 |
| **压力** | 比例阀、变频泵、调压阀 |
| **流量** | 调节阀、变频器 |
| **液位** | 泵、阀门 |
| **张力** | 磁粉制动器、伺服、气压控制 |
| **配方工艺** | 多参数联动控制 |

## ③ 计算/选型示例

- 已知：  
  目标压力0.5MPa，实际压力0.42MPa。
- 公式：  
  **偏差 = 0.5 - 0.42**
- 计算：  
  偏差 = **0.08MPa**
- 结论：  
  PAC可通过PID或比例控制增加输出，使压力接近设定值。

> [!tip] 记忆口诀
> 运动看位置，过程看变量；变量要稳，就要闭环。

# 10 通讯协议

> [!info] 核心结论
> PAC的通讯能力是核心优势之一，能连接远程I/O、伺服、变频器、机器人、视觉、仪表、HMI和MES。

## ① 分类/公式/规则

> [!warning] 通讯选型公式
> **通讯方案 = 下层设备协议 + 上层系统协议 + 实时性 + 数据量 + 诊断需求**

## ② 参考表

| 协议 | 常见用途 |
|---|---|
| **EtherCAT** | 高速多轴运动和远程I/O |
| **PROFINET** | PLC、阀岛、远程I/O、驱动 |
| **EtherNet/IP** | 工业以太网设备 |
| **Modbus TCP/RTU** | 仪表、变频器、通用设备 |
| **CANopen** | 运动设备和现场设备 |
| **OPC UA** | SCADA、MES、上位系统 |
| **MQTT** | 轻量级数据上传和物联网 |

## ③ 计算/选型示例

- 已知：  
  PAC需要控制12个伺服轴，并上传数据到MES。
- 公式：  
  **下层实时控制 + 上层数据通讯 = 双通道通讯能力**
- 计算：  
  伺服轴可走EtherCAT，MES可走OPC UA或数据库接口。
- 结论：  
  选型时需确认PAC是否支持多协议并行运行。

> [!tip] 记忆口诀
> 控设备看实时协议，传数据看开放协议。

# 11 数据采集与追溯

> [!info] 核心结论
> PAC通常具备较强数据功能，可记录设备状态、报警、产量、质量、配方、能耗和工艺参数。

## ① 分类/公式/规则

> [!warning] 数据方案公式
> **数据方案 = 采集点数 + 采样周期 + 存储容量 + 时间戳 + 上传方式 + 追溯规则**

## ② 参考表

| 数据类型 | 用途 |
|---|---|
| **报警记录** | 故障分析 |
| **产量数据** | 生产统计 |
| **节拍数据** | 效率分析 |
| **质量数据** | 产品追溯 |
| **工艺参数** | 配方和过程分析 |
| **能耗数据** | 节能管理 |
| **设备状态** | 预测维护 |

## ③ 计算/选型示例

- 已知：  
  每秒采集50个变量，每个变量4字节，保存24小时。
- 公式：  
  **存储量 = 变量数 × 字节数 × 秒数**
- 计算：  
  50 × 4 × 86400 = **17,280,000字节 ≈ 17.3MB**
- 结论：  
  存储量不大，但长期追溯需规划文件管理、数据库和数据上传机制。

> [!tip] 记忆口诀
> 数据要有时间戳，追溯才有意义。

# 12 边缘计算与智能诊断

> [!info] 核心结论
> PAC可在设备现场进行边缘计算，实现状态监测、异常判断、预测维护和本地数据处理。

## ① 分类/公式/规则

> [!note] 边缘计算理解
> 不把所有数据都传到上位机或云端，而是在设备本地先处理，减少通讯压力，提高响应速度。

> [!warning] 边缘计算公式
> **边缘价值 = 本地快速判断 + 减少上传数据 + 提高设备自治能力**

## ② 参考表

| 功能 | 应用 |
|---|---|
| **异常检测** | 振动、温度、压力异常 |
| **预测维护** | 轴承、气缸、阀门寿命分析 |
| **质量判断** | 根据工艺数据判断OK/NG |
| **节拍分析** | 找出慢工位 |
| **能耗分析** | 找出高耗能动作 |
| **本地缓存** | 网络断开时保存数据 |

## ③ 计算/选型示例

- 已知：  
  设备每秒产生大量传感器数据，但MES只需要报警和统计结果。
- 公式：  
  **本地处理 → 只上传结果**
- 计算：  
  PAC可在本地计算平均值、最大值、报警状态，再上传摘要。
- 结论：  
  可降低网络和上位系统压力。

> [!tip] 记忆口诀
> 能在现场判断的，就别全丢给上位机。

# 13 安全与可靠性

> [!info] 核心结论
> PAC负责普通控制，但涉及人身安全的急停、安全门、光栅、STO等必须按安全系统设计。

## ① 分类/公式/规则

> [!warning] 安全规则
> **普通逻辑由PAC处理，安全功能由安全PLC/安全继电器/安全I/O处理**

## ② 参考表

| 安全项 | 说明 |
|---|---|
| **急停** | 紧急停止危险动作 |
| **安全门** | 开门禁止危险运动 |
| **安全光栅** | 防止人员进入危险区域 |
| **STO** | 伺服安全转矩关闭 |
| **安全继电器** | 基础安全回路 |
| **安全PLC** | 复杂安全逻辑 |
| **复位确认** | 防止误启动 |

## ③ 计算/选型示例

- 已知：  
  开安全门后，伺服轴必须失去转矩。
- 公式：  
  **安全门打开 → 安全回路触发STO**
- 计算：  
  不能只靠PAC普通程序停止伺服。
- 结论：  
  应采用安全门开关、安全控制单元和伺服STO，安全等级按风险评估确定。

> [!tip] 记忆口诀
> PAC管控制，安全系统管人命。

# 14 常见故障排查

> [!info] 核心结论
> PAC故障排查按 **电源、CPU状态、实时任务、通讯、I/O、轴控、程序、现场设备、数据接口** 的顺序进行。

## ① 分类/公式/规则

> [!warning] 排错公式
> **PAC异常 = 电源异常 + 任务超时 + 通讯掉站 + I/O错误 + 轴报警 + 程序逻辑错误 + 数据接口异常**

## ② 参考表

| 现象 | 可能原因 |
|---|---|
| **控制器不运行** | 电源、系统故障、程序错误 |
| **任务超时** | 程序过重、周期设置过短 |
| **通讯掉站** | 网线、地址、协议、从站故障 |
| **输入无信号** | 传感器、接线、PNP/NPN不匹配 |
| **输出不动作** | 输出点、公共端、负载、电源异常 |
| **伺服报警** | 限位、过载、编码器、参数错误 |
| **数据不上报** | 网络、协议、权限、字段映射问题 |
| **运行变慢** | CPU负载高、通讯任务过多 |

## ③ 计算/选型示例

- 已知：  
  PAC周期任务偶发超时。
- 公式：  
  **任务超时 = 计算量过大 / 通讯阻塞 / 周期设置过短**
- 计算：  
  若任务周期设为5ms，但程序中有大量数据处理，容易超时。
- 结论：  
  应拆分任务优先级，把实时控制和数据处理分开执行。

> [!tip] 记忆口诀
> 实时任务别塞重计算，数据任务别拖慢控制。

# 15 PAC智能控制器快速选型表

> [!info] 核心结论
> PAC选型要从设备复杂度出发，统一评估I/O、运动、过程、通讯、数据、软件、安全和维护能力。

## ① 分类/公式/规则

> [!note] 十步选型法
> ① 统计DI/DO/AI/AO点数  
> ② 统计伺服、步进、变频器、阀岛数量  
> ③ 确认运动轴数和同步要求  
> ④ 确认过程控制和PID数量  
> ⑤ 确认现场总线和工业以太网协议  
> ⑥ 确认HMI、SCADA、MES数据接口  
> ⑦ 确认控制周期和实时性要求  
> ⑧ 确认数据记录、追溯和边缘计算需求  
> ⑨ 确认安全回路和STO需求  
> ⑩ 预留I/O、轴数、通讯和存储扩展余量

> [!warning] 最终选型公式
> **推荐PAC = CPU性能 + 实时周期 + I/O容量 + 轴控能力 + 通讯协议 + 数据能力 + 软件平台 + 安全扩展**

## ② 参考表

| 需求 | 推荐关注点 |
|---|---|
| 简单顺控 | PLC即可 |
| 多轴同步 | PAC运动控制能力 |
| 过程闭环 | PID和模拟量能力 |
| 视觉/机器人集成 | 通讯和数据接口 |
| 数据追溯 | 存储、数据库、OPC UA |
| 边缘计算 | CPU性能和软件开放性 |
| 高可靠运行 | 电源、冗余、诊断和备份 |
| 安全控制 | 安全I/O、STO、安全PLC配合 |

## ③ 计算/选型示例

- 已知：  
  项目有10轴伺服、96DI、64DO、8AI、4AO、2台视觉、MES上传和数据追溯。
- 公式：  
  **PAC需求 = 轴数 + I/O点数 + 视觉通讯 + MES接口 + 数据存储**
- 计算：  
  系统既有实时运动控制，又有数据和上位集成需求。
- 结论：  
  推荐选择 **高性能PAC智能控制器**，并按厂家手册校核轴数、总线周期、通讯连接数和存储能力。

> [!tip] 选型口诀
> 先数点，再数轴；再看协议、周期和数据；简单用PLC，复杂智能用PAC。
